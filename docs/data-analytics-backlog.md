# Trusted Bums Data Analytics Backlog

_Last updated: 2026-06-06 by Codex daily data analytics engineer automation._

## Executive Read

Trusted Bums still has five active analytics and reporting gaps with current evidence: finance report workspaces still filter and group on audit timestamps instead of business-effective dates, `CLIENT_FINANCE` still reaches mixed operational exports, the admin dashboard still depends on an exposed `SECURITY DEFINER` summary RPC with overly broad execute grants in source, access-request and terms-deferral controls still lack admin-grade aging and outcome reporting, and admin email analytics still truncate core reads to 50 rows while giving opens too much weight for a privacy-degraded signal. This run had live Supabase access to project metadata, edge-function inventory, and recent logs, but not direct SQL, advisor output, grants, or policy catalog inspection.

## Active Recommendations

### P1 - Finish business-effective dates in finance report workspaces
- Evidence: [src/pages/client/ClientReports.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientReports.tsx) still uses `dateKey: "createdAt"` for the `CLIENT_FINANCE` customer-payment ledger, revenue-by-customer, and finance-exceptions reports, and `dateKey: "generatedAt"` for invoice register and invoice aging. [src/pages/admin/AdminReports.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminReports.tsx) still groups the finance-operations report on `createdAt`. [src/pages/bum/BumReports.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/bum/BumReports.tsx) still keys payout reporting to `createdAt`. The shared helpers in [src/lib/portalApi.ts](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/portalApi.ts) already define `paymentReportBusinessDate()`, `claimInvoiceBusinessDate()`, and `bumPayoutBusinessDate()`.
- Why it matters: Client Finance users, admins, and Bums can still place the same commercial event in the wrong reporting period even though the data model and shared query helpers already carry better business dates for reconciliation, invoice aging, and payout reporting.
- Recommendation: Re-key finance report workspaces and report filters to business-effective dates: `customer_payment_received_at` for Customer Payment Reports, `paid_at` or `sent_at` with payment fallback for commission invoices, and `paid_at` or `approved_at` for Bum payouts. Keep `created_at` visible only as an audit field.
- Acceptance criteria: Client, admin, and Bum finance report filters and default sort order follow business-effective dates; each row still exposes the audit timestamp separately; and seeded late-entry scenarios land in the expected reporting period.

[src/pages/client/ClientExports.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientExports.tsx) still exports `key_contact_email`, meeting attendee lists, `teams_join_url`, and `transcript_sync_status` alongside finance-safe payment rows. - Why it matters: The current export route still bundles finance reporting with operational contact and meeting context, which breaks the documented finance-only data posture and creates a direct CSV egress path for fields finance users are not approved to handle.
- Recommendation: Split `/client/exports` into finance-safe and operational export types, or gate target and meeting exports to `CLIENT_ADMIN` until a narrower business rule explicitly grants extra Client Finance scope. Because this may require route, RLS, RPC, or edge-function changes, Security and Lead Developer should validate the final design against [docs/business-access-rules.md](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/docs/business-access-rules.md) before rollout.
- Acceptance criteria: `CLIENT_FINANCE` can export company payment and invoice data without target contact emails, attendee lists, join URLs, transcript-adjacent metadata, or other operational detail unless a documented rule explicitly allows it; route allow/deny tests cover each export type by client role.

### P1 - Restrict the admin dashboard summary RPC to admin-only callers
- Evidence: [src/lib/portalApi.ts](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/portalApi.ts) still calls `rpc("admin_dashboard_summary")` for [src/pages/admin/AdminDashboard.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminDashboard.tsx). [supabase/migrations/20260529001000_add_admin_dashboard_summary.sql](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/migrations/20260529001000_add_admin_dashboard_summary.sql) still creates `public.admin_dashboard_summary()` as `SECURITY DEFINER`, leaves it in exposed schema `public`, revokes from `public`, and then grants execute to `authenticated`. Current Supabase guidance says functions are controlled by execute grants rather than RLS, recommends explicit least-privilege grants, and recommends keeping `SECURITY DEFINER` helpers out of exposed schemas when possible ([Securing your API](https://supabase.com/docs/guides/api/securing-your-api), [RLS docs](https://supabase.com/docs/guides/database/postgres/row-level-security), [security definer troubleshooting, last edited 2026-04-08](https://supabase.com/docs/guides/troubleshooting/do-i-need-to-expose-security-definer-functions-in-row-level-security-policies-iI0uOw)).
- Why it matters: Marketplace-wide dashboard aggregates are an admin-only reporting surface. The current grant model makes the privileged helper callable by every authenticated role even though the function body throws for non-admins, which is broader API exposure than needed.
- Recommendation: Revoke execute from generic `authenticated`, move the helper behind a non-exposed schema or admin-only wrapper, and add negative access tests for signed-out and signed-in non-admin roles. Security and Lead Developer should validate the final RPC or service-role pattern against [docs/business-access-rules.md](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/docs/business-access-rules.md) before implementation.
- Acceptance criteria: `anon`, generic `authenticated`, Client, and Bum callers cannot execute the summary RPC; admins still receive the dashboard summary cards; and the final implementation uses explicit admin-only grants rather than exposed broad-role execution.

### P2 - Remove 50-row admin email caps and demote opens to directional-only
- Evidence: [supabase/functions/send-admin-email/index.ts](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/functions/send-admin-email/index.ts) still hard-limits `list_deliveries`, `list_engagement`, and `list_campaigns` to 50 rows, and [src/pages/admin/AdminEmails.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminEmails.tsx) consumes those reads directly. Apple still documents that Mail Privacy Protection prevents senders from seeing whether a recipient opened an email ([Apple Support](https://support.apple.com/en-mide/guide/iphone/iphf084865c7/ios)). The ICO published final storage-and-access-technologies guidance on 2026-04-29 confirming that tracking pixels remain within current PECR and UK GDPR compliance scope ([ICO announcement](https://ico.org.uk/about-the-ico/media-centre/news-and-blogs/2026/04/final-storage-and-access-technologies-guidance-published/)).
- Why it matters: Silent row truncation makes admin delivery reporting incomplete, and open-heavy KPI framing overstates a signal that is both technically degraded and privacy-sensitive.
- Recommendation: Replace fixed 50-row reads with pagination or admin-safe aggregate views, and relabel opens as directional only. Default KPI cards should prioritize sends, failures, suppressions, clicks, and downstream portal actions over opens.
- Acceptance criteria: Admin email views can page or summarize beyond 50 rows without missing historical data; the UI labels opens as directional; and default KPI cards emphasize reliable delivery and action metrics ahead of opens.

## Business Access And Reporting Rules

- Finance business dates: Admins, Client Admins, Client Finance, and Bums need business-effective dates for reconciliation, period filtering, invoice aging, and payout reporting. Non-finance client roles should not gain finance visibility as part of this work. Sensitivity: financial reporting accuracy. Verification: seeded late-entry scenarios prove filters use effective dates while `created_at` remains an audit field.
- Client Finance exports: Client Finance needs company-scoped payment reports, invoice registers, finance dashboards, and finance exports. Client Finance should not receive target contact emails, attendee lists, join URLs, transcript-adjacent metadata, or other operational workflow detail unless [docs/business-access-rules.md](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/docs/business-access-rules.md) is explicitly expanded. Sensitivity: financial plus operational contact data. Verification: role-by-role export content assertions and route allow/deny tests.
- Admin aggregate reporting RPCs: Admins need marketplace-wide operational aggregates for dashboard cards. Public visitors, Client roles, and Bums should not execute privileged summary RPCs or receive marketplace-wide counts through exposed APIs. Sensitivity: admin-only aggregate operational reporting. Verification: execute-grant review plus negative RPC tests for `anon`, `authenticated`, Client, and Bum callers.
- Access-request and terms-deferral reporting: Admins need historical request outcomes, reviewer timestamps, evidence categories, and deferral-limit visibility for onboarding governance. Client Admins may need same-company pending-request visibility only where the business rule explicitly allows it. Client Finance, Bums, and public users should not read raw request evidence, review notes, or deferral histories. Sensitivity: identity/access governance and contract-compliance behavior. Verification: role-by-role report tests plus explicit business-rule text before any new reporting path ships.
- Admin email analytics: Admins need recipient-level delivery and engagement operations data for troubleshooting and campaign operations. Non-admin roles should not read raw delivery history, recipient behavior rows, or raw engagement telemetry. Opens should be treated as aggregate-only directional signal, not definitive recipient action. Sensitivity: recipient operations data and messaging behavior. Verification: admin-only access tests for campaign, delivery, and engagement surfaces, plus review of any new aggregate views.
