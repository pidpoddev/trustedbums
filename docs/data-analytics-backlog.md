# Trusted Bums Data Analytics Backlog

_Last updated: 2026-05-31 by Codex daily data analytics engineer automation._

## Executive Read

The highest-leverage active analytics issues are still report trust and access control, not visualization polish. Finance-facing reports still filter on creation-style timestamps even though the data layer now exposes business-effective date helpers; `CLIENT_FINANCE` still reaches mixed operational/finance exports; the admin dashboard summary RPC is currently exposed as a `SECURITY DEFINER` function callable by `anon` and `authenticated`; admin email reporting is still capped at 50 rows while the live event mix is overwhelmingly open-based; and performance telemetry is now flowing, but redirect-driven route attribution is splitting dashboard landing metrics.

## Active Recommendations

### P1 - Finish business-effective finance dates across reports and exports
- Evidence: `src/lib/portalApi.ts` defines `paymentReportBusinessDate`, `claimInvoiceBusinessDate`, and `bumPayoutBusinessDate`, and `src/test/paymentCommission.test.ts` covers those helpers. But `src/pages/client/ClientReports.tsx` still uses `dateKey: "createdAt"` for the customer payment ledger, revenue-by-customer, and finance exceptions, and `dateKey: "generatedAt"` for invoice views. `src/pages/admin/AdminReports.tsx` still groups the finance queue by `createdAt`, and `src/pages/bum/BumReports.tsx` still filters payouts by `createdAt`. Live Supabase aggregates in this run show only 1 `customer_payment_reports` row, 1 `claim_invoices` row, and 0 `bum_payouts` rows, so late-entry period handling is still source-backed rather than proven with varied live data.
- Why it matters: Client finance users, admins, and Bums can still place the same commercial event into the wrong reporting period, which weakens reconciliation, payout trust, invoice aging, and monthly revenue reporting.
- Recommendation: Use business-effective dates for every finance-facing report and export filter, and keep `created_at` or `updated_at` as secondary audit fields. Apply the same rule across admin, client finance, and Bum earnings surfaces.
- Acceptance criteria: Payment views filter by `customer_payment_received_at`; invoice views filter by `paid_at`, `sent_at`, or source payment date fallback logic rather than `generated_at` or `created_at` alone; payout views filter by `paid_at` or `approved_at`; and each finance row includes both effective date and audit timestamp.

### P1 - Split `CLIENT_FINANCE` exports from operational target and meeting exports
- Evidence: `src/App.tsx` still grants `/client/exports` to `CLIENT_ADMIN` and `CLIENT_FINANCE`, and `tests/e2e/authenticated-role-smoke.spec.ts` still expects Client Finance to open that route. `src/pages/client/ClientExports.tsx` exports target rows with `key_contact_email`, and meeting rows with attendee lists and `teams_join_url`, while `docs/business-access-rules.md` clearly grants Client Finance payment and invoice reporting but does not clearly grant those operational exports.
- Why it matters: The current export surface mixes company finance reporting with operational contact and meeting metadata, so Client Finance can receive data that the business rules have not explicitly approved.
- Recommendation: Split finance-safe exports from operational exports, or gate target and meeting exports to `CLIENT_ADMIN` until `docs/business-access-rules.md` explicitly grants broader Client Finance scope. Because this touches route behavior and may require RLS, RPC, or edge-function changes, Security and Lead Developer should validate the final access model before rollout.
- Acceptance criteria: `CLIENT_FINANCE` can export payment and invoice data without target contact emails, attendee lists, join URLs, or transcript-adjacent meeting metadata unless the business rule explicitly grants them; route allow/deny tests cover each export type by client access role.

### P1 - Restrict the admin dashboard summary RPC to admin-only callers
- Evidence: `src/lib/portalApi.ts` calls `rpc("admin_dashboard_summary")` for `src/pages/admin/AdminDashboard.tsx`. Live Supabase security advisors on 2026-05-31 flagged `public.admin_dashboard_summary()` as a `SECURITY DEFINER` function executable by both `anon` and `authenticated`. Live SQL in this run confirms ACLs of `postgres=X/postgres,anon=X/postgres,authenticated=X/postgres,service_role=X/postgres`, while the function itself lives in exposed schema `public` via `supabase/migrations/20260529001000_add_admin_dashboard_summary.sql`. Current Supabase guidance says security-definer functions should not be created in an exposed schema and function execution should be explicitly revoked/granted as needed ([Supabase RLS docs](https://supabase.com/docs/guides/database/postgres/row-level-security), [Supabase function privileges docs](https://supabase.com/docs/guides/database/functions?example-view=sql&language=sql&queryGroups=example-view&queryGroups=language)).
- Why it matters: Marketplace-wide admin aggregates are a reporting surface. Leaving the privileged summary RPC broadly callable weakens reporting access hygiene and makes future analytics changes easier to mis-expose.
- Recommendation: Revoke `EXECUTE` from `anon` and any non-admin-facing roles, and move the privileged logic into a private schema or a tighter admin-only wrapper so the dashboard uses an explicitly admin-scoped path. Security and Lead Developer should validate the final RPC/service-role pattern against `docs/business-access-rules.md` before implementation.
- Acceptance criteria: Supabase security advisors no longer flag `admin_dashboard_summary`; anonymous and non-admin callers cannot execute the summary RPC; admin dashboard summary cards still load successfully for admins.

### P1 - Replace 50-row admin email reporting caps and stop treating opens as a primary KPI
- Evidence: `supabase/functions/send-admin-email/index.ts` still limits `list_deliveries`, `list_engagement`, and `list_campaigns` to 50 rows. Live Supabase aggregates in this run show 27 `admin_email_deliveries`, 20 `admin_email_events`, and an event mix of 19 `OPEN` versus 1 `CLICK`. The current `public.admin_email_engagement_summary` view groups performance by recipient email/profile and counts opens/clicks, while live SQL shows 7 of 27 deliveries lack a `recipient_profile_id`, which weakens role- or company-level reporting dimensions. Apple’s current Mail Privacy Protection guidance says senders are prevented from seeing whether an email was opened, and the ICO’s current storage-and-access technologies guidance covers tracking pixels and similar technologies ([Apple Support](https://support.apple.com/guide/icloud/use-mail-privacy-protection-mm90f7d05c96/icloud), [ICO](https://ico.org.uk/about-the-ico/media-centre/news-and-blogs/2026/04/final-storage-and-access-technologies-guidance-published/)).
- Why it matters: Silent row truncation leaves operators with incomplete history, open-heavy reporting can overstate campaign success, and missing recipient identity links make role-specific operational reporting less trustworthy.
- Recommendation: Replace fixed 50-row reads with pagination or admin-safe aggregates, label opens as directional only, and strengthen recipient dimensions by preserving role/company context or reliably linking profile ids where that business relationship exists.
- Acceptance criteria: Admin email views can page or summarize beyond 50 rows; KPI labels distinguish directional opens from reliable outcomes such as sends, failures, suppressions, clicks, and downstream portal actions; and recipient-level reports preserve enough identity context to break down results by role or company when appropriate.

### P2 - Normalize SPA performance telemetry route attribution before using dashboard comparisons
- Evidence: `src/components/PerformanceMonitoring.tsx` sends `location.pathname` with each metric, while `src/App.tsx` routes authenticated users through `/dashboard` and immediately redirects them by role. Live Supabase aggregates show `performance_metric_events` now has 1,806 total rows, with 7-day samples split across `/dashboard` and the role-specific dashboards (`/client/dashboard`, `/bum/dashboard`). Current web.dev guidance says Core Web Vitals in SPAs are associated with the initiating top-level navigation URL rather than later soft-route transitions, which makes redirect-heavy attribution easy to fragment if the reporting dimension is not normalized ([web.dev](https://web.dev/articles/vitals-spa-faq)).
- Why it matters: Admin performance reporting can compare the same landing workflow under multiple paths, which weakens route-level trend analysis and makes dashboard regressions harder to trust.
- Recommendation: Capture both the initial path and the resolved role route, or normalize known redirect aliases during ingestion/reporting. Document which field `AdminPerformanceMetrics` uses so operators know whether they are reading raw navigation paths or canonical portal routes. Keep raw telemetry admin-only.
- Acceptance criteria: Redirect-driven dashboard landings no longer appear as competing routes in admin summaries unless both dimensions are explicitly shown; seeded `/dashboard` sign-ins roll up under one canonical route definition; and the reporting doc or UI labels explain the attribution model.

## Business Access And Reporting Rules

- Finance business dates: Admins, Client Admins, Client Finance, and Bums need business-effective dates for reconciliation, period filtering, invoice aging, and payout reporting. Non-finance client roles should not gain finance access as part of this work. Sensitivity: financial reporting accuracy. Verification: seeded late-entry scenarios prove effective dates drive filters while audit timestamps remain visible.
- Client Finance export scope: Client Finance needs company-scoped payment reports, invoice registers, finance dashboards, and finance exports. Client Finance should not receive target contact emails, attendee lists, join URLs, transcript-adjacent metadata, or other operational target-management detail unless a business rule explicitly grants that scope. Sensitivity: financial plus operational contact data. Verification: export-content assertions and route allow/deny tests by client access role.
- Admin aggregate reporting RPCs: Admins need marketplace-wide aggregate counts for operational dashboards. Public visitors, Client roles, and Bums should not call privileged summary RPCs or receive marketplace-wide counts through exposed APIs. Sensitivity: admin-only operational aggregate reporting. Verification: grant review, advisor cleanup, and negative RPC tests for `anon`, `authenticated`, and non-admin signed-in roles.
- Admin email analytics: Admins need recipient-level delivery and engagement operations data for troubleshooting and campaign operations. Non-admin roles should not read raw delivery history, recipient behavior rows, or raw engagement telemetry. Sensitivity: recipient operations data and messaging behavior. Verification: admin-only access tests for campaigns, deliveries, and engagement surfaces, plus review of any new aggregate views.
- Performance telemetry: Admins need raw operational telemetry for troubleshooting. Non-admin roles should not read raw event rows, per-path operational telemetry, origins, or hashed device/browser dimensions; any broader sharing should be aggregate-only. Sensitivity: admin-only operational telemetry. Verification: RLS checks on `performance_metric_events`, admin-only UI access, and explicit labels for any aggregate-only downstream view.
