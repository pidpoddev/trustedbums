# Trusted Bums Data Analytics Backlog

_Last updated: 2026-06-07 by Codex daily data analytics engineer automation._

## Executive Read

Trusted Bums still has four active analytics and reporting gaps with current evidence: finance workspaces still present date filters on audit timestamps instead of business-effective dates, `CLIENT_FINANCE` still reaches mixed operational CSV exports, the admin dashboard summary RPC is still exposed more broadly than the business rule allows, and the admin performance dashboard still computes p75 from a `500`-row browser slice even though live telemetry has grown to `33,397` rows across `49` routes. This run had stronger live Supabase evidence than the previous backlog refresh because advisors, table inventory, and recent logs were callable, but read-only SQL and policy-catalog queries were not.

## Active Recommendations

### P1 - Finish business-effective dates in finance report workspaces
- Evidence: `src/pages/client/ClientReports.tsx` still uses `dateKey: "createdAt"` for the `CLIENT_FINANCE` customer-payment ledger, revenue-by-customer, and finance-exceptions reports, and `dateKey: "generatedAt"` for invoice register and invoice aging. `src/pages/admin/AdminReports.tsx` still groups the finance-operations report on `createdAt`. `src/pages/bum/BumReports.tsx` still keys payout reporting to `createdAt`. The shared helpers in `src/lib/portalApi.ts` already define `paymentReportBusinessDate()`, `claimInvoiceBusinessDate()`, and `bumPayoutBusinessDate()`.
- Why it matters: Client Finance users, admins, and Bums can still place the same commercial event in the wrong reporting period even though the data layer already carries better business dates for reconciliation, invoice aging, and payout reporting.
- Recommendation: Re-key finance report workspaces and report filters to business-effective dates: `customer_payment_received_at` for Customer Payment Reports, `paid_at` or `sent_at` with payment fallback for commission invoices, and `paid_at` or `approved_at` for Bum payouts. Keep `created_at` visible only as an audit field.
- Acceptance criteria: Client, admin, and Bum finance report filters and default sort order follow business-effective dates; each row still exposes the audit timestamp separately; and seeded late-entry scenarios land in the expected reporting period.

### P1 - Split `CLIENT_FINANCE` exports from operational target and meeting exports
- Evidence: `src/App.tsx` still grants `/client/exports` to both `CLIENT_ADMIN` and `CLIENT_FINANCE`. `src/pages/client/ClientExports.tsx` still exports `key_contact_email`, meeting attendee lists, `teams_join_url`, and `transcript_sync_status` alongside finance-safe payment rows. `docs/business-access-rules.md` already says Client Finance must not receive target contacts, meeting attendee lists, Teams join URLs, transcripts, or other operational workflow detail by default.
- Why it matters: The current export route still bundles finance reporting with operational contact and meeting context, which breaks the documented finance-only data posture and creates a direct CSV egress path for fields finance users are not approved to handle.
- Recommendation: Split `/client/exports` into finance-safe and operational export types, or gate target and meeting exports to `CLIENT_ADMIN` until a narrower business rule explicitly grants extra Client Finance scope. Because this may require route, RLS, RPC, or edge-function changes, Security and Lead Developer should validate the final design against `docs/business-access-rules.md` before rollout.
- Acceptance criteria: `CLIENT_FINANCE` can export company payment and invoice data without target contact emails, attendee lists, join URLs, transcript-adjacent metadata, or other operational detail unless a documented rule explicitly allows it; route allow/deny tests cover each export type by client role.

### P1 - Restrict the admin dashboard summary RPC to admin-only callers
- Evidence: `src/lib/portalApi.ts` still calls `rpc("admin_dashboard_summary")`, and `supabase/migrations/20260529001000_add_admin_dashboard_summary.sql` still defines `public.admin_dashboard_summary()` as `SECURITY DEFINER` in the exposed `public` schema. Live Supabase security advisors on 2026-06-07 flagged `public.admin_dashboard_summary()` as executable by both `anon` and `authenticated` and linked the issue to the Data API exposure path. Current Supabase guidance says Data API access is controlled by grants, recommends least-privilege `EXECUTE` grants, and notes that `SECURITY DEFINER` helpers used for RLS do not need to stay in exposed schemas when explicitly schema-qualified ([Securing your API](https://supabase.com/docs/guides/api/securing-your-api), [Row Level Security](https://supabase.com/docs/guides/database/postgres/row-level-security), [security-definer troubleshooting, last edited 2026-04-08](https://supabase.com/docs/guides/troubleshooting/do-i-need-to-expose-security-definer-functions-in-row-level-security-policies-iI0uOw)).
- Why it matters: Marketplace-wide dashboard aggregates are an admin-only reporting surface. The live advisor result means this is not just a source-level concern: the exposed helper is still broader than the business rule allows.
- Recommendation: Revoke broad API execution, move the helper behind a non-exposed schema or an admin-only wrapper, and add negative access tests for signed-out and signed-in non-admin callers. Security and Lead Developer should validate the final RPC or service-role pattern against `docs/business-access-rules.md` before implementation.
- Acceptance criteria: `anon`, generic `authenticated`, Client, and Bum callers cannot execute the summary RPC; admins still receive the dashboard summary cards; and the final implementation uses explicit admin-only grants rather than exposed broad-role execution.

### P1 - Replace raw-row admin performance reporting with aggregate-first summaries
- Evidence: `src/pages/admin/AdminPerformanceMetrics.tsx` still requests `listPerformanceMetricEvents({ ..., limit: 500 })` and computes p75 in the browser, while `src/lib/portalApi.ts` still reads raw `performance_metric_events` rows directly. Live Supabase table inventory on 2026-06-07 shows `public.performance_metric_events` at `33,397` rows, and recent edge-function logs show continuing `performance-beacon` `POST 202` traffic. `docs/business-access-rules.md` already says raw telemetry is admin-only and any broader reporting should use aggregate-only summaries.
- Why it matters: The current admin dashboard cannot be a trustworthy telemetry prioritization surface when it samples only the most recent `500` rows from a table that is already two orders of magnitude larger. It also keeps raw telemetry on the browser path where the business rule already points toward aggregate-only sharing.
- Recommendation: Move `/admin/performance` to server-shaped aggregate summaries by metric, route, time window, and rating buckets, and keep raw row access as a narrower drill-down path only when needed for admin troubleshooting. Data/Analytics, Security, and Lead Developer should validate the summary path against the existing telemetry access rule before rollout.
- Acceptance criteria: Admin performance cards and p75 values come from server-side aggregates rather than browser math over capped raw rows; the page can represent the full selected time window without a `500`-row ceiling; and non-admin roles remain denied on both the route and any supporting summary helper.

## Business Access And Reporting Rules

- Finance business dates: Admins, Client Admins, Client Finance, and Bums need business-effective dates for reconciliation, period filtering, invoice aging, and payout reporting. Non-finance client roles should not gain finance visibility as part of this work. Sensitivity: financial reporting accuracy. Verification: seeded late-entry scenarios prove filters use effective dates while `created_at` remains an audit field.
- Client Finance exports: Client Finance needs company-scoped payment reports, invoice registers, finance dashboards, and finance exports. Client Finance should not receive target contact emails, attendee lists, join URLs, transcript-adjacent metadata, or other operational workflow detail unless `docs/business-access-rules.md` is explicitly expanded. Sensitivity: financial plus operational contact data. Verification: role-by-role export content assertions and route allow/deny tests.
- Admin aggregate reporting RPCs: Admins need marketplace-wide operational aggregates for dashboard cards. Public visitors, Client roles, and Bums should not execute privileged summary RPCs or receive marketplace-wide counts through exposed APIs. Sensitivity: admin-only aggregate operational reporting. Verification: live advisor cleanup plus negative RPC tests for `anon`, `authenticated`, Client, and Bum callers.
- Admin performance telemetry summaries: Admins need route-level performance aggregates for troubleshooting. Non-admin roles should not read raw telemetry rows or admin observability summaries, and any shared reporting should stay aggregate-only. Sensitivity: admin-only operational telemetry. Verification: route allow/deny tests plus seeded aggregate checks that prove the summary path covers the full selected time window.

Keep this as an access-safety watch item until direct policy validation is available again.
