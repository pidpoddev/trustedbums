# Trusted Bums Data Analytics Backlog

_Last updated: 2026-06-04 by Codex daily data analytics engineer automation._

## Executive Read

Trusted Bums still has five active analytics and reporting gaps with direct current evidence. The highest-leverage items remain finance period correctness, Client Finance export overexposure versus the documented business rule, admin-only observability access around `admin_dashboard_summary()`, missing admin governance reporting for access requests and terms deferrals, and weak admin performance telemetry summaries caused by raw SPA paths plus a 500-row query cap. Live Supabase verification was partial but real this run: project metadata, deployed function inventory, and recent edge/postgres logs were available; direct SQL, row counts, view definitions, grants, RLS catalog, and advisor output were not callable from the available tool surface.

## Active Recommendations

### P1 - Finish business-effective finance dates across reports
- Evidence: `src/lib/portalApi.ts` still defines `paymentReportBusinessDate`, `claimInvoiceBusinessDate`, and `bumPayoutBusinessDate`, but the report builders still group/filter on audit timestamps: `src/pages/client/ClientReports.tsx` uses `dateKey: "createdAt"` for the payment ledger, revenue-by-customer, and finance exceptions plus `dateKey: "generatedAt"` for invoice views; `src/pages/admin/AdminReports.tsx` still keys finance operations to `createdAt`; `src/pages/bum/BumReports.tsx` still keys payout reporting to `createdAt`.
- Why it matters: Finance users, admins, and Bums can still land the same commercial event in the wrong reporting period, which weakens reconciliation, invoice aging, payout trust, and period-based revenue reporting.
- Recommendation: Make business-effective dates the primary reporting date on every finance-facing report and export. Keep `created_at` only as a secondary audit field, and apply the same date semantics across admin, client finance, and Bum earnings surfaces.
- Acceptance criteria: Customer payment reporting filters by `customer_payment_received_at`; invoice reporting filters by `paid_at`, `sent_at`, or documented source-payment fallback logic instead of `generated_at` alone; payout reporting filters by `paid_at` or `approved_at`; and finance rows show both effective date and audit timestamp.

### P1 - Enforce finance-safe Client Finance exports
- Evidence: `src/App.tsx` and `src/layouts/ClientLayout.tsx` still grant `/client/exports` to `CLIENT_ADMIN` and `CLIENT_FINANCE`. `src/pages/client/ClientExports.tsx` still exports target rows with `key_contact_email` and meeting rows with attendee lists, `teams_join_url`, and `transcript_sync_status`. `docs/business-access-rules.md` already says Client Finance must be denied target contact emails, attendee lists, Teams join URLs, transcripts, and other operational workflow details unless explicitly granted.
- Why it matters: The implemented export surface still violates the current business-access rule, mixing finance reporting with operational target and meeting data that Client Finance is not supposed to receive by default.
- Recommendation: Split `/client/exports` into finance-safe and operational exports, or gate operational exports to `CLIENT_ADMIN` until a narrower finance exception is explicitly approved. Because this can require route, RLS, RPC, or edge-function changes, Security and Lead Developer should validate the final design against `docs/business-access-rules.md` before rollout.
- Acceptance criteria: `CLIENT_FINANCE` can export company payment and invoice data without target contact emails, attendee lists, join URLs, transcript-adjacent metadata, or other operational workflow fields; `CLIENT_ADMIN` keeps the intended operational exports; and route/data-path allow-deny tests cover each export type by client role.

### P1 - Move admin dashboard summary off a broadly callable exposed RPC
- Evidence: `src/lib/portalApi.ts` still calls `rpc("admin_dashboard_summary")` for `src/pages/admin/AdminDashboard.tsx`. `supabase/migrations/20260529001000_add_admin_dashboard_summary.sql` still creates `public.admin_dashboard_summary()` as `SECURITY DEFINER`, revokes `public`, then grants execute to `authenticated`. Current Supabase guidance says `SECURITY DEFINER` functions should not live in exposed schemas and API access should be controlled with least-privilege execute grants and careful review of privileged functions ([RLS docs](https://supabase.com/docs/guides/database/postgres/row-level-security), [security-definer troubleshooting](https://supabase.com/docs/guides/troubleshooting/do-i-need-to-expose-security-definer-functions-in-row-level-security-policies-iI0uOw), [API security](https://supabase.com/docs/guides/api/securing-your-api)).
- Why it matters: Marketplace-wide admin aggregates are a reporting surface. Leaving the summary helper callable to every authenticated browser session weakens observability access hygiene and increases the chance of future aggregate leakage.
- Recommendation: Revoke execute from non-admin callers and move the privileged summary logic behind a private-schema helper, admin-only wrapper, or service-role-backed server path. Security and Lead Developer should validate the final reporting path against `docs/business-access-rules.md` before implementation.
- Acceptance criteria: Non-admin callers cannot execute the summary helper; privileged logic no longer relies on an exposed broadly callable `public` RPC; and admin dashboard summary cards still load for admins.

### P1 - Add admin-only aging and outcome reporting for access requests and terms deferrals
- Evidence: `supabase/migrations/20260531143000_harden_profile_bootstrap.sql` adds `client_company_access_requests` with `request_type`, `status`, `evidence`, `reviewed_at`, and `review_note`. `supabase/migrations/20260531195900_add_terms_acceptance_deferrals.sql` adds `terms_acceptance_deferrals` with a three-deferral guard. The workflow paths exist in `supabase/functions/admin-access-requests/index.ts`, `supabase/functions/profile-bootstrap/index.ts`, and `supabase/functions/client-team/index.ts`, and `src/pages/admin/AdminClients.tsx` now exposes an admin review queue. But the current admin UI is still action-oriented only: the access-review card shows pending requests with approve/deny buttons, while `src/pages/admin/AdminDashboard.tsx`, `src/pages/admin/AdminReports.tsx`, and the export/report helpers still provide no age buckets, approval-rate trends, reviewer throughput, or deferral-limit reporting.
- Why it matters: Trusted Bums now records role-sensitive onboarding and contract-deferral events but still lacks the admin reporting needed to spot stuck approvals, repeated deferrals, and policy bottlenecks before they hurt conversion, compliance, or support operations.
- Recommendation: Add an admin-only governance report for access requests and terms deferrals with pending-vs-reviewed counts, age buckets, outcomes by request type and company, reviewer timestamps, and current-terms deferral counts. Keep raw evidence, review notes, and detailed deferral history admin-only unless a narrower same-company operational view is explicitly approved. Because this may require new RLS, RPC, or service-role reporting paths, Security and Lead Developer should validate the design against `docs/business-access-rules.md`.
- Acceptance criteria: Admins can review pending and historical access-request outcomes with age buckets and reviewer timestamps; admins can see who or which companies are approaching the three-deferral limit for the current terms version; Client Admins see only same-company operational items they are allowed to act on; Client Finance, Bums, and public users cannot read raw request evidence, review notes, or deferral histories.

### P2 - Fix admin performance telemetry attribution and sample bias
- Evidence: `src/components/PerformanceMonitoring.tsx` still sends only `location.pathname`, so SPA route detail is stored as raw path with no route classification or route-change semantics. `src/pages/admin/AdminPerformanceMetrics.tsx` summarizes route counts and p75 values from `listPerformanceMetricEvents({ days, metricName, rating, limit: 500 })`, and `src/lib/portalApi.ts` hard-caps that query to 1000 rows. Live Supabase edge-function logs from 2026-06-04 show repeated `performance-beacon` `POST | 202` events, so the telemetry path is active now and the admin screen is summarizing a capped subset rather than the full window. Current web.dev SPA guidance still notes that SPA route changes need intentional handling when using Core Web Vitals for route-level reporting ([web.dev SPA/Core Web Vitals FAQ](https://web.dev/articles/vitals-spa-faq?hl=en)).
- Why it matters: Admin observability can now ingest data, but the current report can misstate unique-route counts and p75 values for busy periods because it mixes raw SPA paths with a capped sample instead of a route-aware aggregate.
- Recommendation: Add a normalized route key or route template to the beacon payload, and back the admin screen with aggregate queries or paginated retrieval sized for the selected window instead of a fixed 500-row client summary. Keep this admin-only and validate any new aggregate helper against the existing observability access rule.
- Acceptance criteria: Performance telemetry stores a stable route grouping field; `/admin/performance` can summarize the selected window without depending on an arbitrary 500-row subset; and admins can distinguish raw sample review from rolled-up route-level KPIs.

## Business Access And Reporting Rules

- Finance period reporting: Admins, Client Admins, Client Finance, and Bums need business-effective dates for reconciliation, period filtering, invoice aging, and payout reporting. Non-finance client roles should not gain finance access as part of this work. Sensitivity: financial reporting accuracy. Verification: seeded late-entry scenarios prove effective dates drive filters while audit timestamps remain visible.
- Client Finance exports: The current business rule in `docs/business-access-rules.md` already denies operational target and meeting exports to Client Finance by default. This backlog item should be implemented as a code-and-test alignment to that existing rule, not as a new permission expansion. Sensitivity: finance plus operational contact/meeting data. Verification: export-content assertions and route/data-path allow-deny tests by client role.
- Admin aggregate observability: Admins need marketplace-wide summary counts for dashboard and troubleshooting views. Client roles, Bums, and public users must not call privileged summary RPCs or receive admin-only aggregate counts through exposed browser paths. Sensitivity: admin-only operational aggregate reporting. Verification: grant review plus negative RPC tests for non-admin roles.
- Access-governance reporting: Admins need raw access-request outcomes, reviewer timestamps, review notes, and terms-deferral limit visibility for onboarding governance. Client Admins may need only same-company pending requests they are explicitly allowed to review. Client Finance, Bums, and public users must not read raw request evidence, review notes, or deferral histories. Sensitivity: identity, access-governance, and contract-compliance behavior. Verification: role-by-role report tests and explicit business-rule coverage before any new reporting path ships.
- Performance telemetry and observability: Public/front-end runtimes may write only the intended telemetry payload; Admins may read raw telemetry and route aggregates; non-admin portal roles must not read `performance_metric_events`, `/admin/performance`, or any future aggregate helper. Sensitivity: operational telemetry and internal system-state reporting. Verification: positive write tests for approved origins plus route and direct data-path deny tests for non-admin roles.
