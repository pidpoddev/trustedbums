## Active Recommendations

### P1 - [TB-0044] Split Client Finance reporting into a finance-safe read model
- Evidence: Exact head `af944fe` still sends both Client Admin and Client Finance through the same `listCustomerPaymentReports()` and `listClaimInvoices()` reads in `src/lib/portalApi.ts`, including `opportunity_claims(contact_name, contact_company, bum_user_id)` and reporter `profiles(full_name, email)` joins. The committed business contract in `docs/business-access-rules.md` says Client Finance should receive finance-safe rows that omit that operational context, but the shared read path has not changed. Live production still has `0` `customer_payment_reports` rows and `0` `claim_invoices` rows on 2026-06-17, while GA now shows the affected surfaces are in use over the last 7 days: `client_payments` `118` events / `65` active users, `client_reports` `76` / `43`, and `client_exports` `70` / `43`.
- Why it matters: The current UI models and export shapes are safer than the old route set, but the browser payload still hydrates data the finance role does not need. The new GA route evidence means this is no longer a dormant surface; once live finance rows appear, the over-broad payload becomes immediately user-facing on already active routes.
- Recommendation: Create a dedicated finance-safe projection for Client Finance payment and invoice reads, either with role-aware selects or a dedicated RPC/view, and route Client Finance dashboard, payments, reports, and exports through that projection. Keep the richer operational joins only for Admin and Client Admin workflows that actually need them.
- Acceptance criteria: Client Finance payloads omit contact names, contact companies, Bum IDs, and unrelated reporter identity; Client Admin and Admin retain richer fields only where required; response-shape tests prove the finance-safe contract; and direct data-path checks deny non-company and non-finance roles.

### P2 - [TB-0046] Make admin email metrics aggregate-first and stop treating opens as decision-grade
- Evidence: Exact head `af944fe` and the deployed `send-admin-email` Edge Function `v8` still cap `list_deliveries`, `list_engagement`, and `list_campaigns` at `50` rows each, while `src/pages/admin/AdminEmails.tsx` still builds the headline engagement card from `deliveries.filter((delivery) => delivery.opened_at)` and `deliveries.filter((delivery) => delivery.clicked_at)`. Live production now shows `33` deliveries, `22` campaigns, `31` events, `30` `OPEN` events, and `1` `CLICK` event on 2026-06-17. Live `information_schema.columns` still shows `admin_email_engagement_summary` is recipient-level, not aggregate-only, and the current Supabase performance advisors still flag unindexed admin-email foreign keys including `admin_email_campaigns.{created_by,template_id}`, `admin_email_deliveries.{campaign_id,created_by,recipient_profile_id,template_id}`, and `admin_email_events.{delivery_id,recipient_profile_id}`. Apple still documents Mail Privacy Protection as preventing senders from seeing whether a recipient opened an email, and Supabase still documents unindexed foreign keys as a join-performance risk.
- Why it matters: The admin email dashboard will turn sample-based once activity grows past the hard cap, and its most visible engagement number still leans on the least trustworthy signal in the dataset. The current live schema posture also means the same reporting joins will get more expensive as campaign activity grows.
- Recommendation: Add aggregate KPI helpers for admin email campaigns and engagement, move the headline cards to those aggregates, paginate or filter raw delivery and campaign tables, relabel opens as proxy signals, and add or explicitly waive the missing admin-email foreign-key indexes once the final aggregate query shape is locked.
- Acceptance criteria: Admin email summary cards read aggregate data instead of capped lists; raw delivery and campaign tables support bounded navigation; open metrics are labeled as proxy or partial signals; the shared KPI surface avoids recipient-level rows; and the next SQL-backed review confirms either new indexes or an explicit waiver for the flagged admin-email joins used by the final query shape.

## Business Access And Reporting Rules

- Client Finance payment and invoice reporting: Current-head `docs/business-access-rules.md` still contains the correct finance-safe rule, but the current `src/lib/portalApi.ts` read path still violates it by hydrating operational relationship and reporter fields for Client Finance. Roles that need the data: Client Finance, Client Admin, and Admin. Roles that should not receive the raw operational context: Client Finance, other companies, Bums, and Public Visitors. Sensitivity: financial plus operational relationship context. Implementation follow-up should add a finance-safe read model or role-aware projection that matches the committed rule and validate it with response-shape plus allow/deny tests.
- Admin email reporting: Current-head `docs/business-access-rules.md` still contains the admin-only raw reporting rule plus the aggregate-only exception path. Roles that need raw data: Admin and approved internal automation only. Roles that should not see raw rows: Client Admins, Client Finance, Client Members, Bums, and Public Visitors. Sensitivity: recipient identity, message content, click behavior, and suppression state. No new rule text is needed right now; the remaining work is to implement aggregate-first KPI helpers that follow the existing rule instead of relying on recipient-level summary rows.
- Claim client notification previews: Current head adds `public.claim_client_notification_previews`, a sanitized latest-email preview view intended for Admin or same-company claim reviewers, and live security advisors flag it as a `SECURITY DEFINER` view. Roles that need the data: Admin and same-company client claim operators. Roles that should not receive it: other companies, unrelated Bums, Public Visitors, finance-only users, and any aggregate/shared reporting surface. Sensitivity: sanitized but still operational email subject/body context. If this view remains, `docs/business-access-rules.md` should add explicit rule text for the allowed reviewer set and should block broader reporting reuse until Security replaces or clears the live `SECURITY DEFINER` posture.
