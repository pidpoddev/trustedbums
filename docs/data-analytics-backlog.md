### P1 - Redefine admin email analytics around reliable outcomes instead of opens
- Evidence: The deployed `email-track` function records only `OPEN` and `CLICK`, updates `engagement_score` by `1` or `3`, and stores open events without distinguishing privacy-proxy fetches. The deployed `send-admin-email` function still caps list responses at `50` rows for deliveries, campaigns, and engagement reads. [`/Users/ryan.peterson/Documents/Trusted Bums LOCAL/trustedbums/supabase/migrations/20260519023000_add_email_engagement_controls.sql`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/migrations/20260519023000_add_email_engagement_controls.sql) defines additional event types such as `BOUNCE`, `COMPLAINT`, `REPLY`, and `PORTAL_VISIT`, but the active ingestion path does not write them. Apple still states Mail Privacy Protection hides IP information and prevents senders from knowing whether an email was opened ([Apple Support](https://support.apple.com/en-lb/guide/iphone/iphf084865c7/ios)), and the ICO's 2026-04-29 guidance still treats tracking pixels as privacy-relevant storage/access technology ([ICO](https://ico.org.uk/about-the-ico/media-centre/news-and-blogs/2026/04/final-storage-and-access-technologies-guidance-published/)).
- Why it matters: An open-heavy "most engaged" view can mis-rank recipients, hide real delivery problems, and overstate campaign performance while the read APIs silently truncate history.
- Recommendation: Recenter admin email reporting on delivered, failed, suppressed, clicked, bounced, complained, replied, and downstream portal-action metrics; label opens as directional only; and replace fixed 50-row list reads with pagination or aggregate queries. Any new summary views or service-role aggregation paths should be validated against `docs/business-access-rules.md` before implementation.
- Acceptance criteria: Admin email dashboards no longer use opens as the primary KPI, surfaced metrics match real writer paths, list views no longer truncate silently at 50 rows, and metric definitions are documented for operators.

## Business Access And Reporting Rules

- Client Finance export scope: Client Finance needs company-scoped payment reports, invoices, finance dashboards, and finance CSV exports. Client Finance should not see target contact emails, meeting attendee lists, meeting URLs, or other operational workflow detail unless `docs/business-access-rules.md` explicitly grants it. Sensitivity: financial plus operational contact data. Verification: route-guard QA plus RLS/API deny tests for `/client/exports` datasets by role. This likely requires route and possibly RPC/RLS changes, so Security and Lead Developer should validate it.
- Admin performance telemetry: Admins need aggregate operational telemetry for reliability and troubleshooting. Client users, Bums, and public visitors should not read raw `performance_metric_events`. Aggregate reporting can remain admin-only; any future shared dashboard should expose route- or environment-level aggregates only, not raw samples. Sensitivity: operational telemetry with hashed user-agent context. Verification: confirm admin-only reads continue after any schema or view changes and ensure new summary views obey RLS/business rules.
- Admin email analytics: Admins need campaign delivery, suppression, complaint, bounce, click, and downstream action reporting. Client users and Bums should not access raw delivery histories or recipient-level engagement data. Opens should be treated as aggregate-only and directional because of mailbox privacy distortion. Sensitivity: recipient email operations and messaging behavior. Verification: confirm admin-only reads for delivery/engagement paths, document metric definitions in `docs/business-access-rules.md`, and review any new aggregate view or service-role job with Security and Lead Developer.
