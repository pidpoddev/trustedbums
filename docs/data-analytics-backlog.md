_Last updated: 2026-06-08 by Codex._

## Executive Read

Trusted Bums still has two active analytics and reporting gaps with current evidence: finance workspaces still present date filters on audit timestamps instead of business-effective dates, and `CLIENT_FINANCE` still reaches mixed operational CSV exports. The admin performance dashboard now uses aggregate-first reporting for its primary view: metric cards use `admin_performance_metric_summary`, route rows use live migration `20260608020645 add_admin_performance_route_summary`, and the page no longer imports raw `performance_metric_events` for the table. The earlier admin dashboard summary RPC exposure is no longer active: refreshed live Supabase security advisors show only leaked-password protection disabled.

## Active Recommendations

### P1 - Finish business-effective dates in finance report workspaces
- Evidence: `src/pages/client/ClientReports.tsx` still uses `dateKey: "createdAt"` for the `CLIENT_FINANCE` customer-payment ledger, revenue-by-customer, and finance-exceptions reports, and `dateKey: "generatedAt"` for invoice register and invoice aging. `src/pages/admin/AdminReports.tsx` still groups the finance-operations report on `createdAt`. `src/pages/bum/BumReports.tsx` still keys payout reporting to `createdAt`. The shared helpers in `src/lib/portalApi.ts` already define `paymentReportBusinessDate()`, `claimInvoiceBusinessDate()`, and `bumPayoutBusinessDate()`.
- Why it matters: Client Finance users, admins, and Bums can still place the same commercial event in the wrong reporting period even though the data layer already carries better business dates for reconciliation, invoice aging, and payout reporting.
- Recommendation: Re-key finance report workspaces and report filters to business-effective dates: `customer_payment_received_at` for Customer Payment Reports, `paid_at` or `sent_at` with payment fallback for commission invoices, and `paid_at` or `approved_at` for Bum payouts. Keep `created_at` visible only as an audit field.
- Acceptance criteria: Client, admin, and Bum finance report filters and default sort order follow business-effective dates; each row still exposes the audit timestamp separately; and seeded late-entry scenarios land in the expected reporting period.

### P1 - Split `CLIENT_FINANCE` exports from operational target and meeting exports
- Evidence: `src/App.tsx` still grants `/client/exports` to both `CLIENT_ADMIN` and `CLIENT_FINANCE`. `src/pages/client/ClientExports.tsx` still exports `key_contact_email`, meeting attendee lists, `teams_join_url`, and `transcript_sync_status` alongside finance-safe payment rows. `docs/business-access-rules.md` already says Client Finance must not receive target contacts, meeting attendee lists, Teams join URLs, transcripts, or other operational workflow detail by default.
- Why it matters: The current export route still bundles finance reporting with operational contact and meeting context, which breaks the documented finance-only data posture and creates a direct CSV egress path for fields finance users are not approved to handle.
- Recommendation: Split `/client/exports` into finance-safe and operational export types, or gate target and meeting exports to `CLIENT_ADMIN` until a narrower business rule explicitly grants extra Client Finance scope. Because this may require route, RLS, RPC, or edge-function changes, Security and Lead Developer should validate the final design against `docs/business-access-rules.md` before rollout.
- Acceptance criteria: `CLIENT_FINANCE` can export company payment and invoice data without target contact emails, attendee lists, join URLs, transcript-adjacent metadata, or other operational detail unless a documented rule explicitly allows it; route allow/deny tests cover each export type by client role.

### P1 - Admin performance aggregate reporting verified
- Evidence: `src/pages/admin/AdminPerformanceMetrics.tsx` now requests server-shaped metric and route summaries, and `src/test/accessBoundaryRegression.test.ts` prevents the page from importing `listPerformanceMetricEvents`. Live Supabase migration inventory includes `20260608020645 add_admin_performance_route_summary`; generic non-admin SQL context is denied by the helper, while a simulated admin JWT claim returns route aggregate rows with p75 and count fields.
- Why it matters: The admin dashboard can now prioritize telemetry across the selected time window without browser math over capped raw rows.
- Recommendation: Keep this item on watch for current-head authenticated route proof, not active implementation. Any future raw telemetry drill-down should be explicitly admin-only, bounded, and separate from the aggregate landing view.
- Acceptance criteria: Met for source, live helper deployment, admin-only helper enforcement, and local QA. Remaining proof: current-head hosted `/admin/performance` route smoke or browser walkthrough after deployment.

## Business Access And Reporting Rules

- Finance business dates: Admins, Client Admins, Client Finance, and Bums need business-effective dates for reconciliation, period filtering, invoice aging, and payout reporting. Non-finance client roles should not gain finance visibility as part of this work. Sensitivity: financial reporting accuracy. Verification: seeded late-entry scenarios prove filters use effective dates while `created_at` remains an audit field.
- Client Finance exports: Client Finance needs company-scoped payment reports, invoice registers, finance dashboards, and finance exports. Client Finance should not receive target contact emails, attendee lists, join URLs, transcript-adjacent metadata, or other operational workflow detail unless `docs/business-access-rules.md` is explicitly expanded. Sensitivity: financial plus operational contact data. Verification: role-by-role export content assertions and route allow/deny tests.
- Admin aggregate reporting RPCs: Admins need marketplace-wide operational aggregates for dashboard cards. Public visitors, Client roles, and Bums should not execute privileged summary RPCs or receive marketplace-wide counts through exposed APIs. Sensitivity: admin-only aggregate operational reporting. Current status: live helper/RPC advisors are cleared after migration `20260607235839`; remaining work is seeded negative RPC/route proof for `anon`, Client, and Bum callers.
- Admin performance telemetry summaries: Admins need route-level performance aggregates for troubleshooting. Non-admin roles should not read raw telemetry rows or admin observability summaries, and any shared reporting should stay aggregate-only. Sensitivity: admin-only operational telemetry. Verification: route allow/deny tests plus seeded aggregate checks that prove the summary path covers the full selected time window.

Keep this as an access-safety watch item until direct policy validation is available again.
