# Trusted Bums Data Analytics Backlog

_Last updated: 2026-06-03 by Codex daily data analytics engineer automation._

## Executive Read

Trusted Bums still has five active analytics/reporting gaps with direct current evidence: finance reports still key off audit timestamps instead of business-effective dates, `CLIENT_FINANCE` still has access to mixed operational and finance exports, the admin dashboard still depends on an exposed `SECURITY DEFINER` summary RPC, access-request and terms-deferral workflows still lack admin-grade aging and outcome reporting, and admin email analytics still cap core reads at 50 rows while leaning on open data that is no longer trustworthy as a primary KPI. Live Supabase verification was partial this run: project and function inventory were available, but deeper project actions already required reauthentication and direct SQL/RLS inspection was not callable from the available tool surface.

## Active Recommendations

### P1 - Finish business-effective finance dates across reports and exports
- Evidence: `src/lib/portalApi.ts` still defines `paymentReportBusinessDate`, `claimInvoiceBusinessDate`, and `bumPayoutBusinessDate`, and the list helpers already sort with them. `src/pages/client/ClientReports.tsx` still uses `dateKey: "createdAt"` for the customer payment ledger, revenue-by-customer, and finance exceptions, plus `dateKey: "generatedAt"` for invoice views. `src/pages/admin/AdminReports.tsx` still groups the finance operations report on `createdAt`, and `src/pages/bum/BumReports.tsx` still keys payout reporting to `createdAt`.
- Why it matters: Finance users, admins, and Bums can still land the same commercial event in the wrong reporting period, which weakens reconciliation, invoice aging, payout trust, and revenue reporting.
- Recommendation: Use business-effective dates for every finance-facing report and export filter, and keep `created_at` only as a secondary audit field. Apply the same logic across admin, client finance, and Bum earnings/reporting surfaces.
- Acceptance criteria: Payment reports filter by `customer_payment_received_at`; invoice reports filter by `paid_at`, `sent_at`, or source payment fallback logic rather than `generated_at` alone; payout reports filter by `paid_at` or `approved_at`; and each finance row shows both effective date and audit timestamp.

### P1 - Split `CLIENT_FINANCE` exports from operational target and meeting exports
- Evidence: `src/App.tsx` still grants `/client/exports` to `CLIENT_ADMIN` and `CLIENT_FINANCE`. `src/pages/client/ClientExports.tsx` still exports target rows with `key_contact_email`, and meeting rows with attendee lists, `teams_join_url`, and `transcript_sync_status`. `tests/e2e/authenticated-role-smoke.spec.ts` still treats client-finance access to `/client/exports` as expected behavior.
- Why it matters: The current export surface mixes finance reporting with operational contact and meeting data, so Client Finance can receive company data that the current business rules do not explicitly grant.
- Recommendation: Split finance-safe exports from operational exports, or gate target and meeting exports to `CLIENT_ADMIN` until `docs/business-access-rules.md` explicitly grants broader Client Finance scope. Because this touches route behavior and may require RLS, RPC, or edge-function changes, Security and Lead Developer should validate the final access model against `docs/business-access-rules.md` before rollout.
- Acceptance criteria: `CLIENT_FINANCE` can export payment and invoice data without target contact emails, attendee lists, join URLs, transcript-adjacent metadata, or other operational target-management fields unless the business rule explicitly grants that scope; route allow/deny tests cover each export type by client access role.

### P1 - Restrict the admin dashboard summary RPC to admin-only callers
- Evidence: `src/lib/portalApi.ts` still calls `rpc("admin_dashboard_summary")` for `src/pages/admin/AdminDashboard.tsx`. `supabase/migrations/20260529001000_add_admin_dashboard_summary.sql` still creates the function as `SECURITY DEFINER` in exposed schema `public` and grants execute to `authenticated`. Current Supabase guidance says `SECURITY DEFINER` helpers should not be created in exposed schemas and API access should be controlled by explicit execute grants and least privilege ([RLS docs](https://supabase.com/docs/guides/database/postgres/row-level-security), [Supabase troubleshooting](https://supabase.com/docs/guides/troubleshooting/do-i-need-to-expose-security-definer-functions-in-row-level-security-policies-iI0uOw), [Supabase API security](https://supabase.com/docs/guides/api/securing-your-api)).
- Why it matters: Marketplace-wide admin aggregates are a reporting surface. Leaving the privileged summary RPC broadly callable weakens reporting access hygiene and raises the chance of future aggregate leakage.
- Recommendation: Revoke execute from non-admin callers, and move the privileged logic behind a private-schema or explicitly admin-only wrapper. Security and Lead Developer should validate the final RPC or service-role pattern against `docs/business-access-rules.md` before implementation.
- Acceptance criteria: Non-admin callers cannot execute the summary RPC; the helper no longer lives in an exposed schema or no longer has broad execute grants; and admin dashboard summary cards still load successfully for admins.

### P1 - Add admin-only aging and outcome reporting for access requests and terms deferrals
- Evidence: `supabase/migrations/20260531143000_harden_profile_bootstrap.sql` adds `client_company_access_requests` with `request_type`, `status`, `evidence`, `reviewed_at`, and `review_note`. `supabase/migrations/20260531195900_add_terms_acceptance_deferrals.sql` adds `terms_acceptance_deferrals` with a three-skip guard. `supabase/functions/admin-access-requests/index.ts`, `supabase/functions/profile-bootstrap/index.ts`, and `supabase/functions/client-team/index.ts` create and review these records. Current UI coverage is still workflow-only: `src/pages/admin/AdminClients.tsx` loads admin access requests, `src/pages/client/ClientTeam.tsx` only exposes same-company request review, and I did not find aging or outcome reporting for either object in `src/pages/admin/AdminDashboard.tsx`, `src/pages/admin/AdminReports.tsx`, or export/report helpers.
- Why it matters: Trusted Bums now has role-sensitive onboarding and terms-deferral controls, but operators still lack trend, aging, approval-rate, and repeat-deferral visibility. That weakens operational decision support, auditability, and the ability to spot stuck access workflows before they affect conversion or support load.
- Recommendation: Add an admin-only access governance report that covers pending vs reviewed access requests, age buckets, approval/denial outcomes by request type and company, and terms-deferral counts by current terms version. Keep raw request evidence and deferral history admin-only unless a narrower same-company operational view is explicitly approved. Because this may require new RLS, RPC, or edge-function reporting paths, Security and Lead Developer should validate the final design against `docs/business-access-rules.md`.
- Acceptance criteria: Admins can review pending and historical access-request outcomes with age buckets and reviewer timestamps; admins can see deferral counts and users or companies approaching the three-skip limit; Client Admins see only same-company operational requests they are allowed to review; Client Finance and Bums cannot read raw request evidence, review notes, or deferral histories.

### P2 - Replace 50-row admin email caps and stop treating opens as a primary KPI
- Evidence: `supabase/functions/send-admin-email/index.ts` still limits `list_deliveries`, `list_engagement`, and `list_campaigns` to 50 rows, and `src/pages/admin/AdminEmails.tsx` still reads those capped operations directly. Current Apple guidance says Mail Privacy Protection prevents senders from seeing whether an email was opened, and the ICO published final storage-and-access technologies guidance on 2026-04-29 confirming tracking pixels remain inside current compliance scope ([Apple Support](https://support.apple.com/guide/iphone/use-mail-privacy-protection-iphf084865c7/26/ios/26), [ICO news release](https://ico.org.uk/about-the-ico/media-centre/news-and-blogs/2026/04/ico-publishes-final-storage-and-access-technologies-guidance/), [ICO guidance overview](https://ico.org.uk/for-organisations/direct-marketing-and-privacy-and-electronic-communications/guidance-on-the-use-of-storage-and-access-technologies/what-are-storage-and-access-technologies/)).
- Why it matters: Silent row truncation leaves operators with incomplete delivery history, and open-heavy reporting can overstate campaign success or create privacy-sensitive interpretations of a weak signal.
- Recommendation: Replace fixed 50-row reads with pagination or admin-safe aggregates, and relabel opens as directional only. Promote sends, failures, suppressions, clicks, and downstream portal actions ahead of opens in default KPI views.
- Acceptance criteria: Admin email views can page or summarize beyond 50 rows; the UI labels opens as directional; and default KPI cards emphasize reliable delivery and action metrics over open counts.

## Business Access And Reporting Rules

- Finance business dates: Admins, Client Admins, Client Finance, and Bums need business-effective dates for reconciliation, period filtering, invoice aging, and payout reporting. Non-finance client roles should not gain finance access as part of this work. Sensitivity: financial reporting accuracy. Verification: seeded late-entry scenarios prove effective dates drive filters while audit timestamps remain visible.
- Client Finance export scope: Client Finance needs company-scoped payment reports, invoice registers, finance dashboards, and finance exports. Client Finance should not receive target contact emails, attendee lists, join URLs, transcript-adjacent metadata, or other operational workflow detail unless `docs/business-access-rules.md` explicitly grants that scope. Sensitivity: financial plus operational contact data. Verification: export-content assertions and route allow/deny tests by client access role.
- Admin aggregate reporting RPCs: Admins need marketplace-wide aggregate counts for operational dashboards. Public visitors, Client roles, and Bums should not call privileged summary RPCs or receive marketplace-wide counts through exposed APIs. Sensitivity: admin-only operational aggregate reporting. Verification: grant review and negative RPC tests for non-admin signed-in roles.
- Access-request and terms-deferral reporting: Admins need raw historical request outcomes, reviewer timestamps, request evidence, and deferral-limit visibility for onboarding governance. Client Admins may need same-company pending request visibility for operational approval only where the business rule explicitly allows it. Client Finance, Bums, and public users should not read raw request evidence, review notes, or terms-deferral histories. Sensitivity: onboarding identity/access governance and contract-compliance behavior. Verification: role-by-role report tests plus explicit documentation updates in `docs/business-access-rules.md` before any new reporting path ships.
- Admin email analytics: Admins need recipient-level delivery and engagement operations data for troubleshooting and campaign operations. Non-admin roles should not read raw delivery history, recipient behavior rows, or raw engagement telemetry. Sensitivity: recipient operations data and messaging behavior. Verification: admin-only access tests for campaigns, deliveries, and engagement surfaces, plus review of any new aggregate views.
