# Trusted Bums Accessibility Backlog

_Last updated: 2026-05-28 by Codex daily accessibility specialist automation._

## Executive Read

The highest-confidence accessibility risks are still route-level semantics and operability gaps inside active portal pages, not shell navigation. The shared skip link remains implemented in the admin, client, and Bum layouts, so the active backlog stays focused on four validated issues: placeholder-only search fields, visible labels that do not name the actual control, the shared mobile filter panel without disclosure state, and dense tables that still enter the accessibility tree without explicit names.

Evidence quality is slightly better than the previous run but still below real operability validation. `pnpm run qa:env` passes after sourcing `.env.qa`, `pnpm run lint` completes with warnings only, and `pnpm run test` passed. However, `curl -I --max-time 15 "$QA_BASE_URL"` still timed out on DNS resolution on 2026-05-28, the repo still has no `@axe-core/playwright` or `axe-core`, and a local `vite preview` build currently renders `Trusted Bums configuration needed` instead of the real public UI. Authenticated screenshots, keyboard walkthroughs, screen-reader notes, and fresh browser evidence are still missing.

### P1 - Give route-level search fields stable accessible names
- Evidence: Placeholder-only search inputs remain on active routes including [`src/pages/admin/AdminClients.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminClients.tsx), [`src/pages/admin/AdminOpportunities.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminOpportunities.tsx), [`src/pages/admin/AdminPayouts.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminPayouts.tsx), [`src/pages/admin/AdminBums.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminBums.tsx), [`src/pages/bum/BumClients.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/bum/BumClients.tsx), [`src/pages/bum/BumContacts.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/bum/BumContacts.tsx), [`src/pages/bum/BumOpportunities.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/bum/BumOpportunities.tsx), [`src/pages/client/ClientPayments.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientPayments.tsx), [`src/pages/client/ClientTargets.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientTargets.tsx), and [`src/pages/client/ClientTrainings.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientTrainings.tsx). [`src/components/PortalGlobalSearch.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/PortalGlobalSearch.tsx) still shows the correct pattern by exposing a stable accessible name.
- Why it matters: Placeholder-only naming breaks once text is entered and is unreliable for screen readers, voice input, translation, and low-vision users.
- Recommendation: Standardize a compact search or filter input pattern that always provides either a visible `<Label htmlFor>` or explicit `aria-label` or `aria-labelledby`, and use it across admin, client, and Bum route filters.
- Acceptance criteria: Every visible route-level search field exposes a stable accessible name in the accessibility tree that does not depend on placeholder text.

### P1 - Bind visible labels to custom selects and unbound edit fields
- Evidence: Unbound labels remain in active edit forms and filter controls, including the client edit dialog in [`src/pages/admin/AdminClients.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminClients.tsx), legal and admin forms in [`src/pages/admin/AdminLegal.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminLegal.tsx), payment-related forms in [`src/pages/client/ClientPayments.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientPayments.tsx), and the contact editor in [`src/pages/bum/BumContactDetail.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/bum/BumContactDetail.tsx). Common patterns are plain `<Label>` elements above Radix `SelectTrigger` controls with no `id` or `aria-labelledby`, plus repeated field groups such as phone-number editing without a programmatic group name.
- Why it matters: Screen-reader and speech-input users can focus a control whose visible prompt is never announced as its accessible name, creating direct WCAG 2.2 Info and Relationships and Name, Role, Value risk.
- Recommendation: Add stable IDs and `htmlFor` pairs for native inputs and textareas, wire each custom `SelectTrigger` to its visible label with `id` plus `aria-labelledby` or `aria-label`, and name repeated-input sections with `fieldset` and `legend` or equivalent announced group labeling where needed.
- Acceptance criteria: Focusing each affected input, textarea, select trigger, and grouped section announces the same visible label or group name shown on screen.

### P2 - Expose the shared mobile filter panel as a disclosure
- Evidence: [`src/components/FilterPanel.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/FilterPanel.tsx) still toggles mobile filter visibility with local state only. The `Adjust` button has no `aria-expanded` or `aria-controls`, and the revealed content has no linked region semantics. This component is used on active routes including [`src/pages/admin/AdminOpportunities.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminOpportunities.tsx), [`src/pages/bum/BumClients.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/bum/BumClients.tsx), and [`src/pages/bum/BumOpportunities.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/bum/BumOpportunities.tsx).
- Why it matters: Assistive technology users do not get the expanded or collapsed state or the relationship between the trigger and the revealed filter content on small screens.
- Recommendation: Implement the mobile filter toggle as an APG disclosure with button semantics, `aria-expanded`, `aria-controls`, a stable controlled-panel ID, and a panel name derived from the visible Filters heading.
- Acceptance criteria: On mobile viewports, the filter toggle announces expanded or collapsed state and points to a discoverable controlled panel.

### P2 - Give dense tables explicit accessible names
- Evidence: Dense tables still render without a `<caption>` or `aria-labelledby` connection in active reporting and admin surfaces including [`src/pages/admin/AdminDashboard.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminDashboard.tsx), [`src/pages/admin/AdminPerformanceMetrics.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminPerformanceMetrics.tsx), [`src/pages/client/ClientTeam.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientTeam.tsx), [`src/components/reports/ReportsWorkspace.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/reports/ReportsWorkspace.tsx), and the request table in [`src/pages/client/ClientRequests.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientRequests.tsx). The visible section titles near these tables are not programmatically linked to the `<Table>` elements.
- Why it matters: Screen-reader users entering table navigation hear generic table announcements and must infer which queue, report, or assignment list they reached.
- Recommendation: Add visible or screen-reader-only captions to major tables, or connect each table to its visible section heading with `aria-labelledby`. Where the first data cell identifies the row subject, expose it as a row header.
- Acceptance criteria: Each major table announces a unique name before row and column navigation begins, and record-subject cells are exposed as row headers where appropriate.

## Current Standards And Time-Sensitive Notes

- W3C’s WCAG overview was updated on 2026-05-26 and still positions WCAG 2.2 as the current WCAG release. It also notes WCAG 2.2 is available as ISO/IEC 40500:2025, with the December 2024 WCAG 2.2 update expected to land in ISO form later in 2026. Keep this backlog anchored to WCAG 2.2, not 2.1. Sources: [W3C WCAG Overview](https://www.w3.org/WAI/standards-guidelines/wcag/) and [W3C WCAG 2.2 Recommendation](https://www.w3.org/TR/wcag/).
- W3C’s current disclosure guidance still expects a disclosure button to expose `aria-expanded`, with `aria-controls` optional but recommended for the controlled content. That maps directly to the current mobile filter toggle gap. Source: [WAI-ARIA APG Disclosure Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/disclosure/).
- MDN’s `placeholder` guidance, last modified on 2026-04-20, still states that placeholder text must not replace a `<label>` and explicitly notes that labels are what assistive technology announces. That directly applies to the route-level search fields still using placeholders as the only prompt. Source: [MDN `placeholder`](https://developer.mozilla.org/docs/Web/HTML/Reference/Attributes/placeholder).
- Playwright’s current accessibility-testing guidance still recommends combining end-to-end coverage with `@axe-core/playwright` and manual assessment because automated checks only catch part of the problem space. That remains directly relevant because this repo has route audits but no automated accessibility assertions. Source: [Playwright accessibility testing guide](https://playwright.dev/docs/accessibility-testing).
