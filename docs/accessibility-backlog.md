# Trusted Bums Accessibility Backlog

_Last updated: 2026-06-06 by Codex daily accessibility specialist automation._

## Executive Read

Current source, current W3C guidance, and narrow local QA still point to the same three implementation-ready accessibility issues: the mobile portal sidebar still behaves like an unnamed modal and hides its close control, the public contact form still reports validation only through toasts, and the public sign-up intent dialog still shows visible validation text without programmatic error wiring. Validation evidence improved slightly on the local build path: `pnpm run build` passed and sourcing `.env.qa` restored the expected QA contract, but stronger operability proof is still blocked because the checked-in public Playwright smoke spec skips without `QA_BASE_URL`, the repo still has no axe dependency, and authenticated browser, screen-reader, and screenshot evidence are still missing.

## Active Recommendations

### P1 - Give the mobile sidebar sheet an accessible name and visible dismiss control
- Evidence: [`src/components/ui/sidebar.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/ui/sidebar.tsx:155) still renders the mobile portal navigation inside `SheetContent`, and [`src/components/ui/sidebar.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/ui/sidebar.tsx:159) still hides the built-in close button with `[&>button]:hidden`. [`src/components/ui/sheet.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/ui/sheet.tsx:60) still provides a close button by default, but the sidebar wrapper adds no `SheetTitle`, `aria-label`, or other dialog name.
- Why it matters: On mobile, this navigation is a modal dialog. Current APG guidance still expects a modal dialog to have an accessible name and strongly recommends a visible close button in the tab sequence.
- Recommendation: Keep the in-sheet close button visible and add a stable dialog name such as `Portal navigation`, either through `SheetTitle` or `aria-label`, for the Admin, Client, and Bum mobile sidebars.
- Acceptance criteria: On mobile widths, opening the sidebar exposes a visible dismiss button, the dialog is announced with a clear name, `Escape` and the dismiss button both close it, and focus returns to the `Toggle Sidebar` trigger.

### P1 - Add field-level errors and verification messaging to the public contact form
- Evidence: [`src/pages/Index.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/Index.tsx:151) still rejects invalid name, email, and message values only through a destructive toast, and [`src/pages/Index.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/Index.tsx:160) still rejects missing Turnstile completion only through a second toast. The rendered fields in [`src/pages/Index.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/Index.tsx:550) through [`src/pages/Index.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/Index.tsx:647) still have no inline error text, `aria-invalid`, or error-message `aria-describedby` wiring.
- Why it matters: Toast-only validation forces users to infer which field failed and can disappear before screen-reader, keyboard, or magnification users recover. This is still an error-identification and recovery risk on the main public conversion form.
- Recommendation: Add persistent inline error text for each invalid field and the Turnstile block, mark invalid fields with `aria-invalid`, and move focus to the first invalid field or a concise error summary after submit.
- Acceptance criteria: Invalid submission keeps the user on the form, exposes visible error text next to every failing field, wires each message with `aria-describedby`, marks failing fields with `aria-invalid`, and does not rely on toast timing as the only error signal.

### P2 - Add programmatic error state to the public sign-up intent dialog
- Evidence: [`src/components/SignupIntentDialog.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/SignupIntentDialog.tsx:137) still renders visible inline validation messages for missing account type, invalid email, and missing company name, but [`src/components/SignupIntentDialog.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/SignupIntentDialog.tsx:105) through [`src/components/SignupIntentDialog.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/SignupIntentDialog.tsx:167) still do not add `aria-invalid` or `aria-describedby` to the radio-group items or inputs. [`tests/e2e/staging-smoke.spec.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/tests/e2e/staging-smoke.spec.ts:36) still treats visible error text as the complete validation proof and still expects the stale copy `Select Client or Bum.`
- Why it matters: Visible error text is better than toast-only validation, but screen-reader users still do not get a reliable programmatic link between the failing control and its error state during a public account-creation flow.
- Recommendation: Keep the inline messages, then add stable IDs plus `aria-describedby` and `aria-invalid` to the role, email, and company controls when validation fails. If the role group remains custom, add one group-level error target that assistive technology can announce reliably.
- Acceptance criteria: After an invalid submit, each failing sign-up control exposes `aria-invalid="true"`, its visible error message is linked with `aria-describedby`, and keyboard or screen-reader users can determine which field failed without relying on surrounding visual context.

## Current Standards And Time-Sensitive Notes

- WCAG 2.2 remains the current W3C Recommendation. The W3C WCAG overview page was updated in May 2026, and the WCAG 2.2 errata page was last modified on 2026-05-07. Those updates do not change the current backlog recommendations for dialog naming, visible dismiss controls, or form error identification. Sources: [WCAG 2 Overview](https://www.w3.org/WAI/standards-guidelines/wcag/), [WCAG 2.2 Errata](https://www.w3.org/WAI/WCAG22/errata/).
- Current WAI-ARIA APG modal dialog guidance still says the dialog needs either `aria-labelledby` or `aria-label` and strongly recommends a visible close button in the tab order. That remains directly applicable to the mobile sidebar sheet. Source: [Dialog (Modal) Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/).
- Current WCAG 2.2 error-identification guidance still expects input errors to be identified in text. W3C and GOV.UK current form-validation guidance both continue to recommend field-level messages and an error summary that links to failing fields, which remains the right pattern for the public contact flow. Sources: [Understanding SC 3.3.1 Error Identification](https://www.w3.org/WAI/WCAG22/Understanding/error-identification), [Forms: errors - W3C Design System](https://design-system.w3.org/styles/form-errors.html), [Error summary - GOV.UK Design System](https://design-system.service.gov.uk/components/error-summary/).
- W3C's current `aria-invalid` working example still shows failed fields paired with `aria-describedby` error text, and notes that common screen readers announce the invalid state. That remains the relevant implementation model for the contact form and sign-up dialog. Source: [Forms: Using aria-invalid](https://www.w3.org/WAI/WCAG22/working-examples/aria-invalid-data-format/).
