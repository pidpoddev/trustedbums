# Trusted Bums Accessibility Backlog

_Last updated: 2026-05-31 by Codex daily accessibility specialist automation._

## Executive Read

Current evidence is stronger than a source-only pass but still incomplete. This run revalidated live browser reachability, passed the public privacy and sign-up accessibility smoke paths, reused recent visual-audit screenshots, and isolated three implementation-ready accessibility issues: the persistent privacy launcher can obscure portal navigation, the public contact form still reports validation only through toasts, and the mobile sidebar sheet hides its visible close control. Authenticated assistive-technology proof is still limited because the repo has no axe wiring or screen-reader notes, and some Playwright portal flows remain flaky or blocked by non-accessibility test issues.

## Active Recommendations

### P1 - Stop the privacy launcher from obscuring portal navigation
- Evidence: [`src/components/ConsentManager.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/ConsentManager.tsx:141) keeps a fixed bottom-left `Privacy` button on screen after the banner closes. The recent desktop screenshot artifact [`test-results/visual-ui-audit-authentica-7475d-ctive-states-render-cleanly-chromium/chromium-client_admin-client-target-add-form-open.png`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/test-results/visual-ui-audit-authentica-7475d-ctive-states-render-cleanly-chromium/chromium-client_admin-client-target-add-form-open.png) shows that launcher sitting on top of the left sidebar footer area near `Agreements`.
- Why it matters: A persistent overlay can hide or partially cover the focused control, which is directly relevant to WCAG 2.2 focus visibility and focus-not-obscured expectations on keyboard paths.
- Recommendation: Move the privacy entry point into a non-overlapping location for portal layouts, or reserve layout space so the launcher never covers sidebar/footer navigation at desktop or mobile breakpoints.
- Acceptance criteria: With the consent banner dismissed, no persistent privacy control overlaps any interactive element at 320px, 768px, 1024px, or 1440px widths; the last sidebar/footer links remain fully visible, clickable, and visibly focused.

### P1 - Add field-level errors and verification messaging to the public contact form
- Evidence: [`src/pages/Index.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/Index.tsx:144) blocks submission for invalid name/email/message and missing Turnstile completion, but only announces the problem through destructive toasts. The form fields and verification container do not receive persistent error text, `aria-invalid`, or programmatic error associations.
- Why it matters: Screen-reader and keyboard users should not have to infer which control failed from a transient toast. This affects error identification, recovery speed, and trust on the main public conversion form.
- Recommendation: Add inline validation state for each failing field plus a visible verification error for Turnstile, wire those messages with `aria-describedby`, and move focus to the first invalid control or an error summary after submit.
- Acceptance criteria: Submitting invalid data keeps the user on the form, exposes persistent error text next to each failing field, marks invalid controls with `aria-invalid`, and announces the verification requirement without relying only on toast timing.

### P2 - Restore an explicit close control for the mobile sidebar sheet
- Evidence: [`src/components/ui/sidebar.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/ui/sidebar.tsx:153) renders the mobile navigation in a Radix `Sheet`, but its `SheetContent` class hides the built-in close button with `[&>button]:hidden`.
- Why it matters: Modal navigation should provide an obvious dismiss action for touch, switch, and screen-reader users. The WAI-ARIA dialog guidance still recommends a visible close button inside the dialog.
- Recommendation: Keep a visible close button inside the mobile sheet or add an equally obvious in-sheet dismiss control with a stable accessible name.
- Acceptance criteria: On mobile widths, opening the sidebar exposes a visible dismiss button inside the sheet, that button is keyboard reachable, announced with a clear name, and closes the sheet with focus returned to the trigger.

## Current Standards And Time-Sensitive Notes

- WCAG 2.2 has been the current W3C Recommendation since October 5, 2023. For this repo, the most relevant current additions are `2.4.11 Focus Not Obscured (Minimum)` at Level AA and the stronger focus-appearance guidance in the WCAG 2.2 understanding docs. Source: [W3C WCAG 2.2 overview](https://www.w3.org/WAI/standards-guidelines/wcag/new-in-22/), [Understanding 2.4.11](https://www.w3.org/WAI/WCAG22/Understanding/focus-not-obscured-minimum.html), [Understanding 2.4.13](https://www.w3.org/WAI/WCAG22/Understanding/focus-appearance.html).
- WAI-ARIA APG still expects modal dialogs to trap focus and strongly recommends including a visible close button in the tab order. That is directly relevant to the mobile sidebar sheet pattern here. Source: [WAI-ARIA APG modal dialog pattern](https://www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/).
- Same-page skip links remain a valid accessibility pattern when they become visible on focus and jump to main content. Source: [W3C skip-link easy check](https://www.w3.org/WAI/test-evaluate/easy-checks/skip-link/).

## Access Requests And Evidence Gaps

- The repo still lacks `@axe-core/playwright` or `axe-core`, so this run could not add automated axe findings to the backlog even though live browser reachability was restored.
- Authenticated route evidence is only partial. `zsh -lc 'source .env.qa && pnpm run qa:env'` still did not export the QA contract in this shell pattern, while `set -a; source .env.qa; set +a; pnpm run qa:env` did. Future consultants need that export pattern documented or wrapped in script form.
- No screen-reader notes, narrated keyboard walkthroughs, or AT/browser matrix evidence were available for admin, client, finance, or Bum flows. Keep requesting those until at least the highest-traffic routes have real AT validation.
- Some Playwright authenticated checks remain noisy for non-accessibility reasons, including route-guard behavior and auth/test flakiness, so failed specs cannot be treated as pure accessibility failures without follow-up triage.
