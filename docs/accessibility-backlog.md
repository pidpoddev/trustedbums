# Trusted Bums Accessibility Backlog

_Last updated: 2026-06-04 by Codex daily accessibility specialist automation._

## Executive Read

This run revalidated the current source, current W3C guidance, and the current QA/browser evidence. Four implementation-ready issues remain grounded in current code and current test output: the mobile portal sidebar still opens as an unlabeled dialog and hides its built-in close button, the public contact form still reports validation only through toasts, the public sign-up intent dialog still renders visible validation text without programmatic error wiring, and `/client/payments` now ships duplicate visible headings with the same text. Deployed public-route accessibility validation is still blocked because sourced Playwright smoke against `https://trustedbums.com/` failed at first navigation with `net::ERR_ABORTED`.

## Active Recommendations

### P1 - Give the mobile sidebar sheet an accessible name and visible dismiss control
- Evidence: [`src/components/ui/sidebar.tsx`](src/components/ui/sidebar.tsx:155) renders portal mobile navigation inside Radix `SheetContent`, but hides the built-in close button with `[&>button]:hidden` at line 159. [`src/components/ui/sheet.tsx`](src/components/ui/sheet.tsx:60) shows that the underlying sheet does provide a focusable `Close` control, and the mobile sidebar wrapper does not add a `SheetTitle`, `aria-label`, or equivalent accessible name.
- Why it matters: Mobile navigation is behaving like a modal dialog. Current APG guidance expects modal dialogs to have an accessible name and strongly recommends a visible close control in the tab order.
- Recommendation: Keep the in-sheet close button visible and add a stable dialog name such as `Portal navigation`, either through `SheetTitle` or `aria-label`, for the client, admin, and Bum mobile sidebars.
- Acceptance criteria: On mobile widths, opening the sidebar exposes a visible dismiss button, the dialog is announced with a clear name, `Escape` and the dismiss button both close it, and focus returns to the `Toggle Sidebar` trigger.

### P1 - Add field-level errors and verification messaging to the public contact form
- Evidence: [`src/pages/Index.tsx`](src/pages/Index.tsx:151) rejects invalid name, email, and message values, and line 160 rejects missing Turnstile completion, but both states are surfaced only through destructive toasts. The form controls do not receive `aria-invalid`, inline error copy, or programmatic `aria-describedby` associations.
- Why it matters: Toast-only validation makes users infer which control failed and can disappear before screen-reader, keyboard, or magnification users recover. That is a direct error-identification and recovery risk on the primary public conversion form.
- Recommendation: Add persistent inline error text for each invalid field and the Turnstile block, mark invalid controls with `aria-invalid`, and move focus to the first invalid control or a concise error summary after submit.
- Acceptance criteria: Invalid submission leaves the user on the form, exposes visible error text next to every failing control, wires each message with `aria-describedby`, marks the failing fields with `aria-invalid`, and does not rely on toast timing as the only error signal.

### P2 - Add programmatic error state to the public sign-up intent dialog
- Evidence: [`src/components/SignupIntentDialog.tsx`](src/components/SignupIntentDialog.tsx:137) renders visible inline messages for missing role, invalid email, and missing company name, but the `RadioGroupItem` and `Input` controls still do not receive `aria-invalid` or `aria-describedby`. [`tests/e2e/staging-smoke.spec.ts`](tests/e2e/staging-smoke.spec.ts:36) still checks only for visible text messages before Clerk handoff, not programmatic error state.
- Why it matters: This dialog is part of the public account-creation flow. Visible error text alone is better than toast-only messaging, but screen-reader users still do not get a programmatic link between the failing field and its error state.
- Recommendation: Keep the inline messages, then add stable IDs plus `aria-describedby` and `aria-invalid` to the role, email, and company controls when validation fails. If the role group remains custom, also expose one concise group-level error target that assistive tech can announce reliably.
- Acceptance criteria: After an invalid submit, each failing sign-up control exposes `aria-invalid="true"`, its visible error message is linked with `aria-describedby`, and keyboard or screen-reader users can determine which field failed without relying on surrounding visual context.

### P2 - Disambiguate duplicate visible headings on Client Payments
- Evidence: [`src/pages/client/ClientPayments.tsx`](src/pages/client/ClientPayments.tsx:383) renders the page `<h1>` as `Customer Payment Reports`, and the same route renders a second visible heading with the exact same text in [`CardTitle`](src/pages/client/ClientPayments.tsx:684). [`src/components/ui/card.tsx`](src/components/ui/card.tsx:17) confirms `CardTitle` is an `<h3>`. Current GitHub-authenticated smoke also relies on an exact accessible heading name for this route in [`tests/e2e/authenticated-role-smoke.spec.ts`](tests/e2e/authenticated-role-smoke.spec.ts:35), and the current access-needs log records a client-finance failure on the duplicated accessible heading.
- Why it matters: Duplicate visible headings with identical text make heading navigation noisier for screen-reader and keyboard users and increase the chance of ambiguous route assertions, documentation, and support guidance for a finance-only workflow.
- Recommendation: Keep the page-level heading as the route name, then rename the lower card heading to the section purpose, such as `Reported customer payments`, or remove heading semantics if that card title does not represent a distinct section.
- Acceptance criteria: `/client/payments` exposes one route-level heading named `Customer Payment Reports`, subordinate sections use distinct labels, and heading-list navigation no longer presents two identically named stops on the same page.

## Current Standards And Time-Sensitive Notes

- WCAG 2.2 remains the current W3C Recommendation. For this repo, `2.4.11 Focus Not Obscured (Minimum)` and `2.4.13 Focus Appearance` remain the most relevant newer guidance for persistent overlays and keyboard focus visibility. Source: [What’s New in WCAG 2.2](https://www.w3.org/WAI/standards-guidelines/wcag/new-in-22/), [Understanding 2.4.13](https://www.w3.org/WAI/WCAG22/Understanding/focus-appearance.html).
- Current WAI-ARIA APG guidance for modal dialogs still expects an accessible name, focus containment, and strongly recommends a visible close button in the tab sequence. That applies directly to the mobile sidebar sheet implementation. Source: [Dialog (Modal) Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/).
- Current WCAG 2.2 guidance still expects input errors to be identified in text and headings and labels to describe purpose clearly; that supports the form-error and duplicate-heading recommendations above. Sources: [Understanding SC 3.3.1 Error Identification](https://www.w3.org/WAI/WCAG22/Understanding/error-identification.html), [Understanding SC 2.4.6 Headings and Labels](https://www.w3.org/WAI/WCAG22/Understanding/headings-and-labels.html), [W3C aria-invalid working example](https://www.w3.org/WAI/WCAG22/working-examples/aria-invalid-data-format/).
- W3C’s WCAG 2.2 errata page was last modified on 2026-05-07 and only adds editorial corrections since publication, so no newer normative change alters the recommendations above. Source: [WCAG 2.2 Errata](https://www.w3.org/WAI/WCAG22/errata/).
