## Active Recommendations

### P1 - [TB-0060] Reserve one intentional mobile support zone for privacy and chat across authenticated shells
- Evidence: Exact-head hosted `Visual UI Audit` run `27857691601` on `e231cc0` still shows the fixed `Privacy choices` chip and chat launcher intruding into live mobile workspace areas. In `mobile-chrome-admin-admin-dashboard.png`, the stack overlaps the Priority queues card. In `mobile-chrome-admin-admin-handoffs.png`, the same support stack lands on the filter/search zone. Current source still explains the overlap: [ConsentManager.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/ConsentManager.tsx) keeps the launcher fixed at the lower-right edge, [ConversationDock.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/ConversationDock.tsx) still mounts a separate fixed chat launcher, and the shared portal layouts only partly compensate with larger bottom padding.
- Why it matters: Persistent support disclosures that cover live controls raise a real accessibility risk on mobile authenticated routes because they can obscure actionable content and keyboard focus. Current W3C guidance for Focus Not Obscured (Minimum) explicitly calls out persistent disclosures such as chat interfaces and recommends displacement, reflow, or another escape path that keeps the focused item available.
- Recommendation: Treat privacy and chat as one coordinated mobile support zone across authenticated shells. Either merge the privacy action into the same support surface as chat, or reserve shared bottom-right spacing so both controls clear cards, queue copy, search/filter controls, tabs, and CTAs on Admin, Client, and Bum mobile pages. Keep an explicit keyboard dismissal or focus-preserving reveal path if the support surface can remain open.
- Acceptance criteria: Fresh exact-head mobile screenshots for `/admin`, `/admin/handoffs`, `/client/opportunities/new`, and one Bum workflow route show no bottom-right overlap with cards, filters, tabs, queues, or CTAs. Keyboard users can move through the affected area without the support surface obscuring the focused item, and the shared support zone respects `env(safe-area-inset-bottom)`.

## Current Standards And Time-Sensitive Notes

- WCAG 2.2 remains the current W3C baseline, and the W3C WCAG overview still encourages teams to use the latest version. Its overview page is currently marked updated on 2026-05-26. Source: https://www.w3.org/WAI/standards-guidelines/wcag/
- Current W3C guidance for Success Criterion 2.4.11 Focus Not Obscured (Minimum) explicitly calls out persistent user-openable disclosures such as chat interfaces. Relevant passing patterns include displacement, content reflow, or another mechanism that lets users reveal the focused item without having to backtrack through the overlay. Source: https://www.w3.org/WAI/WCAG22/Understanding/focus-not-obscured-minimum.html
- WCAG 2.2 target-size minimum guidance still centers on a `24 by 24 CSS pixel` minimum unless an exception applies. Current [ConsentManager.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/ConsentManager.tsx) still keeps the collapsed launcher above that floor, so this standard does not support reopening `TB-0032` on current head. Source: https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum.html
- Current WAI-ARIA APG dialog guidance still expects focus to move into a dialog when it opens, return to the invoking control when it closes, and include a visible close button in the tab sequence. That remains the relevant standard for the shared mobile sidebar sheet and any modal consent or settings flows. Source: https://www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/
