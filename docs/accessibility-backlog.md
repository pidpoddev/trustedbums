## Active Recommendations

### P1 - [TB-0029] Give the mobile portal sidebar an accessible name and a visible dismiss control
- Evidence: `src/components/ui/sidebar.tsx` still renders the mobile portal navigation inside `SheetContent` and hides the default close button with `[&>button]:hidden` at [line 159](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/ui/sidebar.tsx:159). The sidebar wrapper still adds no `SheetTitle`, `aria-label`, or equivalent dialog name before rendering children at [lines 155-168](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/ui/sidebar.tsx:155).
- Why it matters: On mobile, the portal navigation is a modal dialog. Current WAI-ARIA APG guidance still expects the dialog to have an accessible name and strongly recommends a visible close button in the tab sequence.
- Recommendation: Keep the in-sheet close button visible and add a stable dialog name such as `Portal navigation` through `SheetTitle` or `aria-label` for the Admin, Client, and Bum mobile sidebars.
- Acceptance criteria: On mobile widths, opening the sidebar exposes a visible dismiss control, the dialog is announced with a clear name, `Escape` and the dismiss control both close it, and focus returns to the `Toggle Sidebar` trigger.

### P1 - [TB-0030] Add programmatic error state and group labeling to the public sign-up intent dialog
- Evidence: `src/components/SignupIntentDialog.tsx` still renders visible validation text for missing account type, invalid email, and missing company name at [lines 105-151](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/SignupIntentDialog.tsx:105) and [lines 154-169](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/SignupIntentDialog.tsx:154), but the radio group and inputs still do not add `aria-invalid`, `aria-describedby`, `fieldset`/`legend`, or equivalent `aria-labelledby` wiring. Fresh local Playwright on `http://127.0.0.1:8080` passed both public signup-validation checks in [`tests/e2e/staging-smoke.spec.ts`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/tests/e2e/staging-smoke.spec.ts:42), confirming the visible error copy exists while also confirming that the current proof stops at visible text.
- Why it matters: This is the public account-entry path for both Client and Bum signups. Users can see the validation copy, but assistive technology still lacks a reliable programmatic connection between the failing controls, the account-type group, and the visible error messages.
- Recommendation: Keep the visible validation copy, then add `aria-invalid` plus stable `aria-describedby` targets to the email and company fields, and expose the account-type radio group through `fieldset`/`legend` or `aria-labelledby` with one group-level error target.
- Acceptance criteria: After an invalid submit, the account-type group is announced with a stable accessible name and linked error, each failing input exposes `aria-invalid="true"`, each visible message is linked with `aria-describedby`, and keyboard or screen-reader users can identify every failing control without relying on surrounding visual context.

### P2 - [TB-0031] Give the admin scrum tracker search and select controls programmatic labels
- Evidence: The new admin route at [`src/pages/admin/AdminScrumTracker.tsx`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/admin/AdminScrumTracker.tsx:382) renders the tracked-items search box with only placeholder text, and the tracked-items filter selects at [lines 387-414](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/admin/AdminScrumTracker.tsx:387) plus the add-item selects at [lines 332-351](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/admin/AdminScrumTracker.tsx:332) render adjacent `Label` text without `htmlFor`, `aria-labelledby`, or other control association. Current local Vitest only proves the route and sidebar entry exist in [`src/test/adminScrumTracker.test.ts`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/test/adminScrumTracker.test.ts:1); this new surface still has no authenticated browser or visual accessibility proof.
- Why it matters: Admin operators need reliable search and filtering when triaging live bugs and release blockers. Placeholder-only search and unnamed custom selects make the route harder to use with screen readers, speech input, and form-navigation shortcuts.
- Recommendation: Give the search input an explicit accessible name and associate every filter or create-form select with visible text through `aria-labelledby`, an explicit trigger `id`, or an equivalent control-specific label pattern already used elsewhere in the app.
- Acceptance criteria: Screen readers announce stable names such as `Search scrum items`, `Filter by status`, `Filter by priority`, `Filter by type`, `Priority`, `Source`, and `Type`; clicking the visible label focuses or opens the associated control where applicable; and the route can be navigated by label rather than placeholder text alone.

### P2 - [TB-0032] Enlarge the collapsed `Privacy choices` launcher to meet WCAG 2.2 target-size expectations
- Evidence: After the banner closes, `src/components/ConsentManager.tsx` still renders the persistent launcher as a fixed `Button` with `h-5`, `px-2`, and `text-[10px]` at [lines 141-156](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/ConsentManager.tsx:141). Local anonymous Playwright checks now confirm the privacy manager itself is operable on `127.0.0.1:8080`, but this run still had no fresh mobile screenshot evidence that would justify a target-size exception for the collapsed launcher.
- Why it matters: The privacy launcher is a site-wide legal and consent control. Small fixed targets are harder to activate for touch and motor-impaired users, especially on mobile.
- Recommendation: Increase the launcher hit area to at least 24 by 24 CSS pixels, or replace it with an equivalently reachable same-page control that already meets the target-size expectation.
- Acceptance criteria: The collapsed `Privacy choices` control exposes at least a 24 by 24 CSS pixel hit area on public and portal pages, remains visually discoverable, and does not rely on a separate page to provide the only adequately sized control.

## Current Standards And Time-Sensitive Notes

- WCAG 2.2 remains the current W3C baseline for this backlog. The current W3C WCAG overview still points teams to the latest WCAG 2.2 materials, and the current WCAG 2.2 errata remain editorial rather than priority-changing for these findings. Sources: https://www.w3.org/WAI/standards-guidelines/wcag/ and https://www.w3.org/WAI/WCAG22/errata/
- Current WAI-ARIA APG modal dialog guidance still says dialogs need either `aria-labelledby` or `aria-label`, and it strongly recommends a visible close button in the tab sequence. That remains directly applicable to the mobile sidebar sheet. Source: https://www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/
- Current W3C form-label guidance still says form controls should be explicitly or implicitly associated with labels, and current WAI accessible-name guidance still treats placeholder and title content as fallback naming sources rather than the preferred control-label pattern. That directly applies to the new admin scrum tracker controls. Sources: https://www.w3.org/WAI/tutorials/forms/labels/ and https://www.w3.org/WAI/ARIA/apg/practices/names-and-descriptions/
- Current WCAG 2.2 target-size guidance still says pointer targets should be at least 24 by 24 CSS pixels unless a defined exception applies. That still applies to the collapsed site-wide `Privacy choices` launcher. Source: https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum.html
