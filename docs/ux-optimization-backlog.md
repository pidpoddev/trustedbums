# Trusted Bums UX Optimization Backlog

_Last updated: 2026-06-03 by Codex daily UX consultant automation._

## Executive Read

The highest-confidence UX work is still concentrated in trust-critical first touch and recovery paths. Public sign-up can still erase a manually entered company name, and the homepage contact form still forces visitors to recover from errors through destructive toasts instead of field-level guidance.

Recent terms-gating changes introduced a new dashboard clarity gap. Client users can now defer updated terms, but the dashboard still points them to `Company Profile` instead of the dedicated agreement flow, and access-role redirects still land silently on the dashboard with no explanation of what was blocked or what to do next.

Authenticated live validation remains blocked. Today’s authenticated recommendations are therefore source-backed, not freshly browser-verified.

## Active Recommendations

### P1 - Preserve typed company names in the public signup flow
- Evidence: [src/components/SignupIntentDialog.tsx:44](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/SignupIntentDialog.tsx:44) still resets `companyName` whenever the role is `CLIENT` and `getKnownClientForEmail(email)` returns no match, so editing the email can erase a manual company entry before sign-up is complete. Current Baymard guidance still calls clearing entered data one of the most damaging form UX failures and recommends preserving typed input during recovery ([Baymard checkout flow UX guide](https://baymard.com/learn/checkout-flow-ux-optimization)).
- Why it matters: This is a top-of-funnel trust step. Silent data loss makes account creation feel brittle before a prospect has any reason to trust the platform.
- Recommendation: Preserve a manually entered `companyName` unless the user explicitly accepts a matched workspace. Keep the match helper text, but separate matched-workspace behavior from the first-time workspace path so email edits do not wipe user input.
- Acceptance criteria: Editing the email no longer clears a manually entered company name when no match exists; matched workspaces still prefill clearly; the dialog supports both existing-workspace and first-time-workspace paths without silent data loss.

### P1 - Add field-level validation and reassurance to the homepage contact form
- Evidence: [src/pages/Index.tsx:144](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/Index.tsx:144) still validates only `name`, `email`, and `message` inside `submitContactForm` and shows destructive toasts on failure, while the rendered form in [src/pages/Index.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/Index.tsx) still has no inline error copy, linked summary, or `aria-invalid` wiring. Current W3C and GOV.UK guidance still recommends text error identification, error summaries linked to fields, and accessible invalid-state wiring ([W3C SC 3.3.1 Understanding](https://www.w3.org/WAI/WCAG22/Understanding/error-identification.html), [W3C Design System form errors](https://design-system.w3.org/styles/form-errors.html), [GOV.UK Error summary](https://design-system.service.gov.uk/components/error-summary/)).
- Why it matters: This is the main public handoff for buyers and prospective Bums. Toast-only validation increases recovery friction and weakens trust exactly when a visitor is deciding whether to share contact details.
- Recommendation: Add inline validation for each invalid field, an error summary on submit, `aria-invalid` and described-by wiring, and a short trust cue near the submit button that explains reply timing and how the submission is handled.
- Acceptance criteria: Invalid submit shows field-level messages and a linked summary; invalid fields expose accessible error state; recovery does not require guessing which field failed; the submit area includes a short expectation-setting note such as response window or handling reassurance.

### P1 - Make client-dashboard recovery paths match access and terms states
- Evidence: [src/components/ClientAccessRoute.tsx:30](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/ClientAccessRoute.tsx:30) still redirects unauthorized client users to `/client/dashboard` with `state={{ deniedFrom: location.pathname }}`, but [src/pages/client/ClientDashboard.tsx:52](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientDashboard.tsx:52) still does not read `deniedFrom` or explain the redirect. The same dashboard now sends users with unaccepted current terms to `/client/profile` at [src/pages/client/ClientDashboard.tsx:113](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientDashboard.tsx:113) and [src/pages/client/ClientDashboard.tsx:128](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientDashboard.tsx:128), even though the dedicated agreement surfaces are [src/pages/client/ClientAgreements.tsx:29](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientAgreements.tsx:29) and [src/pages/client/ClientTerms.tsx:33](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientTerms.tsx:33). Current W3C notification guidance still recommends concise messaging that explains what happened and what the user can do next ([W3C User Notification](https://www.w3.org/WAI/tutorials/forms/notifications/)).
- Why it matters: Silent redirects and misleading recovery links make role boundaries and legal-state requirements feel arbitrary, which increases support load and lowers trust in the client portal IA.
- Recommendation: Teach the dashboard to surface a dismissible recovery notice for `deniedFrom`, name the current client role, point to the nearest allowed workflow, and send outstanding-terms actions to `/client/agreements` or `/client/terms` instead of `Company Profile`.
- Acceptance criteria: A redirected user sees a visible notice naming the blocked destination, current role, and next allowed action; deferred or outdated-terms prompts link to the real agreement flow; ordinary dashboard visits do not show the notice.

### P1 - Add urgency and next-step cues to the admin handoff queues
- Evidence: [src/pages/admin/AdminHandoffs.tsx:255](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminHandoffs.tsx:255) still sorts target responses, intro requests, and public contact conversions by age, but the queue tables at [src/pages/admin/AdminHandoffs.tsx:355](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminHandoffs.tsx:355) only expose status, owner, created timestamp, and update actions. The underlying models already include `admin_next_action`, `admin_priority`, and `notification_error` in [src/lib/contactApi.ts:33](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/contactApi.ts:33) and [src/lib/portalApi.ts:737](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/portalApi.ts:737), while notification failure only appears in [src/pages/admin/AdminTroubleshooting.tsx:102](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminTroubleshooting.tsx:102). Current USWDS table guidance still recommends brief, scannable cells and clear scan behavior for dense operational tables ([USWDS table guidance](https://designsystem.digital.gov/components/table/)).
- Why it matters: This page is positioned as the place to rescue stalled handoffs. Without urgency and next-step cues, admins must infer what is actually stuck, which increases response delays for leads, Bum responses, and intro requests.
- Recommendation: Add visible aging states and next-step context to each queue. Start with an SLA bucket or stale badge, owner follow-up text, and a compact summary that distinguishes new, stale, and unowned work. Surface notification failure state for public contact submissions in the queue itself.
- Acceptance criteria: Open items older than the chosen SLA threshold are visually distinct; admins can see the next required action without opening another page; queue summaries distinguish fresh vs stale work; notification failures are visible for contact submissions that need manual follow-up.

### P2 - Reduce commission-plan request friction to a minimum viable schedule
- Evidence: [src/pages/client/ClientOpportunityNew.tsx:1210](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientOpportunityNew.tsx:1210) through [src/pages/client/ClientOpportunityNew.tsx:1266](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientOpportunityNew.tsx:1266) still mark Year 2 through Year 6+ commission inputs `required`, even though the request payload already normalizes blank later-year values to `0` upstream. Current W3C form guidance still recommends asking only for what is truly required and clearly distinguishing required inputs from optional detail ([W3C Forms Tutorial](https://www.w3.org/WAI/tutorials/forms/), [W3C Validating Input](https://www.w3.org/WAI/tutorials/forms/validation/)).
- Why it matters: Clients often know the near-term commission structure before they know the full long-tail payout schedule. Requiring six percentages up front increases hesitation and encourages invented data during opportunity setup.
- Recommendation: Make Year 1 the only default required rate, move later-year percentages behind an optional multi-year schedule choice, and explain how omitted years are handled during admin review.
- Acceptance criteria: A client can submit a commission-plan request with a minimum viable required set; later-year fields are clearly optional unless the user opts into a multi-year schedule; helper text explains fallback handling for omitted years.
