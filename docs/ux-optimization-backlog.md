### P1 - [TB-0096] Do not send Client Members to an inaccessible commission-plan route
- Evidence: Current dirty worktree source still grants `Client Member` access to `/client/opportunities` in [App.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/App.tsx:159), but the only visible `Commission Plans` destination remains restricted to `Client Admin` and `Client Finance` in [ClientLayout.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/layouts/ClientLayout.tsx:57) and [App.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/App.tsx:167). The opportunity form now tells users to request a new structure under `Finance > Commission Plans` in [ClientOpportunityNew.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/client/ClientOpportunityNew.tsx:1439), after the inline commission-plan request flow was removed from the workspace.
- Why it matters: If this worktree ships as-is, a legitimate opportunity-managing role hits a dead-end at the exact point where commission-plan guidance is supposed to unblock the workflow. That makes the permission model feel broken and forces Members to guess whether the action is unsupported or merely hidden.
- Recommendation: Either restore a `Client Member`-safe request path inside the opportunity workspace or rewrite the helper copy so it explicitly names the allowed owner path, such as asking a `Client Admin` or `Client Finance` user to request the plan. Do not link Members to a route they cannot open.
- Acceptance criteria: `Client Member` users who can open `/client/opportunities` are never told to navigate to an inaccessible route; the alternative path is explicit and role-accurate; and a targeted role smoke confirms Members can finish opportunity setup without a navigation dead-end.

### P1 - [TB-0064] Make the client-only signup CTA labels explicit across `/`
- Evidence: The client-only path on `/` is still labeled three different ways in current source: the header button remains `Sign up` in [Index.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/Index.tsx:288), the hero CTA says `Create Client account` at [Index.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/Index.tsx:357), and the contact-section CTA still uses `Client signup` at [Index.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/Index.tsx:590). Targeted local public signup smoke passed on `127.0.0.1:8080` after a sourced rebuild, so the issue remains wording drift rather than a broken dialog. Current W3C and GOV.UK guidance still favors labels that clearly describe the action users are about to take: [W3C labels guidance](https://www.w3.org/WAI/WCAG22/Understanding/headings-and-labels.html) and [GOV.UK button guidance](https://design-system.service.gov.uk/components/button/).
- Why it matters: After the landing split, the generic header `Sign up` label still reads like a universal path even though `/` is client-only and `/bums` owns Bum applications. Mixed client CTA labels add hesitation right where a buyer or operator is deciding whether they are in the correct funnel.
- Recommendation: Standardize the client-only signup action on `/` to one explicit client label system such as `Create Client account`, and reserve Bum-application language for `/bums` and Bum-specific CTAs. If the header needs a shorter mobile label, add nearby context that still makes the client-only action unmistakable.
- Acceptance criteria: Every CTA on `/` that opens the client-locked signup dialog uses explicit client wording; the public header no longer uses generic `Sign up` for that path; Bum application CTAs remain Bum-specific; and smoke plus visual tests are updated to the approved label set.

### P2 - [TB-0065] Reduce the first-layer consent banner footprint on mobile
- Evidence: Hosted screenshot `mobile-chrome-public-consent-settings.png` from `Complete Visual UI Audit` run `27325075884` still shows the first consent layer occupying most of the initial phone-height view and competing with the hero’s first CTA instead of yielding after the user sees the headline. Current source still renders that first layer with `max-h-[78vh]`, two text blocks, and a stacked action group in [ConsentManager.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/ConsentManager.tsx:93). The privacy flow itself remains healthy in source and local smoke; the issue is presentation weight, not missing controls. Current ICO guidance still emphasizes clear first-layer choices while allowing more detailed controls to sit behind the secondary settings flow: [ICO guidance overview](https://ico.org.uk/for-organisations/direct-marketing-and-privacy-and-electronic-communications/guidance-on-the-use-of-storage-and-access-technologies/about-this-guidance/) and [ICO consent-in-practice guidance](https://ico.org.uk/for-organisations/direct-marketing-and-privacy-and-electronic-communications/guidance-on-the-use-of-storage-and-access-technologies/how-do-we-manage-consent-in-practice/).
- Why it matters: The consent mechanism is still the dominant first mobile interaction because it visually overtakes the hero before the visitor can act on the page, turning a compliance step into the primary product experience.
- Recommendation: Keep equally prominent consent choices, but tighten the first-layer copy and action layout so the hero headline and one primary CTA remain visible and unobscured on first mobile visit. Keep the granular category explanations and toggles behind `Customize`, and record Trust or Legal review for any copy or layout changes.
- Acceptance criteria: On first visit at mobile widths, the initial consent layer preserves equally prominent choice controls while leaving the hero headline and one primary CTA visible and tappable; detailed category explanations stay behind `Customize`; and the revised presentation is reviewed by the Trust or Legal owner.
