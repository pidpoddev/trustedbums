# Trusted Bums UX Optimization Backlog

_Last updated: 2026-06-06 by Codex daily UX consultant automation._

## Executive Read

The highest-leverage UX issues are still concentrated in trust-sensitive intake and client-finance task routing. Public signup still clears a manually typed company name when the email stops matching a known client workspace, and the homepage contact form still relies on destructive toasts instead of field-level recovery.

Fresh live evidence still shows a client-finance workflow break in production. GitHub `E2E Smoke` run [`26933527284`](https://github.com/pidpoddev/trustedbums/actions/runs/26933527284) on 2026-06-04 reached authenticated finance routes, then failed because finance global search returned the user to `/client/dashboard` instead of `/client/payments`, while `/client/payments` rendered duplicate `Customer Payment Reports` headings.

Local browser verification remains blocked on this runner. On 2026-06-06, `pnpm run qa:env` still passed after sourcing `.env.qa`, but `curl -I -L --max-time 20 https://trustedbums.com` still failed with DNS resolution timeout. Recommendations below therefore stay grounded in current source review plus GitHub-hosted QA evidence rather than fresh local screenshots.

## Active Recommendations

### P1 - Preserve typed company names in the public signup flow
- Evidence: [SignupIntentDialog.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/SignupIntentDialog.tsx:44) still calls `setCompanyName("")` whenever the selected role is `CLIENT` and `getKnownClientForEmail(email)` returns no match, so editing the email can erase a manually entered company name before signup is complete. Baymard still treats lost entered form data as a major form-friction pattern ([Baymard](https://baymard.com/learn/form-design)).
- Why it matters: This is an early trust moment for prospective clients. Silent data loss makes the workflow feel brittle before the visitor has committed any time or information.
- Recommendation: Preserve manually entered `companyName` unless the user explicitly accepts a matched workspace or switches away from the client flow. Keep workspace-match guidance, but separate "existing workspace match" behavior from "first-time workspace request" behavior.
- Acceptance criteria: Editing the email no longer clears a manually entered company name when no workspace match exists; matched workspaces still prefill clearly; first-time workspace requests retain the typed company name unless the user intentionally replaces it.

### P1 - Add inline validation and reassurance copy to the homepage contact form
- Evidence: [Index.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/Index.tsx:144) still validates only `name`, `email`, and `message` inside `submitContactForm` and responds with destructive toasts, while the rendered form still has no inline errors, linked error summary, or `aria-invalid` wiring. Current W3C and GOV.UK guidance continues to recommend field-level error identification plus a linked summary for invalid submit states ([W3C form validation](https://design-system.w3.org/styles/form-validation.html), [GOV.UK error summary](https://design-system.service.gov.uk/components/error-summary/), [W3C notifications](https://www.w3.org/WAI/tutorials/forms/notifications/)).
- Why it matters: This is the main public handoff for buyers and future Bums. Toast-only validation forces users to guess which field failed and weakens trust exactly when they are deciding whether to share contact details.
- Recommendation: Add field-level validation messages, a linked error summary on submit, `aria-invalid` and `aria-describedby` wiring, and a short expectation-setting note near submit that explains reply timing or handling.
- Acceptance criteria: Invalid submit shows inline errors and a linked summary; invalid fields expose accessible error state; users can recover without guessing; the submit area includes a short reassurance message about follow-up timing or handling.

### P1 - Make client access recovery and terms recovery explicit on the dashboard
- Evidence: [ClientAccessRoute.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/ClientAccessRoute.tsx:29) still redirects blocked client users to `/client/dashboard` with `state={{ deniedFrom: location.pathname }}`, but [ClientDashboard.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientDashboard.tsx:52) still does not read `deniedFrom` or explain the redirect. The finance and non-finance next-action cards also still send agreement recovery to `/client/profile` at [ClientDashboard.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientDashboard.tsx:114) and [ClientDashboard.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientDashboard.tsx:129), even though dedicated agreement flows live at [ClientAgreements.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientAgreements.tsx:1) and [ClientTerms.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientTerms.tsx:1). W3C guidance still recommends telling users what happened and what to do next after blocked actions ([W3C notifications](https://www.w3.org/WAI/tutorials/forms/notifications/)).
- Why it matters: Silent redirects and misdirected recovery links make role boundaries and legal requirements feel arbitrary, which increases support load and lowers confidence in the client portal.
- Recommendation: Surface a dismissible dashboard notice when `deniedFrom` is present, name the current client role, explain the blocked destination, and route agreement actions to `/client/agreements` or `/client/terms` instead of `Company Profile`.
- Acceptance criteria: A redirected user sees a visible notice naming the blocked page, current role, and next allowed action; outdated-terms prompts route to the agreement flow; ordinary dashboard visits do not show the notice.

### P1 - Fix client-finance search routing and payment-page hierarchy
- Evidence: GitHub `E2E Smoke` run [`26933527284`](https://github.com/pidpoddev/trustedbums/actions/runs/26933527284) failed on 2026-06-04 because client-finance search for `payments` navigated to `https://trustedbums.com/client/dashboard` instead of `/client/payments`, and because Playwright found two exact-match `Customer Payment Reports` headings on the payment screen. Source still explains both issues: [PortalGlobalSearch.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/PortalGlobalSearch.tsx:464) submits Enter to `results[0]` without weighting exact page matches, and [ClientPayments.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientPayments.tsx:384) plus [ClientPayments.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientPayments.tsx:684) still render the same finance label as both the page title and a secondary card heading.
- Why it matters: Client-finance is a narrow, high-intent role. Search should land that user directly in the finance workflow they asked for, and the destination page should expose a single obvious primary task.
- Recommendation: Weight exact page-title matches ahead of dashboard-text matches for Enter-to-search behavior, and rename the secondary payment-report section so only one primary heading uses `Customer Payment Reports`.
- Acceptance criteria: Entering `payments` from finance search lands on `/client/payments`; direct finance-page matches outrank dashboard matches for Enter submission; the payment screen exposes one clear primary heading with distinct secondary section labels.

### P2 - Add urgency and next-step cues to the admin handoff queues
- Evidence: [AdminHandoffs.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminHandoffs.tsx:255) still sorts target responses, intro requests, and public contact conversions by age, but the queue tables at [AdminHandoffs.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminHandoffs.tsx:355) still expose only status, owner, created time, update, and actions. Underlying data already includes `admin_next_action`, `admin_priority`, and `notification_error` in [portalApi.ts](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/portalApi.ts:698) and [portalApi.ts](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/portalApi.ts:737), while notification failure is still surfaced separately in [AdminTroubleshooting.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminTroubleshooting.tsx:102). Current USWDS table guidance still favors scannable summaries for dense operational tables ([USWDS tables](https://designsystem.digital.gov/components/table/)).
- Why it matters: This is the admin rescue queue for leads, target responses, and intro requests. Without urgency and next-step context, admins must infer what is actually stuck and what to do next.
- Recommendation: Surface priority, next action, stale/SLA state, and notification failure directly in each queue row or responsive summary so admins can triage without leaving the page.
- Acceptance criteria: Open items older than the chosen SLA threshold are visually distinct; each row shows the next required action; contact submissions with failed notifications are visible in the handoff queue; admins can distinguish fresh, stale, and unowned work at a glance.
