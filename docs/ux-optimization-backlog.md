# Trusted Bums UX Optimization Backlog

_Last updated: 2026-05-31 by Codex daily UX consultant automation._

## Executive Read

The highest-confidence UX gaps are still concentrated in first-touch conversion and role clarity. The public signup dialog can still erase a manually entered company name, and the homepage contact form still relies on toast-only validation instead of field-level recovery and trust-setting copy at the point of submission.

Admin handoffs remain the biggest workflow-efficiency gap introduced by recent portal changes. The queue now centralizes rescue work, but it still hides urgency, next-step context, and notification health even though those signals already exist in the data model. Authenticated QA evidence improved this run: `pnpm run qa:env` passed, `pnpm run test` passed, `pnpm run lint` returned only 7 existing warnings, and Playwright produced fresh desktop screenshot artifacts for client admin, client finance, and admin routes. Broader route-walkthrough evidence is still partial because the interaction audit still stops on a false-positive skip-link assertion and one client-finance Chromium flow hit a transient `ERR_ADDRESS_UNREACHABLE` while returning to `/client/dashboard`.

## Active Recommendations

### P1 - Preserve typed company names in the public signup flow
- Evidence: [src/components/SignupIntentDialog.tsx:44](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/SignupIntentDialog.tsx:44) still resets `companyName` whenever the role is `CLIENT` and `getKnownClientForEmail(email)` returns no match, so editing the email can erase a manual company entry.
- Why it matters: This is a top-of-funnel trust step. Silent data loss makes account creation feel brittle before a prospect has any reason to trust the platform.
- Recommendation: Preserve a manually entered `companyName` unless the user explicitly accepts a matched workspace. Keep the match helper text, but separate it from the first-time workspace path so email edits do not wipe user input.
- Acceptance criteria: Editing the email no longer clears a manually entered company name when no match exists; matched workspaces still prefill clearly; the dialog supports both existing-workspace and first-time-workspace paths without silent data loss.

### P1 - Add field-level validation and reassurance to the homepage contact form
- Evidence: [src/pages/Index.tsx:144](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/Index.tsx:144) still validates only `name`, `email`, and `message` inside `submitContactForm` and shows destructive toasts on failure, while the rendered form at [src/pages/Index.tsx:608](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/Index.tsx:608) through [src/pages/Index.tsx:652](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/Index.tsx:652) still has no inline error copy, linked summary, `aria-invalid` wiring, or response-expectation copy near submit. Current W3C and GOV.UK guidance still favors field-level errors, linked summaries, and explicit invalid-state wiring ([W3C User Notification](https://www.w3.org/WAI/tutorials/forms/notifications/), [W3C Design System form errors](https://design-system.w3.org/styles/form-errors.html), [GOV.UK Error summary](https://design-system.service.gov.uk/components/error-summary/)).
- Why it matters: This is the main public handoff for buyers and prospective Bums. Toast-only validation increases recovery friction and weakens trust exactly when a visitor is deciding whether to share contact details.
- Recommendation: Add inline validation for each invalid field, an error summary on submit, `aria-invalid` and described-by wiring, and a short trust cue near the submit button that explains reply timing and how the submission is handled.
- Acceptance criteria: Invalid submit shows field-level messages and a linked summary; invalid fields expose accessible error state; recovery does not require guessing which field failed; the submit area includes a short expectation-setting note such as response window or handling reassurance.

### P1 - Add urgency and next-step cues to the admin handoff queues
- Evidence: [src/pages/admin/AdminHandoffs.tsx:255](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminHandoffs.tsx:255) sorts target responses, intro requests, and public contact conversions by age, but the tables at [src/pages/admin/AdminHandoffs.tsx:355](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminHandoffs.tsx:355) through [src/pages/admin/AdminHandoffs.tsx:447](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminHandoffs.tsx:447) still expose only status, owner, created timestamp, and update actions. The supporting contact model already includes `admin_next_action`, `admin_priority`, and `notification_error` at [src/lib/contactApi.ts:33](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/contactApi.ts:33), but the queue does not surface them.
- Why it matters: This page is positioned as the place to rescue stalled handoffs. Without urgency and next-step cues, admins must infer what is actually stuck, which increases response delays for leads, Bum responses, and intro requests.
- Recommendation: Add visible aging states and next-step context to each queue. Start with an SLA bucket or stale badge, owner follow-up text, and a compact summary that distinguishes new, stale, and unowned work. Surface notification failure state for public contact submissions.
- Acceptance criteria: Open items older than the chosen SLA threshold are visually distinct; admins can see the next required action without opening another page; queue summaries distinguish fresh vs stale work; notification failures are visible for contact submissions that need manual follow-up.

### P1 - Explain blocked client-role redirects on the dashboard
- Evidence: [src/components/ClientAccessRoute.tsx:29](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/ClientAccessRoute.tsx:29) still redirects unauthorized client users to `/client/dashboard` with `state={{ deniedFrom: location.pathname }}`, but [src/pages/client/ClientDashboard.tsx:52](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientDashboard.tsx:52) still does not read `deniedFrom` or render any denied-access explanation for either client dashboard shell. Current W3C notification guidance still recommends concise messaging that explains what happened and what the user can do next ([W3C User Notification](https://www.w3.org/WAI/tutorials/forms/notifications/)).
- Why it matters: Silent redirects make role boundaries feel arbitrary, especially for client finance and client member users who are trying to understand what their role can actually do. That increases support load and lowers trust in the portal IA.
- Recommendation: Read `deniedFrom` on the dashboard and show a dismissible notice that names the blocked destination, the current client access role, and the nearest allowed workflow.
- Acceptance criteria: A redirected user sees a visible notice on the next dashboard load naming the blocked destination and current role; the notice includes a next-step action to an allowed page; the notice does not appear on ordinary dashboard visits.

### P2 - Reduce commission-plan request friction to a minimum viable schedule
- Evidence: [src/pages/client/ClientOpportunityNew.tsx:478](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientOpportunityNew.tsx:478) already converts blank later-year commission values to `0` in the request payload, but Year 2 through Year 6+ are still marked `required` at [src/pages/client/ClientOpportunityNew.tsx:1210](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientOpportunityNew.tsx:1210) through [src/pages/client/ClientOpportunityNew.tsx:1266](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientOpportunityNew.tsx:1266). W3C guidance still recommends asking only for what is truly required and clearly identifying required inputs ([W3C Forms Tutorial](https://www.w3.org/WAI/tutorials/forms/), [W3C Validating Input](https://www.w3.org/WAI/tutorials/forms/validation/), [WCAG techniques index updated 2026-04-27](https://w3c.github.io/wcag/techniques/)).
- Why it matters: Clients often know the near-term commission structure before they know the full long-tail payout schedule. Requiring six percentages up front increases hesitation and encourages invented data during opportunity setup.
- Recommendation: Make Year 1 the only default required rate, move later-year percentages behind an optional multi-year schedule choice, and explain how omitted years are handled during admin review.
- Acceptance criteria: A client can submit a commission-plan request with a minimum viable required set; later-year fields are clearly optional unless the user opts into a multi-year schedule; helper text explains fallback handling for omitted years.
