## Active Recommendations

### P0 - Remove the direct public `send-website-email` mail-sending path
- Evidence: Live edge-function inventory on 2026-06-04 shows `send-website-email` deployed with `verify_jwt: false`. Source in [`/Users/ryan.peterson/Documents/Trusted Bums LOCAL/trustedbums/supabase/functions/send-website-email/index.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/functions/send-website-email/index.ts) accepts unauthenticated POSTs, sets `Access-Control-Allow-Origin: *`, and calls Microsoft Graph using `MICROSOFT_CLIENT_SECRET`. A defensive direct POST with an unsupported template returned `400` from the function without auth or anti-abuse gating. The hardened `submit-contact` path verifies origin, Turnstile, and rate limits first.
- Why it matters: A public secret-backed mail sender can flood operations mailboxes, damage sender reputation, and undermine the public-contact business rule that requires abuse checks before mail is sent.
- Recommendation: Make `submit-contact` the only anonymous public contact path. Merge the notification send into `submit-contact` after validation or require a server-only caller proof that browsers and third parties cannot supply.
- Acceptance criteria: Direct unauthenticated calls to `send-website-email` are denied before template parsing; valid homepage contact submissions still send exactly one intended notification through the validated path; invalid Turnstile, origin, honeypot, and rate-limit cases do not send mail.

### P1 - Close the public `email-track` open redirect
- Evidence: Live inventory shows `email-track` deployed with `verify_jwt: false`. Source in [`/Users/ryan.peterson/Documents/Trusted Bums LOCAL/trustedbums/supabase/functions/email-track/index.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/functions/email-track/index.ts) redirects to the raw `u` query parameter after `recordEvent()`, and `recordEvent()` returns silently when the delivery id is missing or invalid. A live `HEAD` probe with an invalid delivery id returned `302 Location: https://example.com/`.
- Why it matters: Attackers can wrap arbitrary destinations in a Trusted Bums Supabase function URL, creating phishing, domain reputation, and security-tool blocking risk.
- Recommendation: Redirect only after a valid delivery row is found and the destination matches an allowlisted scheme and host policy. Return `404` or a safe Trusted Bums landing page for invalid delivery ids, expired links, malformed URLs, and unapproved hosts.
- Acceptance criteria: Invalid delivery ids never redirect off-domain; valid tracked links redirect only to approved `https` destinations stored or validated at send time; tests cover invalid id, missing `u`, non-HTTP schemes, external disallowed host, and approved destination.

### P1 - Revoke broad `public` Data API grants and make exposure opt-in
- Evidence: Historical live SQL from 2026-06-02 showed broad `anon` and `authenticated` table privileges across `public`, including sensitive tables such as `profiles`, `contact_submissions`, `customer_targets`, and admin email tables. Today's connector exposed advisors and edge-function inventory, but not read-only SQL to re-run catalog grant checks. - Acceptance criteria: Sensitive internal tables are not broadly reachable over the Data API; remaining grants are enumerated per object; regression checks prove one allowed and one denied direct API call for every public surface that remains.

### P2 - Constrain invitation redirect URLs for Client Team and Bum invitations
- Evidence: [`/Users/ryan.peterson/Documents/Trusted Bums LOCAL/trustedbums/supabase/functions/client-team/index.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/functions/client-team/index.ts) and [`/Users/ryan.peterson/Documents/Trusted Bums LOCAL/trustedbums/supabase/functions/invite-bum/index.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/functions/invite-bum/index.ts) accept a caller-supplied `redirectUrl` and forward it to Clerk invitations. ## Business Rule Alignment

- `Public contact intake`: `submit-contact` aligns with the public intake rule by validating origin, Turnstile Siteverify, honeypot, and rate limits before inserting `contact_submissions`. `send-website-email` conflicts because it remains directly public and can parse unauthenticated requests outside that validated intake path.
- `Admin email tracking`: `docs/business-access-rules.md` does not yet define public email tracking and click redirect behavior. Current `email-track` behavior conflicts with trust expectations because invalid delivery ids still redirect to arbitrary external URLs. Product Ops and Trust should define whether tracked URLs may ever leave Trusted Bums domains, how long links remain valid, and what invalid links should show.
- `Admin-only operations`: live advisors confirm `admin_dashboard_summary()` and several security-definer helpers remain externally callable in exposed `public`, which conflicts with admin-only observability and RLS-helper intent.
- `Profile bootstrap and client-team approvals`: source now follows the business-rule direction by treating unsafe signup metadata as intent, then using service-role functions plus Clerk verification to assign company and role state. Remaining gaps are deterministic allow/deny proof and redirect allowlisting for invitation flows.
- `Customer targets`: local policy source still treats `bum_saved_items` as sufficient for target reads, while the current business rule denies saved-target-only access unless Product Ops explicitly approves it.
- Required direct checks before closing current access work: denied direct `send-website-email`; allowed `submit-contact`; invalid `email-track` click does not redirect externally; allowed first-domain claim; pending public-email company request; approved same-domain request; denied direct `profiles.company_id` and `profiles.client_access_role` mutation; allowed same-company Client Admin team change; denied cross-company client-team action; allowed explicit-entitlement Bum target read; and denied saved-item-only Bum target read unless the rule changes.
