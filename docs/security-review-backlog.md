# Trusted Bums Security Review Backlog

_Last updated: 2026-05-31 by Codex daily security engineer automation._

## Executive Read

Today’s live recheck removed two stale assumptions from the prior backlog: `submit-contact` is now deployed, and live `customer_targets` RLS is no longer the old broad Bum-read policy. The highest-confidence remaining risk is still authorization bootstrap. Local auth bootstrap and user-sync paths still read role and company hints from Clerk `unsafeMetadata`, Clerk’s current docs still describe `unsafeMetadata` as frontend-writeable, and live `profiles` policy still lets signed-in users create or update their own non-admin profile rows without pinning `company_id` or `client_access_role`. The other active live gaps are the still-public `send-website-email` function, externally callable `SECURITY DEFINER` helpers, and broad `anon`/`authenticated` grants that keep the Data API surface wider than the business contract.

## Active Recommendations

### P0 - Stop client-controlled role and company bootstrap from becoming authoritative
- Evidence: [`src/contexts/AuthContext.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/contexts/AuthContext.tsx:94) still reads `role`, `companyName`, `clientAccessRole`, and `bumId` from Clerk `publicMetadata` and `unsafeMetadata`; [`src/components/SignupIntentDialog.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/SignupIntentDialog.tsx:61) writes `role` and `companyName` into Clerk `unsafeMetadata` during sign-up; [`src/lib/portalApi.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/portalApi.ts:1702) upserts `profiles.role`, `profiles.is_admin`, `profiles.company_id`, and `profiles.client_access_role` from that client-derived state. Clerk’s current metadata docs say `unsafeMetadata` can be read and set from the frontend and can be tampered with by malicious users ([Clerk metadata docs](https://clerk.com/docs/guides/users/extending)). Live Supabase SQL on 2026-05-31 shows `public.profiles` policy `Users can update safe own profile fields`, but its `WITH CHECK` only constrains `id`, `is_admin = false`, and `role in ('CLIENT','BUM')`; it does not pin `company_id` or `client_access_role`. Live grants also still expose `INSERT` and `UPDATE` on `public.profiles` to both `anon` and `authenticated`.
- Why it matters: A signed-in user may be able to attach themselves to another company, switch between Bum and Client posture, or self-assign elevated client finance/admin visibility without an admin-approved identity flow. That is a direct authorization and tenant-isolation risk across the portal, direct Data API, and any policy using `current_company_id()` or profile role fields.
- Recommendation: Make Clerk metadata advisory only for onboarding UX, not authoritative authorization state. Move role, company assignment, Bum identity, and client access role changes onto an admin-verified server path; restrict `profiles` self-write capability to safe preference fields such as display name, timezone, and date format; deny self-service writes to `role`, `is_admin`, `company_id`, `client_access_role`, and any other authorization-bearing columns.
- Acceptance criteria: A signed-in user cannot change their own `role`, `company_id`, `client_access_role`, `is_admin`, or Bum assignment through browser calls, direct Data API, or sign-up metadata; allowed self-service preference edits still work; one denied direct profile mutation and one denied cross-company attachment case are covered by tests.

### P0 - Make `send-website-email` internal-only and keep public intake on `submit-contact`
- Evidence: `docs/business-access-rules.md` requires public contact submissions to include current anti-abuse proof and to fail when verification or throttling fails. Local homepage code in [`src/pages/Index.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/Index.tsx:67) calls [`src/lib/contactApi.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/contactApi.ts:112), which invokes live [`submit-contact`](https://supabase.com/dashboard/project/vaoqvtxqvbptyxddpoju/functions) for Turnstile verification, rate limiting, origin checks, and server-side `contact_submissions` writes. Live `submit-contact` source now enforces those checks before it invokes `send-website-email`. Live `send-website-email` source still accepts unauthenticated `POST`, sets `Access-Control-Allow-Origin: *`, and sends Microsoft Graph mail directly without Turnstile, origin, or throttle checks. Live SQL on 2026-05-31 shows the direct `INSERT` bypass into `public.contact_submissions` has been removed, but the direct mail-sending function remains exposed.
- Why it matters: The intended anti-abuse path is now present, but attackers can still bypass it and directly trigger notification mail. That keeps mailbox-flood, sender-reputation, and support-noise risk alive against the shared `bums@trustedbums.com` operations flow.
- Recommendation: Keep `submit-contact` as the only public contact entrypoint and make `send-website-email` callable only from a trusted server path. At minimum, require verified internal auth or a server-only secret plus strict origin checking before Microsoft Graph mail can be sent.
- Acceptance criteria: Anonymous callers cannot invoke `send-website-email` successfully; homepage submissions still work through `submit-contact`; validated submissions store abuse-control fields and send at most one internal notification.

### P2 - Clarify whether saved targets are a valid Bum entitlement or remove them from target-read RLS
- Evidence: `docs/business-access-rules.md` allows Bum target access only when the Bum has an explicit marketplace relationship such as accepted or assigned work and explicitly denies unrelated target reads through direct Supabase access, portal APIs, extension APIs, or search. Live `extension-api-v1` on 2026-05-31 now aligns with that rule for Bum `/context` requests by returning no `customerTargets`. Live SQL shows the new `public.customer_targets` policy `Bums can read explicitly assigned customer targets`, but one of its live predicates still grants access when a matching `bum_saved_items` row exists. Local source still lets Bums save `CUSTOMER_TARGET` items directly through [`src/lib/portalApi.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/portalApi.ts:2295) and [`src/pages/bum/BumOpportunities.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/bum/BumOpportunities.tsx:709), while the business rules do not currently say whether a saved item is itself an entitlement.
- Why it matters: This is no longer the old broad read exposure, but it is still an access-rule ambiguity on a direct Data API path. If a saved item is not supposed to preserve visibility, the current policy can let target access outlive the actual marketplace relationship.
- Recommendation: Decide explicitly whether a saved target is a valid Bum entitlement. If not, remove `bum_saved_items` from the read policy and test post-acceptance, post-meeting, and post-unsave lifecycle cases. If yes, add that rule and its deny boundaries to `docs/business-access-rules.md`.
- Acceptance criteria: The business rule explicitly states whether saved targets preserve read access; direct table access, portal behavior, and extension behavior all match that rule for one allowed saved-item case and one denied saved-item-only case.

### P1 - Remove externally callable `SECURITY DEFINER` RPCs from exposed `public`
- Evidence: Live Supabase security advisors on 2026-05-31 still flag `anon` and `authenticated` execution of `public.admin_dashboard_summary()`, `current_company_id()`, `is_admin()`, `is_bum()`, `company_has_customer_targets(uuid)`, `is_conversation_participant(uuid)`, `conversation_company_id(uuid)`, and `can_add_conversation_participant(uuid)`. Live `information_schema.routine_privileges` also shows `PUBLIC` execute grants on several of those helper functions. The admin UI still fetches browser RPC `admin_dashboard_summary()` in [`src/lib/portalApi.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/portalApi.ts:4519), while route guards and tests still treat `/admin` as admin-only in [`src/components/ProtectedRoute.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/ProtectedRoute.tsx:1) and [`src/test/routeGuards.test.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/test/routeGuards.test.tsx:1). Supabase’s current guidance says `SECURITY DEFINER` helpers used in RLS do not need to live in exposed schemas or be exposed to PostgREST ([security-definer guidance](https://supabase.com/docs/guides/troubleshooting/do-i-need-to-expose-security-definer-functions-in-row-level-security-policies-iI0uOw)).
- Why it matters: This expands the externally callable auth-helper and admin-summary surface beyond the documented business contract, increases enumeration risk, and weakens trust that route guards and database exposure actually match.
- Recommendation: Move RLS-only helpers into a private schema or revoke `EXECUTE` from `anon`, `authenticated`, and `PUBLIC` wherever direct RPC access is not an intentional product contract. Serve admin summaries through an admin-verified server path instead of public RPC.
- Acceptance criteria: Supabase security advisors return no externally callable `SECURITY DEFINER` findings for admin-only or internal helper functions; anonymous and non-admin direct RPC calls fail by design; `/admin` still works for admins.

### P1 - Revoke default broad Data API grants and make exposure opt-in
- Evidence: Live SQL on 2026-05-31 shows `anon` and `authenticated` still hold broad table grants including `SELECT`, `INSERT`, `UPDATE`, `DELETE`, `TRUNCATE`, `TRIGGER`, and `REFERENCES` on `profiles`, `customer_targets`, and `performance_metric_events`, plus `SELECT`, `UPDATE`, `DELETE`, `TRUNCATE`, `TRIGGER`, and `REFERENCES` on `contact_submissions`, even though the business rules only intend narrow subsets of those operations. Supabase’s current API security guidance says grants and RLS work together, default grants on `public` are historically broad, and existing projects should revoke automatic grants so exposure becomes explicit. Supabase’s May 7, 2026 changelog also notes that new projects now default away from auto-exposing new `public` tables, which reinforces that older projects still need an explicit grant cleanup ([Securing your API](https://supabase.com/docs/guides/api/securing-your-api), [May 2026 changelog](https://supabase.com/changelog/45702-developer-update-may-2026)).
- Why it matters: RLS currently blocks some dangerous row access, but the exposed privilege surface is wider than the product contract and makes future policy mistakes, helper exposure, or newly created tables/functions easier to turn into real vulnerabilities.
- Recommendation: Revoke default `public` schema table and function grants for `anon`, `authenticated`, and `PUBLIC`, then regrant only the exact objects and verbs the product intentionally exposes. Treat internal helper tables and functions as private by default.
- Acceptance criteria: Default broad grants are revoked in `public`; `profiles`, `contact_submissions`, `customer_targets`, `performance_metric_events`, and internal helper functions expose only the minimum intended verbs; regression tests cover one allowed and one denied call for each public surface that remains.

## Business Rule Alignment

- `Profiles and authorization bootstrap`: the starter role matrix implies Admin, Client Admin, Client Finance, Client Member, and Bum access are assigned roles, not self-declared browser state. Current code and live `profiles` policy still allow client-controlled role and tenant hints to influence authoritative access state, and `docs/business-access-rules.md` does not yet define which profile fields are self-editable versus admin-controlled.
- `Public contact intake`: `docs/business-access-rules.md` allows visitor submissions only with current anti-abuse proof and denies requests that fail verification, rate limits, nonce checks, or other server-side abuse controls. Live `submit-contact` now aligns with that rule for homepage submissions, but live public `send-website-email` calls still conflict with it because they remain callable outside the verified intake path.
- `Customer Targets`: the deployed `extension-api-v1` and live `customer_targets` RLS are materially closer to the rule that unrelated Bums must not see target data, but the live policy still treats `bum_saved_items` as a read entitlement without an explicit business rule that says saved targets should preserve access.
- `Admin-only operations`: route guards and tests still treat `/admin` and `/admin/performance` as admin-only, but live public RPC execute grants on `admin_dashboard_summary()` and helper functions conflict with that intent.
- Required direct checks before closing this access work: one denied direct profile mutation for `company_id`; one denied direct profile mutation for `client_access_role`; one denied self-switch between Bum and Client; one validated public contact submit through the intended path; one denied direct `send-website-email` call; one allowed Bum explicit-entitlement target read; one denied Bum no-relationship target read; one allowed client own-company target read; one denied cross-company client read; one denied anon/non-admin direct RPC call to admin-only helpers; and one explicit saved-item-only target-read case that proves the final business rule.

## Time-Sensitive Threat Notes

- `pnpm audit --prod --json` on 2026-05-31 still reports `js-cookie` `GHSA-qjx8-664m-686j` as a high-severity runtime dependency through `@clerk/react -> @clerk/shared`; first patched line: `>=3.0.6` ([GitHub Advisory](https://github.com/advisories/GHSA-qjx8-664m-686j)).
- The same audit still reports `uuid` `GHSA-w5hq-g745-h8pq` as a moderate runtime dependency through `@clerk/chrome-extension`; first patched line: `>=11.1.1` ([GitHub Advisory](https://github.com/advisories/GHSA-w5hq-g745-h8pq)).
- Clerk’s current metadata docs state that `unsafeMetadata` can be read and set from the frontend and can be tampered with by malicious users, so it should not be used as an authoritative authorization source ([Clerk metadata docs](https://clerk.com/docs/guides/users/extending)).
- Cloudflare’s current Turnstile guidance says server-side Siteverify is mandatory, tokens are single-use, and replayed or expired tokens must fail validation ([Validate the token](https://developers.cloudflare.com/turnstile/get-started/server-side-validation/), [Token validation analytics](https://developers.cloudflare.com/turnstile/turnstile-analytics/token-validation/)).
- Supabase’s current API security guidance says exposed tables should grant only the privileges each role needs, and existing projects should explicitly revoke default broad grants on `public` if they want exposure to be opt-in ([Securing your API](https://supabase.com/docs/guides/api/securing-your-api)).
- Supabase’s May 7, 2026 platform update says new projects can now opt out of automatic Data API exposure for `public` tables and that this exposure is becoming opt-in by default, which makes explicit grant cleanup more urgent on older projects that still carry historical broad grants ([May 2026 changelog](https://supabase.com/changelog/45702-developer-update-may-2026)).
- Live Supabase security advisors still warn that leaked password protection is disabled. That is lower priority if Clerk is the only production login path, but it should be explicitly accepted or enabled before any Supabase password auth is exposed ([Password security](https://supabase.com/docs/guides/auth/password-security)).
