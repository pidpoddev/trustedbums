Current local source in [supabase/functions/email-track/index.ts](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/supabase/functions/email-track/index.ts) now allowlists `EMAIL_TRACKING_ALLOWED_HOSTS`, rejects non-`https` or disallowed hosts, and returns `404` when the delivery id is missing or invalid.
- Why it matters: The live function still allows a Trusted Bums-branded open redirect, which is a phishing, browser-reputation, and trust-boundary problem even though the fix already exists locally.
- Recommendation: Deploy the current local `email-track` implementation, then add a post-deploy smoke check for one denied invalid-delivery click and one allowed approved-host click.
- Acceptance criteria: Live `email-track` source matches the hardened local implementation; invalid delivery ids never redirect; invalid or non-allowlisted destinations return an error instead of a `302`; approved `https` Trusted Bums destinations still redirect and record the click event.

### P1 - Constrain invitation redirect URLs for Client Team and Bum invitations
- Evidence: Current source in [supabase/functions/client-team/index.ts](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/supabase/functions/client-team/index.ts) and [supabase/functions/invite-bum/index.ts](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/supabase/functions/invite-bum/index.ts) still accepts a caller-provided `redirectUrl`, returns it unchanged from `normalizeRedirectUrl()` when present, and passes it into Clerk `redirect_url`.
- Why it matters: Invitation emails are high-trust messages. A caller-controlled redirect target turns an otherwise authorized invite into a phishing or brand-trust risk if an admin account or upstream caller is misused.
- Recommendation: Allow only configured Trusted Bums and Clerk sign-in origins, or ignore caller-supplied redirect URLs entirely and choose the final redirect server-side.
- Acceptance criteria: External redirect URLs are rejected or normalized to approved origins; default login redirects still work for valid invitations; tests cover one approved origin, one missing origin fallback, and one disallowed external origin.

### P1 - Align Bum saved-target reads with the current business rule
- Evidence: [docs/business-access-rules.md](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/docs/business-access-rules.md) still says a Bum should not gain `customer_targets` read access from a saved item alone unless Product Ops explicitly approves that entitlement. Current migration [supabase/migrations/20260529180000_restrict_bum_customer_target_reads.sql](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/supabase/migrations/20260529180000_restrict_bum_customer_target_reads.sql) still grants a read path when a `bum_saved_items` row exists for the target.
- Why it matters: If saved items count as entitlement, target visibility can outlive the accepted workflow relationship that was supposed to justify access.
- Recommendation: Either remove the saved-item predicate from the target-read policy or update the business rule to define exactly when saved targets remain visible, which fields remain visible, and how that visibility is revoked.
- Acceptance criteria: Business rule text and target-read policy agree; one allowed explicit-entitlement read and one denied saved-item-only read are covered by deterministic tests or seeded QA checks.

- Acceptance criteria: Each authorization-bearing service-role path has at least one allowed and one denied case, with audit-event expectations where state changes occur.
