## Active Recommendations

### P1 - Upgrade the root web Clerk dependency chain past the `js-cookie` advisory
- Evidence: The current importer in [pnpm-lock.yaml](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/pnpm-lock.yaml) still resolves root `@clerk/react` to `6.6.4`, which pulls `@clerk/shared` `4.12.0` and `js-cookie` `3.0.5`. The same lockfile also shows the extension-side Clerk path already on `@clerk/react` `6.7.1` with `@clerk/shared` `4.13.1` and `js-cookie` `3.0.7`. GitHub's advisory `GHSA-qjx8-664m-686j`, updated May 21, 2026, marks `js-cookie` `<= 3.0.5` as affected and `3.0.7` as patched.
- Why it matters: Trusted Bums uses Clerk on the primary web auth path, so leaving the web app on the older Clerk shared package keeps the shipped login/session surface on the vulnerable cookie helper even though the extension-side dependency graph is already patched.
- Recommendation: Upgrade the root web Clerk packages and regenerate the lockfile until the web importer no longer resolves through `@clerk/shared` `4.12.0` and `js-cookie` `3.0.5`.
- Acceptance criteria: The root web app no longer resolves `js-cookie` `3.0.5`; current auth flows still pass lint and targeted auth/route tests; any remaining Clerk version split between web and extension is intentional and documented.

### P1 - Align Bum saved-target reads with the current business rule
- Evidence: [docs/business-access-rules.md](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/docs/business-access-rules.md) still says a Bum should not gain `customer_targets` read access from a saved item alone unless Product Ops explicitly approves that entitlement. Current migration [supabase/migrations/20260529180000_restrict_bum_customer_target_reads.sql](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/supabase/migrations/20260529180000_restrict_bum_customer_target_reads.sql) still grants a read path when a `bum_saved_items` row exists for the target.
- Why it matters: If saved items count as entitlement, target visibility can outlive the accepted workflow relationship that was supposed to justify access.
- Recommendation: Either remove the saved-item predicate from the target-read policy or update the business rule to define exactly when saved targets remain visible, which fields remain visible, and how that visibility is revoked.
- Acceptance criteria: Business rule text and target-read policy agree; one allowed explicit-entitlement read and one denied saved-item-only read are covered by deterministic tests or seeded QA checks.

- Why it matters: Trusted Bums increasingly relies on service-role edge functions for authorization-bearing workflows. Without explicit allow/deny regression tests, a future refactor can bypass the intended RLS or business-rule boundary without tripping the current test set.
- Recommendation: Add focused test or fixture coverage for first-domain claim, public-email company review, same-domain approval, denied cross-company client-team changes, denied direct auth-field mutation, denied external invitation redirects, and admin-only email operations.
- Acceptance criteria: Each authorization-bearing service-role path has at least one allowed and one denied case, with audit-event expectations where state changes occur.

Current app source still uses `<BrowserRouter>` in [src/App.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/App.tsx). [Source](https://github.com/advisories/GHSA-2j2x-hqr9-3h42)
- Current Supabase API guidance says functions are controlled by `EXECUTE` grants rather than RLS, recommends reviewing any `SECURITY DEFINER` functions carefully, and recommends an opt-in exposure model or dedicated API schema. [Source](https://supabase.com/docs/guides/api/securing-your-api)
- Current Supabase RLS guidance warns against using end-user-modifiable metadata for authorization decisions and recommends keeping internal helper surfaces out of exposed paths when possible. [Source](https://supabase.com/docs/guides/database/postgres/row-level-security)
- Current Clerk docs still say `unsafeMetadata` can be updated from the frontend with `User.updateMetadata()`, so Trusted Bums must continue treating it as onboarding input only, never as authoritative authorization state. [Source](https://clerk.com/docs/guides/users/extending)
- Current Cloudflare Turnstile guidance still expects server-side validation plus checks on expected `action` and hostname. [Source](https://developers.cloudflare.com/turnstile/get-started/server-side-validation/)
