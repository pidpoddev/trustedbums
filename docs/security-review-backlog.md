## Time-Sensitive Threat Notes

- Current production dependency audit is clean. `corepack pnpm audit --prod --json` returned zero advisories in this run, so the earlier `uuid` and `react-router` backlog notes are now stale and have been removed from the active queue.
- Current Supabase API guidance says grants and RLS work together for Data API access, functions are controlled by `EXECUTE` grants rather than RLS, and existing `public`-schema defaults can expose new tables or functions earlier than intended. That is directly relevant to the new `public.admin_scrum_items` rollout and to any future admin-only table or helper added in `public`. [Source](https://supabase.com/docs/guides/api/securing-your-api)
- Current Supabase RLS guidance warns against using user-modifiable metadata for authorization decisions. It explicitly calls out user-editable metadata as unsafe for authorization use, which matches the current Trusted Bums rule that Clerk onboarding metadata remains non-authoritative input only. [Source](https://supabase.com/docs/guides/database/postgres/row-level-security)
- Current Clerk metadata guidance still says frontend code can update `unsafeMetadata` through `User.updateMetadata()`. Trusted Bums should continue treating that surface as onboarding input only, never as the source of truth for role, company, or admin authority. [Source](https://clerk.com/docs/guides/users/extending)
- Current Cloudflare Turnstile guidance still requires server-side token validation and recommends checking expected action and hostname. Keep that validation on the public intake path every time those functions change. [Source](https://developers.cloudflare.com/turnstile/get-started/server-side-validation/)
