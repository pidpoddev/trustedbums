# Trusted Bums Security Review Backlog

_Last updated: 2026-06-04 by Codex daily security engineer automation._

## Executive Read

Current highest-confidence risk is still the public mail-sending and tracking surface. `send-website-email` remains anonymously callable and secret-backed, while `email-track` still redirects invalid delivery IDs to arbitrary external URLs. The other durable issues are narrower and source-backed: invitation functions still trust caller-supplied redirect URLs, customer-target RLS still grants Bum access from saved items despite the current business rule denying that default, and the app still ships three runtime advisories (`js-cookie`, `uuid`, `react-router`).

Live Supabase trust-boundary validation regressed during this run. The connector could still read project metadata, publishable keys, and edge-function inventory for `vaoqvtxqvbptyxddpoju`, but the security-advisor call now requires reauthentication and arbitrary read-only SQL/catalog inspection was not available. I kept only findings that are still grounded in current source, runnable checks, current business rules, prior live evidence explicitly labeled as historical, or current vendor/advisory guidance.

## Active Recommendations

### P0 - Remove the direct public `send-website-email` mail-sending path
- Evidence: Live edge-function inventory still shows `send-website-email` deployed with `verify_jwt: false`. [`supabase/functions/send-website-email/index.ts`](../supabase/functions/send-website-email/index.ts) accepts anonymous POSTs with `Access-Control-Allow-Origin: *` and sends Microsoft Graph mail with `MICROSOFT_CLIENT_SECRET`. A defensive unauthenticated probe on 2026-06-04 still reached template parsing and returned `{"error":"Unsupported email template."}` instead of rejecting the caller first. [`supabase/functions/submit-contact/index.ts`](../supabase/functions/submit-contact/index.ts) already performs origin, Turnstile, honeypot, and rate-limit checks before invoking this function.
- Why it matters: This preserves a public secret-backed email sender outside the approved public-intake abuse controls, creating mailbox flood, sender-reputation, and marketplace trust risk.
- Recommendation: Make `submit-contact` the only anonymous path that can trigger website-contact email. Either merge the mail send into `submit-contact` after validation or require a server-only proof that browsers and third parties cannot mint.
- Acceptance criteria: Direct unauthenticated calls to `send-website-email` fail before template parsing or Graph calls; valid homepage submissions still create one intake record and one intended notification; invalid Turnstile, origin, honeypot, or throttled requests do not send mail.

### P1 - Stop `email-track` from redirecting invalid delivery IDs to arbitrary destinations
- Evidence: Live edge-function inventory still shows `email-track` deployed with `verify_jwt: false`. [`supabase/functions/email-track/index.ts`](../supabase/functions/email-track/index.ts) redirects to the raw `u` query parameter even when `recordEvent()` cannot find a matching delivery. A live 2026-06-04 `HEAD` probe to `/functions/v1/email-track/click?d=00000000-0000-0000-0000-000000000000&u=https://example.com/` returned `302 Location: https://example.com/`.
- Why it matters: A Trusted Bums-hosted redirect that works for invalid IDs can be abused in phishing and reputation attacks and can increase browser or mail-tool distrust of the domain.
- Recommendation: Redirect only after a valid delivery row is found and the destination passes an explicit allowlist policy. Return `404` or a safe Trusted Bums landing page for invalid IDs, malformed URLs, unapproved hosts, and expired links.
- Acceptance criteria: Invalid delivery IDs never redirect off-domain; only approved `https` destinations can be followed for valid tracked links; tests cover invalid IDs, missing `u`, non-HTTP schemes, disallowed hosts, and allowed destinations.

### P1 - Pin invitation redirects to approved Trusted Bums origins
- Evidence: [`supabase/functions/client-team/index.ts`](../supabase/functions/client-team/index.ts) and [`supabase/functions/invite-bum/index.ts`](../supabase/functions/invite-bum/index.ts) both accept caller-supplied `redirectUrl` values and forward them to Clerk invitations after auth checks. `normalizeRedirectUrl()` only trims the provided value or falls back to `origin + "/login"`; it does not validate host or scheme.
- Why it matters: Invitation emails are high-trust. Allowing arbitrary redirect targets lets an authorized or compromised caller turn a legitimate invitation into a phishing path.
- Recommendation: Replace arbitrary redirect acceptance with a hard allowlist of Trusted Bums app origins and the approved Clerk sign-in callback origin set. Ignore or reject any unapproved host, protocol-relative path, or non-HTTPS destination.
- Acceptance criteria: Only configured Trusted Bums origins are accepted as invitation redirects; rejected redirects are logged without sending the invite; tests cover allowed production/local URLs and denied external, protocol-relative, malformed, and mixed-scheme values.

### P1 - Remove saved-target Bum reads unless Product Ops explicitly changes the rule
- Evidence: `docs/business-access-rules.md` currently says a Bum without an explicit accepted or assigned relationship is denied target access and specifically says a saved target alone should not grant visibility unless Product Ops approves it. Current migration [`supabase/migrations/20260529180000_restrict_bum_customer_target_reads.sql`](../supabase/migrations/20260529180000_restrict_bum_customer_target_reads.sql) still grants `customer_targets` reads when a matching `bum_saved_items` row exists.
- Why it matters: This is a direct business-rule mismatch at the RLS boundary. It can expose client target details to Bums who only bookmarked a target and never received an approved workflow relationship.
- Recommendation: Remove the `bum_saved_items` branch from the Bum `customer_targets` select policy unless Product Ops intentionally approves saved-target visibility and updates the business rule first.
- Acceptance criteria: A Bum can read a target only through explicit accepted workflow relationships defined in the business rule; a saved-item-only Bum deny test fails through direct Supabase access, extension API context, and portal surfaces unless the rule is explicitly changed.

### P1 - Move exposed `SECURITY DEFINER` helpers and admin summary access behind a private contract
- Evidence: Current source still defines multiple `SECURITY DEFINER` helpers in exposed `public`, including `current_company_id()`, `is_admin()`, `is_bum()`, `company_has_customer_targets(uuid)`, conversation helper functions, and `admin_dashboard_summary()`. [`supabase/migrations/20260529001000_add_admin_dashboard_summary.sql`](../supabase/migrations/20260529001000_add_admin_dashboard_summary.sql) grants `authenticated` execute on `public.admin_dashboard_summary()`, and [`src/lib/portalApi.ts`](../src/lib/portalApi.ts) calls it directly from the browser with `.rpc("admin_dashboard_summary")`. Historical live advisor output from 2026-06-04 flagged executable `SECURITY DEFINER` functions in `public`, but this run could not freshly re-run advisors because the Supabase advisor tool now requires reauthentication.
- Why it matters: Even when inner checks reject non-admin users, exposed security-definer routines enlarge the externally callable privilege boundary and reduce margin for future policy or function mistakes. Supabase guidance says security-definer functions should not live in exposed schemas.
- Recommendation: Move RLS-only helpers into a private schema, revoke direct `EXECUTE` where PostgREST/browser callers do not need it, and replace browser RPC access to `admin_dashboard_summary()` with an admin-verified server path or a tightly scoped private routine surface.
- Acceptance criteria: Security advisors no longer report executable security-definer functions in exposed schemas for broad roles; admin dashboard summary still works for Admin users and fails for non-admin callers before privileged reads run; helper functions used only inside RLS are not directly callable through PostgREST.

### P1 - Clear current runtime dependency advisories
- Evidence: `pnpm audit --prod --json` on 2026-06-04 reports three runtime advisories: high `js-cookie` `GHSA-qjx8-664m-686j` via `@clerk/react` fixed in `>=3.0.6`; moderate `uuid` `GHSA-w5hq-g745-h8pq` via `@clerk/chrome-extension` fixed in `>=11.1.1`; and moderate `react-router` `GHSA-2j2x-hqr9-3h42` via `react-router-dom`/`react-router` fixed in `6.30.4`. `package.json` still pins `@clerk/react` `^6.6.2`, `@clerk/chrome-extension` `^3.1.30`, and `react-router-dom` `^6.30.1`.
- Why it matters: These packages sit on authentication, extension, and routing surfaces. The React Router advisory is open-redirect related, which overlaps with the repo's own redirect-handling concerns even though GitHub notes the issue does not affect declarative `<BrowserRouter>` mode by itself.
- Recommendation: Upgrade direct dependencies or their parents to patched versions that remove the vulnerable transitive packages, then rerun audit, lint, targeted auth/route tests, and extension build checks.
- Acceptance criteria: `pnpm audit --prod --json` reports no high runtime advisories and no unresolved moderate runtime advisories without an explicit accepted-risk note; targeted auth, route-guard, and extension contract checks still pass.

## Business Rule Alignment

- `Public contact intake`: `submit-contact` aligns with the public intake rule because it validates origin, Turnstile, honeypot, and rate limits before inserting `contact_submissions`. `send-website-email` conflicts because it remains a directly public mail-sending path outside that validated workflow.
- `Tracked email clicks`: the business-access rules do not yet define whether tracked links may ever leave Trusted Bums-owned domains, how invalid or expired links should behave, or whether click-through should be blocked when the delivery row does not exist. Product Ops and Trust should define that rule before the redirect hardening ships.
- `Profile bootstrap and approvals`: source aligns with the current rule by treating Clerk signup metadata as onboarding intent, not authority, and by routing approval/assignment through service-role functions plus explicit review objects. Remaining security work is direct allow/deny proof, not a new business-rule rewrite.
- `Customer targets`: current RLS still conflicts with the documented deny rule by allowing saved-target-only Bum reads. This is the clearest current RLS-to-business-rule mismatch in the repo.
- Required direct allow/deny checks before closing current access work: denied direct `send-website-email`; allowed `submit-contact`; invalid `email-track` IDs do not redirect externally; denied saved-target-only Bum target read; allowed explicitly entitled Bum target read; denied external invitation redirect; allowed approved invitation redirect.

## Time-Sensitive Threat Notes

- Supabase currently says security-definer functions should not be created in schemas exposed by the Data API, and separately advises exposed tables should have least-privilege grants plus RLS. Sources: [Supabase RLS guide](https://supabase.com/docs/guides/database/postgres/row-level-security), [Supabase secure data guide](https://supabase.com/docs/guides/database/secure-data).
- Clerk's current metadata docs still state that `unsafeMetadata` is readable and writable from the frontend and is copied from sign-up into the user object, so it must remain non-authoritative for role, company, or admin decisions. Source: [Clerk user metadata](https://clerk.com/docs/users/user-metadata).
- Chrome extension messaging guidance still says content scripts are less trustworthy than privileged extension code and that message inputs should be validated and privileged actions kept narrow. Chromium also warns against building an open proxy between content scripts and privileged fetchers. Sources: [Chrome messaging security considerations](https://developer.chrome.com/docs/extensions/develop/concepts/messaging), [Chromium content-script fetch guidance](https://www.chromium.org/Home/chromium-security/extension-content-script-fetches/).
- GitHub published `GHSA-2j2x-hqr9-3h42` on 2026-06-03 for `react-router` open redirects in affected imperative redirect flows; patched versions are `6.30.4` and `7.14.1`. Source: [GitHub advisory GHSA-2j2x-hqr9-3h42](https://github.com/advisories/GHSA-2j2x-hqr9-3h42).
