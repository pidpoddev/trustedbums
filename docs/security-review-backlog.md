## Executive Read

The highest-confidence issue is now a live authorization mismatch, not just a source-level policy concern. The deployed `extension-api-v1` function for Supabase project `vaoqvtxqvbptyxddpoju` still returns `customerTargets` to Bum sessions, while `docs/business-access-rules.md` says Bums should only see target data tied to explicit marketplace work and local source has already been tightened to hide those targets from the extension context. ### P1 - Add server-side abuse controls to the public website email trigger
- Evidence: [`src/lib/contactApi.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/contactApi.ts:116) inserts a public contact submission and invokes `send-website-email` after only a honeypot check. [`supabase/functions/send-website-email/index.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/functions/send-website-email/index.ts:14) accepts unauthenticated `POST` requests, allows `Access-Control-Allow-Origin: *`, and sends mail through Microsoft Graph with no captcha, nonce, or throttling. [`supabase/config.toml`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/config.toml:33) keeps `verify_jwt = false` for the function. Current platform guidance still makes this materially relevant: Supabase recommends putting custom server-side logic in Edge Functions and protecting exposed data/API paths with explicit access controls rather than assuming the browser path is sufficient ([Supabase secure data guidance](https://supabase.com/docs/guides/database/secure-data/)); public-edge abuse remains a common operational spam pattern.
- Why it matters: A scripted caller can turn the website contact path into a mailbox-flood or reputation-harm vector against `WEBSITE_CONTACT_NOTIFY_TO`, creating support noise, Graph usage, and possible email-deliverability impact.
- Recommendation: Keep the form public if needed, but require verifiable anti-abuse proof before the mail send path: CAPTCHA or Turnstile verification, a short-lived server nonce, and short-window IP plus email throttling with clear logging.
- Acceptance criteria: Requests without valid anti-abuse proof are rejected; repeated submits from the same IP or email hit a server-side limit; automated checks cover valid submit, invalid proof, and throttled repeat submit.

### P2 - Remove production endpoint and key fallbacks from shipped web and extension clients
- Evidence: [`src/lib/supabase.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/supabase.ts:3) still falls back to a concrete production Supabase URL and publishable key when env vars are missing. [`chrome-extension/trustedbums/src/popup.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/chrome-extension/trustedbums/src/popup.ts:1) still falls back to production extension API and Clerk sync hosts. [`chrome-extension/trustedbums/manifest.json`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/chrome-extension/trustedbums/manifest.json:22) retains broad Supabase, app, and Clerk host permissions. Current vendor guidance remains explicit that Chrome extensions need only the host permissions required for their cookie/API access ([Chrome permissions guide](https://developer.chrome.com/docs/extensions/develop/concepts/declare-permissions), [Chrome cookies API](https://developer.chrome.com/docs/extensions/reference/api/cookies)), and Clerk's current production deployment guide requires explicit production keys, sync host, host permissions, and allowed origins for Chrome extensions ([Clerk deploy Chrome extension guide](https://clerk.com/docs/deployments/deploy-chrome-extension/)).
- Why it matters: Missing or miswired env configuration can silently target production infrastructure from local, preview, or repackaged builds. In the extension, broad host permissions and cookie access amplify the blast radius if packaging inputs drift.
- Recommendation: Fail closed when required runtime or packaging env vars are absent, inject production values only in the explicit release build, and add a packaging check that rejects fallback production URLs or keys in generated artifacts.
- Acceptance criteria: Missing env vars fail build or startup clearly; no production fallback endpoints or keys remain in built artifacts; extension packaging verifies the final API host, Clerk host, and allowed origins before release.
