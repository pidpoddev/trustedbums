## Active Recommendations

### P1 - [TB-0087] Remove public EXECUTE from customer-lead duplicate helper and narrow exposed grants
- Evidence: Live SQL on 2026-06-12 confirms `public.find_customer_lead_duplicate(p_vendor_company_id uuid, p_customer_domain text)` is `SECURITY DEFINER`, owned by `postgres`, and executable by `anon` and `authenticated` in the live project. The live function definition still accepts a caller-supplied `p_vendor_company_id`, reads `customer_targets`, `reverse_opportunities`, `companies`, and `company_domains`, and returns `customer_name` plus `status` without checking `current_company_id()`, `current_user_id()`, or caller role. Current source in [supabase/migrations/20260611120000_add_customer_lead_duplicate_check.sql](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/supabase/migrations/20260611120000_add_customer_lead_duplicate_check.sql) and [supabase/migrations/20260611124500_match_customer_lead_domain_aliases.sql](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/supabase/migrations/20260611124500_match_customer_lead_domain_aliases.sql) still grants `EXECUTE` on that helper to `anon` and `authenticated`. The companion helper `public.normalize_customer_domain(text)` also remains callable by exposed roles and still has no explicit `search_path` in the live catalog.
- Why it matters: `SECURITY DEFINER` bypasses RLS on the underlying tables. That lets a public or signed-in caller probe whether an arbitrary vendor company already has a matching customer target or customer lead for a domain, which conflicts with the business rule that Bums manage only their own leads and unmatched clients must not browse lead details.
- Recommendation: Move duplicate detection behind a Portal Domain API or make the helper invoker-safe and caller-scoped by deriving the vendor company from the authenticated session instead of accepting an arbitrary company id. Revoke anonymous execute, keep any RLS-bypassing helper out of the public RPC surface, and add an explicit `search_path` to `normalize_customer_domain()` if it remains callable.
- Acceptance criteria: Direct RPC calls with an arbitrary vendor company id no longer return cross-company lead or target metadata; exposed-role `EXECUTE` is removed or the helper proves caller-scoped company binding; source or live tests cover anonymous deny, foreign-company deny, and legitimate same-company allow paths; and `normalize_customer_domain()` no longer shows a search-path warning in live review.

### P1 - [TB-0081] Revoke public EXECUTE on admin scrum audit helpers
- Evidence: Live SQL on 2026-06-12 confirms both `public.record_admin_scrum_item_audit_event()` and `public.set_admin_scrum_item_audit_fields()` are still `SECURITY DEFINER` functions in `public` and are still executable by `anon` and `authenticated`. Current source in [supabase/migrations/20260609100000_harden_admin_scrum_tracker_audit.sql](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/supabase/migrations/20260609100000_harden_admin_scrum_tracker_audit.sql) still creates both helpers in the exposed schema without revoking exposed-role `EXECUTE`.
- Why it matters: Admin-only tracker table RLS does not stop `/rest/v1/rpc` execution when the helper functions themselves remain callable by public or signed-in roles. That leaves admin-only audit logic on a broader external surface than the business rule allows.
- Recommendation: Revoke `EXECUTE` on both helpers from `PUBLIC`, `anon`, and `authenticated`, or move them to a non-exposed schema while keeping the triggers intact. Add a regression that locks the revoke-or-private-schema contract so later migrations cannot silently reopen the RPC surface.
- Acceptance criteria: Live grant review no longer shows exposed-role `EXECUTE` on either helper; `/admin/scrum` create and update flows still work through the trigger path; and source tests assert the helpers are not publicly executable.

### P1 - [TB-0085] Require fail-closed auth for claim decision sync
- Evidence: Live edge-function inventory shows `sync-claim-decision-replies` is still active as version `3` with `verify_jwt = false`, and live deployed function source still only rejects callers when `CLAIM_DECISION_SYNC_SECRET` exists. Current scheduler source in [supabase/migrations/20260609124500_add_claim_decline_reasons_and_email_decisions.sql](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/supabase/migrations/20260609124500_add_claim_decline_reasons_and_email_decisions.sql) still posts to the public function URL every five minutes and sends `x-sync-secret` as `coalesce(..., '')`, which means missing secret state degrades into an empty header rather than a hard failure. This run still did not have vault-secret visibility, so the secret-presence question remains unverified even though the fail-open code path is confirmed live.
- Why it matters: This function uses service-role access plus Microsoft Graph mailbox access to process client claim decisions. If the secret is absent, removed, or misconfigured, anonymous callers could trigger mailbox-backed claim mutation work on a public endpoint.
- Recommendation: Make the function fail closed by requiring a non-empty sync secret at startup or on every request, or move the invocation to a JWT-protected internal-only path. Mirror that fail-closed requirement into the scheduler path so cron setup also errors when the secret is missing, then add a regression test that proves missing-secret deployments are rejected.
- Acceptance criteria: Deploy or startup fails when the sync secret is missing, or the function rejects all calls without a valid internal auth mechanism; scheduled invocations prove they use non-empty auth material; and a regression test asserts the function cannot run with `verify_jwt = false` plus an empty or missing secret.

### P1 - [TB-0091] Harden extension API rate limits and negative-path contract coverage
- Evidence: The Chrome extension API remains the only current partner-style API contract. Source review of [supabase/functions/extension-api-v1/index.ts](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/supabase/functions/extension-api-v1/index.ts) still shows auth, entitlement checks, idempotency, and audit behavior, but no explicit per-user, per-token, per-origin, or equivalent abuse-control posture for `GET /context` or `POST /page-captures`. Current negative-path coverage still does not prove wrong-issuer JWT rejection, stable deny envelopes across malformed requests, or direct Data API deny behavior for `extension_page_captures`.
- Why it matters: A valid or stolen Clerk token could repeatedly create draft captures containing URLs, selected text, notes, and audit events. The partner-API contract now requires rate limits, idempotency, audit, and stable error behavior; the current extension API only satisfies part of that contract.
- Recommendation: Add rate-limit or equivalent abuse-control enforcement for extension API calls, and add contract tests for invalid destination combinations, mismatched `destinationType` and id fields, invalid or missing `sourceUrl`, invalid `captureType`, idempotent retry behavior, stable error envelopes, CORS allow/deny, unknown endpoint `404`, forged or wrong-issuer JWT rejection, cross-company denial, and direct Data API allow/deny behavior for `extension_page_captures`.
- Acceptance criteria: Extension API v1 has explicit abuse-control or rate-limit behavior; negative-path tests prove invalid requests and forged or wrong-tenant tokens fail closed; direct table writes remain disallowed outside intended paths; and hosted extension API smoke covers at least one authenticated allow path plus stable deny paths before any future native, client, or partner consumer depends on the API.

## Time-Sensitive Threat Notes

- Production dependency risk remains low from the package-audit surface in this run. `corepack pnpm audit --prod --json` returned zero production advisories.
- Current Supabase API guidance says grants determine whether `anon`, `authenticated`, or `service_role` can reach a table, view, or function over the Data API, while RLS determines which rows they can touch. That is the live rationale behind `TB-0087` and `TB-0081`. [Source](https://supabase.com/docs/guides/api/securing-your-api)
- Current Supabase Edge Function guidance says `verify_jwt = false` is for genuinely public functions or handlers with an intentional alternate auth model, not for internal-only jobs that should silently accept missing auth. That is the live rationale behind `TB-0085`. [Source](https://supabase.com/docs/guides/functions/function-configuration)
- Current Supabase platform guidance says new projects no longer expose `public` tables to the Data API automatically, while existing objects keep their current grants until reviewed. Trusted Bums predates that default shift, so `TB-0087` remains real even though the platform default is tightening. [Source](https://supabase.com/changelog)
- Current Supabase password-security guidance says leaked-password protection checks passwords against the HaveIBeenPwned corpus and should be enabled where possible. That is the live rationale behind the still-open `TB-0023` warning. [Source](https://supabase.com/docs/guides/auth/password-security)
- Current Clerk metadata guidance still says frontend code can update `unsafeMetadata`. Trusted Bums should continue treating that surface as onboarding input only, never as the authority for role, company, or admin access. [Source](https://clerk.com/docs/guides/users/extending)
- Current Cloudflare Turnstile guidance still says server-side validation is mandatory because tokens are single-use, can expire, and can be forged. Keep that contract on every public-intake path change. [Source](https://developers.cloudflare.com/turnstile/get-started/server-side-validation/)
