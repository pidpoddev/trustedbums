## Active Recommendations

### P1 - [TB-0108] Remove security definer from claim notification preview view and prove exact allow/deny behavior
- Evidence: Live security advisors on 2026-06-17 flag `public.claim_client_notification_previews` as `security_definer_view`. Live SQL in this run shows the view is reachable by both `anon` and `authenticated`, and current source in [supabase/migrations/20260616124500_add_claim_client_notification_previews.sql](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/supabase/migrations/20260616124500_add_claim_client_notification_previews.sql) creates the view in `public` without `with (security_invoker=on)` while [src/lib/portalApi.ts](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/lib/portalApi.ts) reads it directly for claim-notification previews.
- Why it matters: Views in an exposed schema run with the creator's privileges by default and can bypass the intended RLS posture on underlying claim and email tables. Here, the business rule is company-scoped preview visibility for the relevant client team and Admins, not a broadly exposed public view contract.
- Recommendation: Recreate the view with `with (security_invoker=on)` if the project Postgres version supports it, or move the read path to a private schema or server-owned API that enforces the same company and admin rule explicitly. Validate with the real Clerk token shape used in production, including signed-in sessions that may evaluate under the `anon` database role.
- Acceptance criteria: Live advisors no longer flag `claim_client_notification_previews`; source no longer leaves the view in security-definer mode; exact-head tests or live checks prove Admin allow, same-company client allow, foreign-company deny, Bum deny, and public deny; and the replacement path has a rollback-safe migration plan.

### P1 - [TB-0089] Pin Clerk issuer and JWKS resolution in remaining `verify_jwt = false` service-role Edge Functions
- Evidence: Exact-head source grep on 2026-06-17 still shows the old `issuer?.trim() || clerkFrontendApiUrl?.trim()` JWKS resolution pattern in active functions such as `client-team`, `customer-lead-duplicate-check`, `profile-bootstrap`, `admin-access-requests`, `send-admin-email`, `clerk-user-tools`, `sync-clerk-users`, `invite-bum`, `submit-feedback`, `bum-extension-download`, `dmarc-reports`, `schedule-teams-meeting`, and `sync-teams-attendees`. By contrast, `extension-api-v1` and `portal-contacts` now pin the expected Clerk issuer and reject untrusted issuers. Live Edge Function inventory in this run confirms the affected functions remain active on project `vaoqvtxqvbptyxddpoju`.
- Why it matters: When a service-role function accepts a bearer token, using the token's own untrusted `iss` to choose the JWKS endpoint can let an attacker point verification at attacker-controlled keys and mint a forged signed token for privileged flows.
- Recommendation: Extract a shared Clerk verifier for service-role edge functions that pins the expected Clerk issuer or public key material, rejects missing or foreign `iss`, and optionally checks additional request-context constraints where relevant. Apply it to every Clerk-backed `verify_jwt = false` function that still uses the fallback issuer pattern, and add regression coverage that proves wrong-issuer and forged-issuer tokens fail closed.
- Acceptance criteria: No active Clerk-backed service-role function retains the fallback issuer or JWKS pattern; shared tests prove missing-issuer and wrong-issuer tokens fail closed; and at least one live or hosted allow or deny proof exists for each function class affected: admin-only, client-admin, Bum, self-service, and public-with-bearer.

### P1 - [TB-0023] Keep leaked-password protection blocked only if the plan decision is explicit
- Evidence: Live security advisors on 2026-06-17 still report `auth_leaked_password_protection`. This shell still did not have Auth dashboard-setting visibility, so it could not freshly prove whether the control can now be enabled or remains blocked by plan or access.
- Why it matters: If any password-backed accounts remain enabled without leaked-password protection, previously compromised passwords remain eligible unless another control fully compensates.
- Recommendation: Enable Supabase Auth leaked-password protection if the current plan and dashboard settings permit it. If it is still unavailable, keep the item blocked only with an explicit accepted-risk or plan-upgrade decision in the tracker and access docs.
- Acceptance criteria: Live Auth settings show leaked-password protection enabled, or `TB-0023` records a current accepted-risk or plan decision with compensating controls and ownership.
