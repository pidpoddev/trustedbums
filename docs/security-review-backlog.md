## Active Recommendations

### P0 - Remove the direct public `send-website-email` mail-sending path
- Evidence: Fresh deployed source retrieval on 2026-06-06 confirms `send-website-email` is live with `verify_jwt: false`, and local source in [`/Users/ryan.peterson/Documents/Trusted Bums LOCAL/trustedbums/supabase/functions/send-website-email/index.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/functions/send-website-email/index.ts) still accepts unauthenticated POSTs, sets `Access-Control-Allow-Origin: *`, and calls Microsoft Graph using `MICROSOFT_CLIENT_SECRET`. A fresh direct POST to `https://vaoqvtxqvbptyxddpoju.supabase.co/functions/v1/send-website-email` with an unsupported template returned `400` without any auth or abuse gate. The hardened [`/Users/ryan.peterson/Documents/Trusted Bums LOCAL/trustedbums/supabase/functions/submit-contact/index.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/functions/submit-contact/index.ts) path still verifies origin, Turnstile, and rate limits first, and a fresh fake-token probe returned `403`.
- Why it matters: A public secret-backed mail sender can be abused for mailbox flooding, sender-reputation damage, and bypass of the documented public-intake rule that requires server-side abuse checks before mail is sent.
- Recommendation: Make `submit-contact` the only anonymous public contact path. Move notification sending fully behind `submit-contact`, or require a server-only caller proof that browsers and third parties cannot supply.
- Acceptance criteria: Direct unauthenticated calls to `send-website-email` are denied before template parsing; valid homepage contact submissions still send exactly one intended notification through the validated path; invalid Turnstile, origin, honeypot, and rate-limit cases do not send mail.

### P1 - Close the public `email-track` open redirect
- Evidence: Fresh deployed source retrieval on 2026-06-06 confirms `email-track` is live with `verify_jwt: false`. Local and deployed source in [`/Users/ryan.peterson/Documents/Trusted Bums LOCAL/trustedbums/supabase/functions/email-track/index.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/functions/email-track/index.ts) still redirects to the raw `u` query parameter after `recordEvent()`, and `recordEvent()` returns silently when the delivery id is missing or invalid. A fresh probe to `https://vaoqvtxqvbptyxddpoju.supabase.co/functions/v1/email-track/click?d=00000000-0000-0000-0000-000000000000&u=https%3A%2F%2Fexample.com%2F` returned `302 Location: https://example.com/`.
- Why it matters: Attackers can wrap arbitrary destinations in a Trusted Bums-branded Supabase URL, creating phishing, domain-reputation, and browser-security-tool blocking risk.
- Recommendation: Redirect only after a valid delivery row is found and the destination matches an allowlisted `https` scheme and host policy. Return `404` or a safe Trusted Bums landing page for invalid delivery ids, expired links, malformed URLs, and unapproved hosts.
- Acceptance criteria: Invalid delivery ids never redirect off-domain; valid tracked links redirect only to approved `https` destinations stored or validated at send time; tests cover invalid id, missing `u`, non-HTTP schemes, external disallowed host, and approved destination.

### P1 - Revoke broad `public` Data API grants and make exposure opt-in
- Evidence: Fresh read-only SQL on 2026-06-06 shows `anon` and `authenticated` still each have privileges on `52` objects in exposed `public`, and sensitive tables including `profiles`, `customer_targets`, `bum_saved_items`, `extension_page_captures`, and `contact_submissions` still carry broad grants such as `SELECT`, `INSERT`, `UPDATE`, `DELETE`, `TRUNCATE`, `REFERENCES`, and `TRIGGER` for browser-facing roles. Fresh SQL confirmed all checked `public` tables currently have RLS enabled, but current Supabase guidance still says exposed tables need minimum necessary grants and exposed functions need tightly restricted `EXECUTE`.

- Recommendation: Revoke default table and function privileges in `public` for `anon`, `authenticated`, and `PUBLIC` where not intentionally required, then regrant only object and verb combinations mapped to documented portal, extension, telemetry, and public-intake workflows.
- Acceptance criteria: Sensitive internal tables and helper routines are not broadly reachable over the Data API; remaining grants are enumerated per object and purpose; regression checks prove one allowed and one denied direct API call for every public surface that remains.

### P2 - Constrain invitation redirect URLs for Client Team and Bum invitations
- Evidence: [`/Users/ryan.peterson/Documents/Trusted Bums LOCAL/trustedbums/supabase/functions/client-team/index.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/functions/client-team/index.ts) and [`/Users/ryan.peterson/Documents/Trusted Bums LOCAL/trustedbums/supabase/functions/invite-bum/index.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/functions/invite-bum/index.ts) still pass caller-supplied `redirectUrl` values into Clerk invitations. - Acceptance criteria: Client Team and Bum invitation functions reject or normalize external redirect URLs; default redirects still land on the correct login path; tests cover approved Trusted Bums origin, missing origin, and external disallowed origin.

- Current Supabase API guidance says exposed tables need minimum necessary grants plus RLS, and functions need carefully restricted `EXECUTE`, especially for `SECURITY DEFINER` routines. [Source](https://supabase.com/docs/guides/api/securing-your-api)
- Current Supabase auth/RLS guidance says `SECURITY DEFINER` functions used for policies should not live in an exposed schema. [Source](https://supabase.com/docs/guides/auth/auth-deep-dive/auth-row-level-security)
- Current Clerk docs still show `unsafeMetadata` as frontend-writeable, so it must stay non-authoritative for role, company, and admin state. [Source](https://clerk.com/docs/users/metadata)
- Current Cloudflare Turnstile guidance says tokens are single-use, valid for five minutes, and must be verified server-side. [Source](https://developers.cloudflare.com/turnstile/get-started/server-side-validation/)
- Current Chromium and Chrome extension guidance still treats content scripts as less trustworthy and recommends routing privileged cross-origin fetches through extension-controlled contexts instead of content scripts. [Sources](https://www.chromium.org/Home/chromium-security/extension-content-script-fetches/, [Chrome network requests](https://developer.chrome.com/docs/extensions/develop/concepts/network-requests))
