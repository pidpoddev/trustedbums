# Trusted Bums Trust And Reputation Backlog

_Last updated: 2026-06-04 by Codex daily trust and reputation consultant automation._

## Executive Read

The highest-leverage trust risk still sits on the public contact-to-email path. Current source shows the hardened anonymous intake (`submit-contact`) still depends on a separately callable anonymous mail-send function (`send-website-email`) that exposes wildcard CORS and direct Microsoft Graph `sendMail`.

The next biggest gaps are still public-web trust basics and sender enforcement readiness. Current source still ships root-relative global metadata from [`index.html`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/index.html), current checked-in Apache config in [`public/.htaccess`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/public/.htaccess) still lacks explicit host canonicalization and baseline security headers, and the current session still could not run a mailbox-backed DMARC review because Microsoft Graph device-code authentication was not completed and local 1Password CLI had no configured account to retrieve the Outlook login.

## Active Recommendations

### P0 - Remove public direct mail-send exposure outside the validated contact intake
- Evidence: [`supabase/functions/submit-contact/index.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/functions/submit-contact/index.ts) now enforces origin allowlisting, Turnstile Siteverify, honeypot handling, abuse fingerprinting, and short-window rate limits before inserting `contact_submissions` and invoking `send-website-email`. But [`supabase/functions/send-website-email/index.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/functions/send-website-email/index.ts) still accepts anonymous POST requests, returns `Access-Control-Allow-Origin: *`, and calls Microsoft Graph `users/{sender}/sendMail` directly. [`supabase/config.toml`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/config.toml) still sets `verify_jwt = false` for both functions.
- Why it matters: A directly callable mail-sending endpoint on a new B2B domain is a sender-reputation and abuse-risk surface. If it gets scripted or spammed, Trusted Bums risks mailbox flooding, SmartScreen or gateway distrust, and degraded inbox placement for legitimate mail.
- Recommendation: Make `submit-contact` the only anonymous public intake path. Require `send-website-email` to reject direct anonymous callers with an internal shared secret or equivalent trusted-caller proof, then verify one allowed homepage submission and one denied direct invocation in production.
- Acceptance criteria: Anonymous direct calls to `send-website-email` return deny status; valid homepage contact submissions still succeed through `submit-contact`; invalid Turnstile, origin, or rate-limit cases do not send mail; and production logs show one allowed contact flow plus one denied direct call after deployment.

### P1 - Run a current mailbox-backed DMARC review and move from monitoring to enforcement
- Evidence: Current source still includes the admin-gated DMARC parser in [`supabase/functions/dmarc-reports/index.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/functions/dmarc-reports/index.ts), but this run had no authenticated admin bearer token to execute it against `bums@trustedbums.com`. A direct Microsoft Graph delegated path was also attempted on 2026-06-04 after installing `Microsoft.Graph.Authentication` 2.37.0, but two device-code sign-ins timed out before authentication completed. Local 1Password CLI 2.34.0 was installed, but `op account list` returned no configured accounts, so it could not retrieve the Outlook login. Current Google guidance still says Gmail enforcement has ramped up on non-compliant traffic and BIMI still requires DMARC `p=quarantine` or `p=reject`. Sources: [Email sender guidelines FAQ](https://support.google.com/a/answer/14229414?hl=en), [Set up BIMI](https://support.google.com/a/answer/10911320?hl=en).
- Why it matters: `p=none` provides reporting but no receiver-side enforcement against spoofing. That weakens impersonation resistance, keeps BIMI unavailable, and leaves delivery resilience more exposed as receiver enforcement tightens.
- Recommendation: Run the deployed `dmarc-reports` review against the approved shared mailbox, enumerate every legitimate `trustedbums.com` sender, remediate any SPF or DKIM alignment drift, and advance DMARC to `p=quarantine` or `p=reject` with an explicit rollback owner.
- Acceptance criteria: Recent DMARC aggregate reports from `bums@trustedbums.com` are reviewed; all legitimate senders align through SPF or DKIM for the From domain; unexplained failures are remediated or explicitly accepted; and `_dmarc.trustedbums.com` moves to `p=quarantine` or `p=reject`.

### P1 - Enforce one public host and ship absolute route-specific metadata
- Evidence: [`public/.htaccess`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/public/.htaccess) still contains SPA rewrites only and does not canonicalize `www` to a preferred host. [`index.html`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/index.html) still publishes `<link rel="canonical" href="/" />`, `og:url` as `/`, and root-relative social image URLs. Public routes still include `/`, `/login`, `/privacy-policy`, and `/legal/:slug` in [`src/App.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/App.tsx), while [`src/pages/PrivacyPolicy.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/PrivacyPolicy.tsx) and [`src/pages/LegalDocumentPage.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/LegalDocumentPage.tsx) compute page titles and descriptions in component state but do not write route-specific head tags.
- Why it matters: Canonical drift and generic root-relative metadata weaken crawler consolidation, make shared links look unfinished to buyers and reviewers, and reduce the chance that legal and trust pages render with the right context in search and social previews.
- Recommendation: Redirect all non-preferred public hosts to one canonical host and add a route-aware metadata layer that emits absolute canonical, Open Graph, and Twitter tags for each public route.
- Acceptance criteria: `www` redirects to the preferred host; public pages emit absolute canonical URLs on that host; Open Graph and Twitter tags use absolute URLs; and privacy/legal pages publish route-specific title and description metadata in the document head.

### P1 - Add production HSTS and explicit baseline security headers
- Evidence: Checked-in Apache config in [`public/.htaccess`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/public/.htaccess) still sets content types and SPA rewrites only. It does not set `Strict-Transport-Security`, `Content-Security-Policy`, `Referrer-Policy`, `Permissions-Policy`, or a clickjacking control. Historical live HTTP checks on 2026-05-31 found those headers missing on the public site; today’s runner could not revalidate over HTTP because `curl` timed out during DNS resolution. Current OWASP Secure Headers guidance still treats these as baseline hardening controls. Source: [OWASP Secure Headers Project](https://owasp.org/www-project-secure-headers/).
- Why it matters: Missing explicit browser-security headers weaken enterprise trust review posture, leave HTTPS downgrade and framing behavior implicit, and make a public B2B site look minimally hardened.
- Recommendation: Add HSTS and explicit baseline headers at the host layer, then verify the live response set from the preferred production host after deployment.
- Acceptance criteria: HTTPS responses include `Strict-Transport-Security` with at least one year of `max-age` plus `includeSubDomains`; framing, referrer, and permissions policies are explicit; and any CSP is validated against the deployed app before rollout.

- Google Search Console still treats Security Issues as browser- and search-visible trust problems. Google says harmful behavior can produce warning labels in search results or browser interstitials, and the Security Issues report is the source of truth for verified owners. Source: [Security issues report](https://support.google.com/webmasters/answer/9044101/security-issues-report).
- Google’s current dangerous-site guidance still points non-owners to the Safe Browsing transparency report and tells owners to rely on Search Console alerts, Security Issues, and Manual Actions. Source: [Why is my site labeled as dangerous in Google Search?](https://support.google.com/webmasters/answer/6347750?hl=en).
- Gmail’s current sender guidance says enforcement on non-compliant bulk traffic ramped up starting in November 2025, and failures around SPF, DKIM, DMARC, alignment, PTR, and TLS can now cause temporary or permanent delivery failures. Sources: [Email sender guidelines FAQ](https://support.google.com/a/answer/14229414?hl=en), [Email sender guidelines](https://support.google.com/a/answer/81126?hl=en).
- Google’s current BIMI guidance still requires DMARC `p=quarantine` or `p=reject` with `pct=100`; DMARC `p=none` does not qualify. Source: [Set up BIMI](https://support.google.com/a/answer/10911320?hl=en).
- Microsoft’s current SmartScreen owner guidance still routes false-positive website remediation through the Edge warning-page report flow and WDSI submission path. Source: [Unexpected Block or Warning on Customer Website in Microsoft Edge](https://learn.microsoft.com/en-us/troubleshoot/microsoft-edge/development/unexpected-block-warning).
- OWASP’s current Secure Headers guidance still treats HSTS, CSP, referrer policy, and permissions policy as baseline response-header controls for public web properties. Source: [OWASP Secure Headers Project](https://owasp.org/www-project-secure-headers/).
