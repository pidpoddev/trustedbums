# Trusted Bums Trust And Reputation Backlog

_Last updated: 2026-05-29 by Codex trust and reputation setup._

## Executive Read

The highest-confidence trust risk is now enterprise websecurity blocking, followed by the public contact email path. On 2026-05-29, the user's managed device could not reach `trustedbums.com` because websecurity filters blocked or sinkholed access, even though Google Search Console and Google Safe Browsing were clean. Local system DNS uses managed resolvers at `100.64.0.x`, while direct Google DNS resolves `trustedbums.com` normally, which points to security-filter classification rather than public DNS failure. Local source also still shows `send-website-email` as a public mail-sending edge function with `verify_jwt = false`, wildcard CORS, and no server-side captcha, nonce, or throttling.

The main external-evidence gap has narrowed but is not closed. On 2026-05-29, Google Search Console ownership for `trustedbums.com` was verified by DNS TXT record, Manual Actions showed no issues, Security Issues showed no issues, and Google Safe Browsing Site Status showed "No unsafe content found" with site info last updated on May 16, 2026. A local XML sitemap and robots sitemap directive now exist, and the sitemap was submitted in Search Console, but Google reports the live `/sitemap.xml` response is HTML. That means production is still serving the SPA fallback for the sitemap path until the new static sitemap is deployed. The product side now has a deployed DMARC reader with aggregate XML parsing, a documented Google Search/Safe Browsing setup path, Microsoft Graph `Mail.Read` admin consent, and an Exchange application access policy scoped to `bums@trustedbums.com`. The remaining evidence gaps are live sitemap deployment, live fetch behavior from a non-filtered network, ongoing Search Console data after Google finishes processing, and an end-to-end DMARC metadata/XML fetch through the deployed function.

## Active Recommendations

### P0 - Identify and remediate enterprise websecurity blocking
- Evidence: On 2026-05-29, the user's managed device could not reach `trustedbums.com` because websecurity filters blocked or sinkholed access. `dig trustedbums.com` through the system resolver timed out, while `dig trustedbums.com @8.8.8.8` returned `208.97.186.234`. `scutil --dns` showed managed supplemental DNS resolvers at `100.64.0.1`, `100.64.0.2`, and `100.64.0.3`. Google Search Console ownership was verified, Search Console Security Issues and Manual Actions showed no issues, and Google Safe Browsing showed "No unsafe content found."
- Why it matters: If a corporate buyer, partner, or finance user cannot reach the site from a managed device, the product fails before conversion or onboarding begins. Google-clean status is helpful but not enough for enterprise trust.
- Recommendation: Determine the exact blocking vendor, category, and reason from the websecurity block page or enterprise security logs. Submit recategorization requests to the blocking vendor and major reputation systems, and request a local allow/monitor exception for `trustedbums.com` while vendor databases update.
- Acceptance criteria: The blocking product and category are documented; `trustedbums.com` is recategorized or explicitly allowed; a managed-device retest can resolve and load `https://trustedbums.com`; and the Trust Agent records ongoing checks for Google Safe Browsing, Microsoft SmartScreen/Defender if available, Cisco Talos/Umbrella, Palo Alto PAN-DB, FortiGuard, Zscaler, and any customer-relevant websecurity system.

### P0 - Add abuse controls to the public contact email path before volume grows
- Evidence: [`src/lib/contactApi.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/contactApi.ts) inserts anonymous contact submissions and invokes `send-website-email` after only a honeypot check. [`supabase/functions/send-website-email/index.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/functions/send-website-email/index.ts) accepts unauthenticated `POST`, allows `Access-Control-Allow-Origin: *`, and sends Microsoft Graph email. [`supabase/config.toml`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/config.toml) keeps `verify_jwt = false` for the function. Cloudflare’s current Turnstile guidance requires server-side token validation, and Google’s sender guidance still treats authenticated, well-controlled mail flows as basic trust posture. Sources: [Cloudflare Turnstile server-side validation](https://developers.cloudflare.com/turnstile/get-started/server-side-validation/), [Google email sender guidelines](https://support.google.com/a/answer/81126?hl=en-na).
- Why it matters: A new B2B domain with a public mail-trigger endpoint can lose inbox trust and operator confidence quickly if scripted abuse starts before the rest of the site matures.
- Recommendation: Protect the public intake flow with server-side anti-abuse proof, short-window rate limits, and mailbox-safe fallback behavior before treating the contact path as production-hardened.
- Acceptance criteria: Invalid or repeated abusive submits are rejected without sending notification mail, blocked attempts are distinguishable in logs, and QA can prove one allowed submit plus at least one denied abuse case.

### P1 - Deploy the static sitemap so Google stops reading SPA HTML
- Evidence: `public/sitemap.xml` now exists locally and `public/robots.txt` includes `Sitemap: https://trustedbums.com/sitemap.xml`. `xmllint --noout public/sitemap.xml` passed and `pnpm exec vite build --base=/` passed. - Recommendation: Deploy the current `public/sitemap.xml` and `public/robots.txt` to production through the DreamHost GitHub Actions deployment, then resubmit or wait for Search Console to reread the sitemap.
- Acceptance criteria: `https://trustedbums.com/sitemap.xml` returns XML, Search Console sitemap status is successful, discovered pages are nonzero or explainably pending, and `https://trustedbums.com/robots.txt` advertises the sitemap.
