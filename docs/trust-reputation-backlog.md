### P1 - Add baseline production security headers, including HSTS
- Evidence: [public/.htaccess](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/public/.htaccess) still does not set `Strict-Transport-Security`, `Content-Security-Policy`, `Referrer-Policy`, `Permissions-Policy`, or clickjacking controls. [src/pages/Index.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/Index.tsx) loads Cloudflare Turnstile, so CSP must explicitly allow required third-party origins rather than ship as a placeholder. Current OWASP guidance still treats HSTS, CSP, `X-Frame-Options`, `Referrer-Policy`, and `Permissions-Policy` as active browser-security headers. Sources: [OWASP Secure Headers Project](https://owasp.org/www-project-secure-headers/index.html), [Unexpected SmartScreen block or warning on website](https://learn.microsoft.com/en-us/troubleshoot/microsoft-edge/development/unexpected-block-warning).
- Why it matters: Missing explicit browser-security headers weaken enterprise trust review posture and leave browser trust signals implicit instead of deliberate.
- Recommendation: Add explicit host-layer HSTS and baseline browser-security headers, with a CSP that intentionally covers Clerk, Supabase, and Turnstile dependencies before rollout.
- Acceptance criteria: HTTPS responses include HSTS with at least one year of `max-age` and `includeSubDomains`; framing, referrer, and permissions policies are explicit; CSP is present or staged in report-only mode with required third-party origins accounted for; and live production responses are rechecked after deploy.

### P2 - Remove or quarantine the public pre-store extension zip until permissions are narrowed
- Evidence: [public/downloads/trustedbums-extension.zip](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/public/downloads/trustedbums-extension.zip) is still publicly present. [chrome-extension/trustedbums/manifest.json](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/chrome-extension/trustedbums/manifest.json) still requests `cookies`, `storage`, and host permissions for LinkedIn, `*.supabase.co`, `api.trustedbums.com`, `trustedbums.com`, and Clerk hosts. Current Chrome guidance still says extensions should request only permissions relevant to the main function and use optional/runtime permissions where possible. Sources: [Permission warning guidelines](https://developer.chrome.com/docs/extensions/develop/concepts/permission-warnings), [Permissions](https://developer.chrome.com/docs/extensions/reference/permissions-list).
- Why it matters: A publicly downloadable pre-store extension with broad permissions creates reviewer friction, install distrust, and a download-reputation surface that can spill back onto the main domain.
- Recommendation: Keep the zip off public discovery paths until the distribution plan is final, then reduce permissions to the narrowest production set and lock extension API CORS to the final extension origin.
- Acceptance criteria: The public site no longer exposes the pre-store zip except through an intentional controlled path, extension permissions are reduced to the smallest shipped set, and extension API CORS is restricted to the final production extension origin.
