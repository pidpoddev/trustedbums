## Current Standards And Time-Sensitive Notes

- Google Search Console still treats security issues and manual-action cleanup as review-gated trust work: site owners are told to check Security Issues, audit third-party scripts or elements, fix affected pages, and then use Request Review or reconsideration flows. Source: [Manual actions report](https://support.google.com/webmasters/answer/9044175?hl=en).
- Google Search Central still says sitemaps can be submitted in Search Console, through the Search Console API, or discovered through a `Sitemap:` line in `robots.txt`; Search Console also requires owner permission to submit through the Sitemaps report. Sources: [Build and submit a sitemap](https://developers.google.com/search/docs/crawling-indexing/sitemaps/build-sitemap), [Sitemaps report](https://support.google.com/webmasters/answer/7451001?hl=en).
- Google's current link guidance says crawlable links help Google discover pages, but its spam policies still treat manipulative link practices such as buying links, link exchanges, and large-scale low-quality link building as spam risks. Sources: [Link best practices](https://developers.google.com/search/docs/crawling-indexing/links-crawlable), [Spam policies](https://developers.google.com/search/docs/essentials/spam-policies).
- Bing still supports direct URL and sitemap submission through Webmaster Tools, and IndexNow still supports bulk URL notification after the site hosts a verification key file. DuckDuckGo says most traditional search-result removal should go through Bing first because most traditional links currently come from Bing, so Bing cleanup can affect downstream visibility beyond Bing itself. Sources: [Bing URL Submission](https://www.bing.com/webmasters/help/url-submission-62f2860b), [IndexNow setup](https://www.bing.com/indexnow/getstarted), [DuckDuckGo traditional result removals](https://duckduckgo.com/duckduckgo-help-pages/results/can-a-result-be-removed).
- Chrome’s current unsafe-site guidance still frames these warnings around phishing, social engineering, malware, and unwanted software, and it still routes site owners to fix the issue and request review rather than bypass the warning. Source: [Manage warnings about unsafe sites](https://support.google.com/chrome/answer/99020?hl=en).
- Google’s sender guidance still says DMARC tells receiving servers what to do with mail that fails SPF or DKIM and recommends DMARC reports so you can identify senders impersonating your domain. Source: [Email sender guidelines](https://support.google.com/mail/answer/81126?hl=en).
- Google’s current DMARC and BIMI guidance still expects strong enforcement when BIMI is in play: DMARC policies for BIMI must use `p=quarantine` or `p=reject`, and BIMI requires `pct=100`. Sources: [Set up DMARC](https://knowledge.workspace.google.com/admin/security/set-up-dmarc), [Set up BIMI](https://knowledge.workspace.google.com/admin/security/set-up-bimi).
- Microsoft’s current SmartScreen guidance still lists URL reputation, page content, file behavior, TLS security, user feedback, and dynamic behavior as inputs to site blocking decisions. Source: [Unexpected SmartScreen block or warning on website](https://learn.microsoft.com/en-us/troubleshoot/microsoft-edge/development/unexpected-block-warning).
- Chrome’s current Web Store guidance still says review times vary and can lengthen for dangerous permission requests, broad host permissions, or significant code changes, and it now warns that elevated submission volume has extended review times since April 2026. Source: [Chrome Web Store review process](https://developer.chrome.com/docs/webstore/review-process).
- OWASP Secure Headers still treats HSTS, CSP, framing protection, referrer policy, and permissions policy as active browser trust controls worth keeping in deployed checks. Source: [OWASP Secure Headers Project](https://owasp.org/www-project-secure-headers/index.html).
- Supabase’s current changelog review did not surface a new June 2026 trust-specific breaking change for the current contact, tracking, or mailbox edge functions. The most relevant recent platform-side note remains that newly created tables are no longer auto-exposed to the Data API by default, which matters for future mailbox or telemetry schema additions rather than today’s public trust finding. Source: [Supabase changelog](https://supabase.com/changelog).
