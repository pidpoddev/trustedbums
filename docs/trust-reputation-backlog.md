# Trusted Bums Trust And Reputation Backlog

_Last updated: 2026-05-31 by Codex daily trust and reputation consultant automation._

## Executive Read

The highest-leverage active risk is still the public contact-to-email path. This run confirmed that the hardened `submit-contact` function is deployed and already returning `403` for some rejected submissions in live edge-function logs, but the separate `send-website-email` function is still deployed as a public unauthenticated endpoint with permissive CORS and direct Microsoft Graph send capability. That leaves a live sender-reputation and abuse surface even after the intake hardening work.

The other active trust gaps remain browser- and search-facing rather than pure code defects: `https://www.trustedbums.com` still serves `200` instead of redirecting to the preferred host, the public shell still emits root-relative canonical/Open Graph/Twitter metadata instead of absolute route-specific tags, DMARC is still monitor-only at `p=none`, and production responses still omit HSTS and other baseline security headers.

## Active Recommendations

### P0 - Remove public direct mail-send exposure outside the validated contact intake
- Evidence: Live Supabase function inventory on 2026-05-31 shows `submit-contact` version 1 and `send-website-email` version 1 are both deployed with `verify_jwt: false`. The deployed `submit-contact` source enforces Turnstile validation, origin allowlisting, honeypot handling, abuse fingerprinting, and short-window rate limiting before invoking `send-website-email`. The deployed `send-website-email` source still exposes `Access-Control-Allow-Origin: *`, accepts unauthenticated POST requests, and can call Microsoft Graph `sendMail` directly. Live edge-function logs from 2026-05-31 show recent `POST | 403` events for `submit-contact`, which proves some invalid submissions are now being blocked, but this run did not have a safe execution path to prove that direct public calls to `send-website-email` are blocked upstream.
- Why it matters: A public mail-sending endpoint on a young B2B domain can be abused for mailbox flooding, sender-reputation damage, SmartScreen or gateway distrust, and operational noise at `bums@trustedbums.com`.
- Recommendation: Make `submit-contact` the only public intake path, restrict `send-website-email` to internal trusted callers only, and revalidate production so no anonymous caller can trigger mail or bypass the intake controls.
- Acceptance criteria: Anonymous direct invocation of `send-website-email` is denied; anonymous direct inserts into `contact_submissions` are denied in production; a valid homepage submission still succeeds through `submit-contact`; and logs show both an allowed case and a denied abuse case after the change.

### P1 - Move DMARC from monitor-only to enforcement after a current mailbox-backed review
- Evidence: DNS on 2026-05-31 returns SPF `v=spf1 include:spf.protection.outlook.com -all`, Microsoft 365 DKIM selector CNAMEs for `selector1` and `selector2`, and `_dmarc.trustedbums.com` = `v=DMARC1; p=none; rua=mailto:bums@trustedbums.com; pct=100;`. Live Supabase inventory confirms `dmarc-reports` version 2 is deployed. Its deployed source is admin-gated and designed to read only mailbox metadata plus DMARC aggregate attachments from `bums@trustedbums.com`. This run could not execute a mailbox review because the session lacked an authenticated admin invocation path for that function, so there is still no fresh report-backed view of aligned senders, failures, or spoofing sources.
- Why it matters: `p=none` provides reporting but no receiver-side enforcement against impersonation. That weakens email trust and leaves BIMI unavailable until enforcement is in place.
- Recommendation: Run a current DMARC aggregate review from `bums@trustedbums.com`, enumerate every legitimate sender using the `trustedbums.com` From domain, remediate drift, and then advance the policy beyond `none` with a rollback threshold and named owner.
- Acceptance criteria: Recent aggregate reports are reviewed from the approved shared mailbox; legitimate senders align through SPF or DKIM for the From domain; unexplained failures are remediated or explicitly accepted; and DMARC moves to `p=quarantine` or `p=reject`.

### P1 - Enforce one public host and ship absolute route-specific metadata
- Evidence: `curl -I -L` on 2026-05-31 returned `HTTP/2 200` for both `https://trustedbums.com` and `https://www.trustedbums.com` instead of redirecting `www` to the preferred host. The public shell in [`index.html`](</Users/ryan.peterson/Documents/Trusted Bums LOCAL/trustedbums/index.html>) still sets `<link rel="canonical" href="/" />`, `og:url` as `/`, and social image URLs as root-relative paths. Public routes for `/`, `/login`, `/privacy-policy`, and `/legal/:slug` are present in [`src/App.tsx`](</Users/ryan.peterson/Documents/Trusted Bums LOCAL/trustedbums/src/App.tsx>), and the privacy/legal pages already have route-specific content but no route-specific head metadata.
- Why it matters: Duplicate HTTPS hosts and generic relative metadata weaken crawl consolidation, sharing previews, and buyer confidence on the exact pages used for legitimacy checks.
- Recommendation: Redirect every non-preferred public host to the preferred host and add a metadata layer that emits absolute canonical, Open Graph, and Twitter tags per public route.
- Acceptance criteria: `https://www.trustedbums.com` returns a redirect to the preferred host; public pages emit absolute canonical URLs on that host; Open Graph and Twitter tags use absolute URLs; and privacy/legal pages publish route-specific title and description metadata.

### P1 - Add HSTS and explicit baseline security headers on production responses
- Evidence: `curl -I -L` on 2026-05-31 for the homepage, `robots.txt`, and `sitemap.xml` showed Apache plus cache headers, but no `Strict-Transport-Security`, `Content-Security-Policy`, `X-Frame-Options`, `Referrer-Policy`, or `Permissions-Policy`. Repo inspection in this run found no checked-in header or redirect config beyond `public/robots.txt` and `public/sitemap.xml`.
- Why it matters: Missing baseline headers reduce browser hardening signals, leave downgrade/framing/privacy controls implicit, and make enterprise trust reviews harder because the site appears minimally hardened.
- Recommendation: Add host-level header configuration for HSTS and core secure headers, then verify the exact live response set from the preferred production host.
- Acceptance criteria: HTTPS responses include `Strict-Transport-Security` with at least one year of `max-age` plus `includeSubDomains`; framing, referrer, and permissions policies are explicit; and any CSP is verified against the deployed app instead of added blindly.

## Current Standards And Time-Sensitive Notes

- Cloudflare’s current Turnstile guidance still requires server-side Siteverify validation. Widget rendering alone is not sufficient. Source: [Cloudflare Turnstile server-side validation](https://developers.cloudflare.com/turnstile/get-started/server-side-validation/).
- Google’s current email sender guidance says senders should use SPF or DKIM, recommends SPF, DKIM, and DMARC for domains, and requires alignment for DMARC-based authentication outcomes. Sources: [Google Email sender guidelines](https://support.google.com/a/answer/81126?hl=en-EN), [Google Email sender guidelines FAQ](https://support.google.com/a/answer/14229414?hl=en-EN).
- Google’s current Search Console guidance says sites with security issues can show warning labels in search or browser interstitials, and manual actions can suppress some or all site visibility in search results. Sources: [Security issues report](https://support.google.com/webmasters/answer/9044101?hl=en), [Manual actions report](https://support.google.com/webmasters/answer/9044175?hl=en).
- Microsoft’s current Edge guidance says SmartScreen can block or warn on websites it determines are potentially unsafe, and the owner remediation path runs through Microsoft feedback/submission workflows. Sources: [Microsoft Defender SmartScreen overview](https://www.microsoft.com/en-us/edge/features/microsoft-defender-smartscreen), [Unexpected Block or Warning on Customer Website in Microsoft Edge](https://learn.microsoft.com/en-us/troubleshoot/microsoft-edge/development/unexpected-block-warning).
- Current browser-hardening guidance still treats HSTS and explicit secure headers as baseline practice. MDN’s current HSTS reference says preload requires `includeSubDomains` and at least one year of `max-age`. Sources: [MDN HSTS](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Strict-Transport-Security), [OWASP Secure Headers Project](https://owasp.org/www-project-secure-headers/).
- Google’s current BIMI guidance says DMARC must be at enforcement to support BIMI, so Trusted Bums should not prioritize BIMI while `_dmarc` remains `p=none`. Source: [Google Set up BIMI](https://support.google.com/a/answer/10911320?hl=en).
