# Trusted Bums Trust And Reputation Backlog

_Last updated: 2026-06-06 by Codex daily trust and reputation consultant automation._

## Executive Read

The clearest active trust risk is still the public contact-to-email path. Current source shows the hardened anonymous intake function now does validation, Turnstile verification, and rate limiting, but the downstream `send-website-email` function remains directly callable without JWT verification, returns wildcard CORS, and can send mail through Microsoft Graph on its own.

No material trust-surface code changes landed since the 2026-06-05 audit; recent commits were GTM-document focused, and current source still ships one global root-relative metadata block from [`index.html`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/index.html), checked-in Apache config in [`public/.htaccess`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/public/.htaccess) still lacks explicit canonical-host and baseline security-header controls, and this run still could not perform a live DMARC/dashboard review because no authenticated admin path to the deployed mailbox-review function or webmaster/reputation dashboards was available in-session.

## Active Recommendations

### P0 - Remove direct anonymous access to `send-website-email`
- Evidence: [`supabase/functions/submit-contact/index.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/functions/submit-contact/index.ts) now allowlists origins, verifies Cloudflare Turnstile server-side, rate-limits recent submissions, fingerprints abuse, and writes `contact_submissions` before invoking `send-website-email`. But [`supabase/functions/send-website-email/index.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/functions/send-website-email/index.ts) still accepts anonymous POST requests, returns `Access-Control-Allow-Origin: *`, and calls Microsoft Graph `users/{sender}/sendMail` directly. [`supabase/config.toml`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/config.toml) still sets `verify_jwt = false` for both functions.
- Why it matters: A separately callable mail-sending function on a young B2B domain is a sender-reputation and abuse-risk surface. If scripted, it can damage mailbox trust, trigger gateway distrust, and reduce inbox placement for legitimate Trusted Bums mail.
- Recommendation: Keep `submit-contact` as the only anonymous public intake. Require `send-website-email` to reject direct anonymous callers with a trusted-caller secret or equivalent server-to-server proof, then verify one allowed homepage submission and one denied direct invocation in production.
- Acceptance criteria: Direct anonymous calls to `send-website-email` fail; valid homepage contact submissions still deliver through `submit-contact`; invalid Turnstile, origin, or rate-limit cases do not send mail; and post-deploy logs show one allowed contact flow plus one denied direct call.

### P1 - Run a current mailbox-backed DMARC review and prepare DMARC enforcement
- Evidence: [`supabase/functions/dmarc-reports/index.ts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/supabase/functions/dmarc-reports/index.ts) exists and is designed to parse DMARC aggregate attachments from `bums@trustedbums.com` for authenticated admins. [`docs/shared-mailbox-operations.md`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/docs/shared-mailbox-operations.md) and [`docs/dmarc-report-review.md`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/docs/dmarc-report-review.md) say Microsoft Graph `Mail.Read` app consent and Exchange mailbox scoping were previously completed for `bums@trustedbums.com`, but this run still had no authenticated admin bearer token or dashboard path to execute the review. Current Google guidance says Gmail enforcement on non-compliant traffic ramped up starting in November 2025, and BIMI still requires DMARC `p=quarantine` or `p=reject` with `pct=100`. Sources: [Email sender guidelines FAQ](https://support.google.com/a/answer/14229414?hl=en), [Email sender guidelines](https://support.google.com/a/answer/81126?hl=en), [Set up BIMI](https://support.google.com/a/answer/10911320?hl=en).
- Why it matters: `p=none` reports spoofing but does not ask receivers to quarantine or reject it. That weakens impersonation resistance, keeps BIMI unavailable, and leaves Trusted Bums more exposed as receiver enforcement keeps tightening.
- Recommendation: Execute the deployed DMARC review against `bums@trustedbums.com`, inventory every legitimate sender using `trustedbums.com`, remediate alignment drift, and plan a move to `p=quarantine` or `p=reject` with a named rollback owner.
- Acceptance criteria: Recent aggregate reports are reviewed; all legitimate senders align through SPF or DKIM for the From domain; unexplained failures are remediated or explicitly accepted; and `_dmarc.trustedbums.com` advances from `p=none` to `p=quarantine` or `p=reject`.

### P1 - Canonicalize the public host and ship route-aware absolute metadata
- Evidence: [`public/.htaccess`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/public/.htaccess) still only serves crawler files and SPA rewrites; it does not enforce one public hostname. [`index.html`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/index.html) still emits `<link rel="canonical" href="/" />`, `og:url` as `/`, and root-relative social image URLs. Public routes in [`src/App.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/App.tsx) include `/`, `/login`, `/privacy-policy`, and `/legal/:slug`, while [`src/pages/PrivacyPolicy.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/PrivacyPolicy.tsx) and [`src/pages/LegalDocumentPage.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/LegalDocumentPage.tsx) compute page titles/descriptions in component state but do not write route-specific head tags.
- Why it matters: Canonical drift and generic metadata weaken crawler consolidation, make trust/legal pages look unfinished in previews, and reduce confidence during buyer, reviewer, and reputation-scanner inspection.
- Recommendation: Redirect every non-preferred public host to one canonical host and add a route-aware metadata layer that emits absolute canonical, Open Graph, and Twitter tags for each public route.
- Acceptance criteria: `www` and other non-preferred hosts redirect to the canonical host; public pages emit absolute canonical URLs; Open Graph and Twitter tags use absolute URLs; and privacy/legal pages publish route-specific head metadata.

### P1 - Add baseline production security headers, including HSTS
- Evidence: Checked-in Apache config in [`public/.htaccess`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/public/.htaccess) still does not set `Strict-Transport-Security`, `Content-Security-Policy`, `Referrer-Policy`, `Permissions-Policy`, or clickjacking controls. Current source also loads third-party scripts such as Cloudflare Turnstile from [`src/pages/Index.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/Index.tsx), so any CSP rollout needs an explicit allowlist rather than a placeholder header. Current OWASP guidance still treats HSTS, CSP, `X-Frame-Options`, `Referrer-Policy`, and `Permissions-Policy` as active browser security headers. Source: [OWASP Secure Headers Project](https://owasp.org/www-project-secure-headers/index.html).
- Why it matters: Missing explicit browser-security headers weaken enterprise trust review posture, leave downgrade/framing behavior implicit, and match criteria Microsoft now calls out as part of SmartScreen website trust evaluation. Source: [Unexpected SmartScreen block or warning on website](https://learn.microsoft.com/en-us/troubleshoot/microsoft-edge/development/unexpected-block-warning).
- Recommendation: Add explicit host-layer HSTS and baseline browser-security headers, with a CSP that intentionally covers Clerk, Supabase, and Turnstile dependencies before rollout.
- Acceptance criteria: HTTPS responses include HSTS with at least one year of `max-age` and `includeSubDomains`; framing, referrer, and permissions policies are explicit; CSP is present or staged in report-only mode with required third-party origins accounted for; and live production responses are rechecked after deploy.

### P2 - Remove or quarantine the public pre-store extension zip until permissions are narrowed
- Evidence: [`public/downloads/trustedbums-extension.zip`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/public/downloads/trustedbums-extension.zip) is publicly present. [`chrome-extension/trustedbums/manifest.json`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/chrome-extension/trustedbums/manifest.json) still requests `cookies` plus host permissions for LinkedIn, `*.supabase.co`, `api.trustedbums.com`, `trustedbums.com`, Clerk, and sync hosts. [`docs/chrome-extension.md`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/docs/chrome-extension.md) still says to narrow `extension-api-v1` CORS to the final extension origin before public release and describes the current package as a pre-store zip. Chrome’s permission-warning guidance still says extensions should request only permissions needed for core functionality. Sources: [Permission warning guidelines](https://developer.chrome.com/docs/extensions/develop/concepts/permission-warnings?hl=en), [Permissions](https://developer.chrome.com/docs/extensions/reference/permissions-list?hl=en). Google Search Console also treats harmful or questionable downloads as a security-issues surface. Source: [Security issues report](https://support.google.com/webmasters/answer/9044101).
- Why it matters: Publicly exposing a pre-store browser extension package with broad permissions can weaken extension-install trust, create reviewer friction, and increase the chance of download scrutiny from browsers or security tools.
- Recommendation: Keep the zip off public marketing/discovery paths until the final distribution plan is set, then reduce permissions to the narrowest production set and lock API CORS to the final extension origin.
- Acceptance criteria: The public site no longer exposes the pre-store zip except through an intentional controlled path, extension permissions are reduced to the smallest set the shipped workflow needs, and extension API CORS is restricted to the final production extension origin.
