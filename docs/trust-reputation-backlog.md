## Current Standards And Time-Sensitive Notes

- Google Search Console still says indexing triage should check Manual Actions and Security Issues before assuming a crawl-only problem. Sources: [Manual actions report](https://support.google.com/webmasters/answer/9044175?hl=en), [URL Inspection tool](https://support.google.com/webmasters/answer/9012289?hl=en).
- Google Search Central still says sitemaps can be surfaced through Search Console, the Search Console API, or a `Sitemap:` line in `robots.txt`, and Google still treats crawlable internal links as discovery signals while treating manipulative link schemes as spam-policy risk. Sources: [Build and submit a sitemap](https://developers.google.com/search/docs/crawling-indexing/sitemaps/build-sitemap), [Link best practices](https://developers.google.com/search/docs/crawling-indexing/links-crawlable), [Google Search Essentials](https://developers.google.com/search/docs/essentials).
- Gmail sender guidance still expects DMARC on the sending domain and alignment through SPF or DKIM on the visible `From:` domain. Source: [Email sender guidelines](https://support.google.com/mail/answer/81126?hl=en).
- Bing still recommends IndexNow, direct URL submission, crawlable internal links, and healthy sitemap submission as the supported discovery path. Sources: [URL Submission](https://www.bing.com/webmasters/help/url-submission-62f2860b), [Bing Webmaster Guidelines](https://www.bing.com/webmasters/help/webmaster-guidelines-30fba23a).
- Microsoft’s current SmartScreen guidance still ties warnings to site reputation, phishing or malware signals, content, downloads, TLS posture, and user feedback. Microsoft’s public SmartScreen FAQ still points site owners to the built-in feedback path for false warnings. Sources: [Microsoft Defender SmartScreen overview](https://learn.microsoft.com/en-us/windows/security/operating-system-security/virus-and-threat-protection/microsoft-defender-smartscreen/), [Microsoft Defender SmartScreen FAQ](https://feedback.smartscreen.microsoft.com/smartscreenfaq.aspx).
- Supabase’s current Edge Function docs still say `verify_jwt` is on by default, API keys belong in the `apikey` header rather than the `Authorization` header, and `verify_jwt = false` should be used only when another explicit auth mechanism protects the function. Sources: [Function Configuration](https://supabase.com/docs/guides/functions/function-configuration), [Securing Edge Functions](https://supabase.com/docs/guides/functions/auth), [Authorization headers](https://supabase.com/docs/guides/functions/auth-headers).
- Supabase’s current changelog did not surface a new June 2026 trust-specific breaking change for the current contact, tracking, or mailbox edge functions. The most relevant trust-adjacent standing notes remain the Data API exposure change for new tables and the hosted rate limit on recursive or nested Edge Function calls. Source: [Supabase changelog](https://supabase.com/changelog).
- OWASP Secure Headers still treats HSTS, CSP, framing protection, referrer policy, and permissions policy as active browser trust controls worth keeping in deployed checks. Source: [OWASP Secure Headers Project](https://owasp.org/www-project-secure-headers/index.html).
