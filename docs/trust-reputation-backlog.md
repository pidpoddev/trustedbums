## Current Standards And Time-Sensitive Notes

- Google Search Console still says indexing triage should check Manual Actions and Security Issues before assuming a crawl-only problem. Sources: [Manual actions report](https://support.google.com/webmasters/answer/9044175?hl=en), [URL Inspection tool](https://support.google.com/webmasters/answer/9012289?hl=en).
- Google Search Central still says sitemaps can be surfaced through Search Console, the Search Console API, or a `Sitemap:` line in `robots.txt`, and Google still treats crawlable internal links as discovery signals while treating manipulative link schemes as spam-policy risk. Sources: [Build and submit a sitemap](https://developers.google.com/search/docs/crawling-indexing/sitemaps/build-sitemap), [Link best practices](https://developers.google.com/search/docs/crawling-indexing/links-crawlable), [Google Search Essentials](https://developers.google.com/search/docs/essentials).
- Gmail sender guidance still expects DMARC on the sending domain and alignment through SPF or DKIM on the visible `From:` domain, and the current FAQ says full SPF-plus-DKIM alignment is the more reliable future-proof posture. Sources: [Email sender guidelines](https://support.google.com/mail/answer/81126?hl=en), [Email sender guidelines FAQ](https://support.google.com/mail/answer/14229414?hl=en).
- Bing still recommends IndexNow, direct URL submission, crawlable internal links, and healthy sitemap submission as the supported discovery path. Sources: [URL Submission](https://www.bing.com/webmasters/help/URL-Submission-62f2860b), [Bing Webmaster Guidelines](https://www.bing.com/webmasters/help/webmaster-guidelines-30fba23a).
- Microsoft’s current SmartScreen guidance, updated last month, still ties warnings to site reputation, phishing or malware signals, downloads, TLS posture, and user feedback. Sources: [Microsoft Defender SmartScreen overview](https://learn.microsoft.com/en-us/windows/security/operating-system-security/virus-and-threat-protection/microsoft-defender-smartscreen/), [Microsoft Edge support for Microsoft Defender SmartScreen](https://learn.microsoft.com/en-us/deployedge/microsoft-edge-security-smartscreen).
- Supabase’s current Edge Function docs, updated last week, still say `verify_jwt` is on by default, API keys belong in the `apikey` header rather than the `Authorization` header, and secret-specific auth is the preferred alternative when a function should accept only one named key. Sources: [Function Configuration](https://supabase.com/docs/guides/functions/function-configuration), [Securing Edge Functions](https://supabase.com/docs/guides/functions/auth), [Authorization headers](https://supabase.com/docs/guides/functions/auth-headers).
- Supabase’s current changelog did not surface a new June 2026 trust-specific breaking change for the current contact, tracking, mailbox, or claim-reply functions. The most relevant standing notes remain the Data API exposure change for new tables and the hosted rate limit on recursive or nested Edge Function calls. Source: [Supabase changelog](https://supabase.com/changelog).
