# Trusted Bums Trust And Reputation Backlog

_Last updated: 2026-06-04 by Codex daily trust and reputation consultant automation._

## Executive Read

The highest-leverage active risk is still public infrastructure that can be abused from the Trusted Bums domain. Current source shows the hardened homepage intake now funnels through `submit-contact`, but the downstream `send-website-email` function is still anonymously callable and the public `email-track/click` endpoint still redirects to arbitrary external URLs without requiring a valid delivery record.

The next most important gap is sender-authentication enforcement readiness. Current DNS confirms `_dmarc.trustedbums.com` is still `p=none`, which means Trusted Bums still has reporting but not receiver-side spoofing enforcement. Public-site trust polish also remains incomplete: current checked-in metadata is still root-relative and generic, and the checked-in Apache config still lacks explicit canonical-host and baseline security-header policy.

## Active Recommendations

### P0 - Remove public direct mail-send exposure outside the validated contact intake
- Evidence: [`supabase/functions/submit-contact/index.ts`](supabase/functions/submit-contact/index.ts) now enforces origin allowlisting, Turnstile Siteverify, honeypot handling, abuse fingerprinting, and short-window rate limits before inserting `contact_submissions` and invoking `send-website-email`. But [`supabase/functions/send-website-email/index.ts`](supabase/functions/send-website-email/index.ts) still accepts anonymous POST requests, returns `Access-Control-Allow-Origin: *`, and calls Microsoft Graph `users/{sender}/sendMail` directly. [`supabase/config.toml`](supabase/config.toml) still sets `verify_jwt = false` for both functions.
- Why it matters: A directly callable mail-sending endpoint on a B2B trust-sensitive domain is a sender-reputation and abuse-risk surface. If it gets scripted or spammed, Trusted Bums risks mailbox flooding, SmartScreen or gateway distrust, and degraded inbox placement for legitimate mail.
- Recommendation: Make `submit-contact` the only anonymous public intake path. Require `send-website-email` to reject direct anonymous callers with an internal shared secret or equivalent trusted-caller proof, then verify one allowed homepage submission and one denied direct invocation in production.
- Acceptance criteria: Anonymous direct calls to `send-website-email` return deny status; valid homepage contact submissions still succeed through `submit-contact`; invalid Turnstile, origin, or rate-limit cases do not send mail; and production logs show one allowed contact flow plus one denied direct call after deployment.

### P0 - Close the public tracking-link open redirect on the Trusted Bums domain
- Evidence: [`supabase/functions/email-track/index.ts`](supabase/functions/email-track/index.ts) still allows unauthenticated requests and issues `Response.redirect(targetUrl, 302)` for `/click` using the raw `u` query parameter. If the `deliveryId` lookup fails, `recordEvent` returns early, but the redirect still happens. [`supabase/config.toml`](supabase/config.toml) still sets `verify_jwt = false` for `email-track`.
- Why it matters: A branded open redirect can be reused in phishing, scanner-evasion, and reputation-harm workflows because the initial link still appears to originate from `trustedbums.com`. That is the kind of public endpoint behavior browsers, email gateways, and endpoint-security products scrutinize closely.
- Recommendation: Require every tracking click to resolve only through a stored allowlisted delivery record, and return a safe local fallback when the delivery id or destination is missing, expired, or mismatched. Do not redirect from raw query-string URLs on the public domain.
- Acceptance criteria: `/email-track/click` redirects only when the delivery id exists and the stored destination passes allowlist validation; forged or unknown delivery ids do not redirect externally; and one positive plus one negative production check is captured after rollout.

### P1 - Run a current mailbox-backed DMARC review and move from monitoring to enforcement
- Evidence: Current DNS from `dig @1.1.1.1 _dmarc.trustedbums.com TXT +short` returned `"v=DMARC1; p=none; rua=mailto:bums@trustedbums.com; pct=100;"` on 2026-06-04. The same run confirmed SPF at the apex via `"v=spf1 include:spf.protection.outlook.com -all"` and Microsoft DKIM selectors via `selector1._domainkey.trustedbums.com` and `selector2._domainkey.trustedbums.com` CNAMEs. Current source still includes the admin-gated DMARC parser in [`supabase/functions/dmarc-reports/index.ts`](supabase/functions/dmarc-reports/index.ts), but this run still had no safe authenticated admin invocation path for it. Current Google guidance still says Gmail enforcement on non-compliant traffic ramped up starting in November 2025, and Google’s BIMI guidance still requires DMARC `p=quarantine` or `p=reject` with `pct=100`. Sources: [Email sender guidelines FAQ](https://support.google.com/a/answer/14229414?hl=en), [Set up BIMI](https://support.google.com/a/answer/10911320?hl=en).
- Why it matters: `p=none` provides reporting but no receiver-side enforcement against spoofing. That weakens impersonation resistance, keeps BIMI unavailable, and leaves delivery resilience more exposed as receiver enforcement tightens.
- Recommendation: Run the deployed `dmarc-reports` review against the approved shared mailbox, enumerate every legitimate `trustedbums.com` sender, remediate any SPF or DKIM alignment drift, and advance DMARC to `p=quarantine` or `p=reject` with an explicit rollback owner.
- Acceptance criteria: Recent DMARC aggregate reports from `bums@trustedbums.com` are reviewed; all legitimate senders align through SPF or DKIM for the From domain; unexplained failures are remediated or explicitly accepted; and `_dmarc.trustedbums.com` moves to `p=quarantine` or `p=reject`.

### P1 - Enforce one public host and ship absolute route-specific metadata
- Evidence: [`public/.htaccess`](public/.htaccess) still contains SPA rewrites only and does not canonicalize `www` to a preferred host. [`index.html`](index.html) still publishes `<link rel="canonical" href="/" />`, `og:url` as `/`, and root-relative social image URLs. Public routes still include `/`, `/login`, `/privacy-policy`, and `/legal/:slug` in [`src/App.tsx`](src/App.tsx), while [`src/pages/PrivacyPolicy.tsx`](src/pages/PrivacyPolicy.tsx) and [`src/pages/LegalDocumentPage.tsx`](src/pages/LegalDocumentPage.tsx) compute page titles and descriptions in component state but do not write route-specific head tags. Independent web retrieval in this run could fetch HTML from both [trustedbums.com](https://trustedbums.com) and [www.trustedbums.com](https://www.trustedbums.com), so the public site is reachable from at least one outside network path even though the local runner resolver still failed.
- Why it matters: Canonical drift and generic metadata weaken crawler consolidation, make shared links look unfinished to buyers and reviewers, and reduce the chance that legal and trust pages render with the right context in search and social previews.
- Recommendation: Redirect all non-preferred public hosts to one canonical host and add a route-aware metadata layer that emits absolute canonical, Open Graph, and Twitter tags for each public route.
- Acceptance criteria: `www` redirects to the preferred host; public pages emit absolute canonical URLs on that host; Open Graph and Twitter tags use absolute URLs; and privacy/legal pages publish route-specific title and description metadata in the document head.

### P1 - Add production HSTS and explicit baseline security headers
- Evidence: Checked-in Apache config in [`public/.htaccess`](public/.htaccess) still sets content types and SPA rewrites only. It does not set `Strict-Transport-Security`, `Content-Security-Policy`, `Referrer-Policy`, `Permissions-Policy`, or a clickjacking control. Current OWASP Secure Headers guidance still treats these as baseline response-header controls, and Microsoft’s January 28, 2026 SmartScreen website-warning guidance now explicitly calls out CSP, secure response headers, HTTPS, and stable reputation signals as ways to reduce block risk. Sources: [OWASP Secure Headers Project](https://owasp.org/www-project-secure-headers/), [Unexpected SmartScreen block or warning on website](https://learn.microsoft.com/en-us/troubleshoot/microsoft-edge/development/unexpected-block-warning).
- Why it matters: Missing explicit browser-security headers weaken enterprise trust review posture, leave HTTPS downgrade and framing behavior implicit, and make a public B2B site look minimally hardened to both buyers and reputation systems.
- Recommendation: Add HSTS and explicit baseline headers at the host layer, then verify the live response set from the preferred production host after deployment.
- Acceptance criteria: HTTPS responses include `Strict-Transport-Security` with at least one year of `max-age` plus `includeSubDomains`; framing, referrer, and permissions policies are explicit; and any CSP is validated against the deployed app before rollout.
