## Active Recommendations

### P1 - Replace viewport-floating utility collisions with a coordinated mobile utility rail
- Evidence: Current Mobile Chrome screenshots from run `27083467531` still show fixed utility overlap after the layout padding change shipped in commit `850e507`. `mobile-chrome-admin-admin-dashboard.png` shows `Chat (4)` sitting on top of the `Priority queues` segmented controls while the tiny `Privacy` chip lands on the same card edge. `mobile-chrome-client_finance-client-finance-payments.png` shows `Chat (4)` covering the `Commission invoices` filter region and the `Privacy` chip pinned against the same working area. `mobile-chrome-bum-bum-dashboard.png` shows the chat launcher covering the profile-completeness form controls near `Save and continue`. Source shows the launchers remain fixed-position controls in [src/components/ConversationDock.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/ConversationDock.tsx) and [src/components/ConsentManager.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/ConsentManager.tsx), while the portal layouts now reserve only bottom scroll space in [src/layouts/AdminLayout.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/layouts/AdminLayout.tsx), [src/layouts/ClientLayout.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/layouts/ClientLayout.tsx), and [src/layouts/BumLayout.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/layouts/BumLayout.tsx).
- Why it matters: The shipped padding fix reduced one class of obstruction, but mobile users still see trust and workflow utilities covering tabs, filters, and primary actions at the bottom of the live viewport. That reads as unfinished on the exact routes where operators and Bums need confidence.
- Recommendation: Replace the independent floating `Chat` and `Privacy` controls with a single coordinated mobile utility rail or stacked launcher zone that reserves one bottom-right interaction region, respects `env(safe-area-inset-bottom)`, and keeps at least one utility collapsed when a form, segmented control row, or filter stack occupies the lower viewport.
- Acceptance criteria: The next GitHub Visual UI Audit mobile screenshots for `/admin`, `/client/payments`, `/bum/dashboard`, and `/privacy-policy` show no overlap between utility controls and tappable product UI. The launcher area uses safe-area-aware bottom spacing and preserves a single obvious access path to both chat and privacy controls.

### P1 - Replace the consent reopen micro-chip with a real mobile control and add privacy-page open-state evidence
- Evidence: `mobile-chrome-public-privacy-policy.png` still shows a tiny left-edge `Privacy` chip that is substantially smaller than the surrounding touch targets. Source still renders the closed-state reopen control as `h-5` with `text-[10px]` in [src/components/ConsentManager.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/ConsentManager.tsx). The current visual suite captures `/?consent=reset` in [tests/e2e/visual-ui-audit.spec.ts](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/tests/e2e/visual-ui-audit.spec.ts) but does not capture `/privacy-policy?consent=reset` or `/privacy-policy?consent=open`.
- Why it matters: Consent is a trust control, not a hidden affordance. A tiny reopen chip lowers confidence and increases precision demand on mobile.
- Recommendation: Turn the dismissed consent entry point into a normal-sized `Privacy choices` pill/button that shares the coordinated utility pattern above, then add a Visual UI Audit state for the privacy page with consent controls reopened from the legal route itself.
- Acceptance criteria: The mobile consent reopen control is visibly tappable without precision aiming, its target size and spacing no longer depend on a micro-chip, and the `visual-ui-audit` artifact includes a privacy-page consent-open screenshot.

### P1 - Reframe public legal links and policy sections as mobile trust navigation instead of fine print
- Evidence: `mobile-chrome-public-privacy-policy.png` still renders `Related Legal Links` as dense wrapped inline links under the consent card, and the rest of the policy page becomes a very long undifferentiated text stack on a narrow phone viewport. Source still uses inline/wrap-style legal link presentation in [src/pages/PrivacyPolicy.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/PrivacyPolicy.tsx) and related public trust navigation in [src/pages/Index.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/Index.tsx).
- Why it matters: Public legal pages are a buyer-trust surface. On phones, the current presentation reads more like compressed boilerplate than deliberate trust navigation.
- Recommendation: Group legal destinations into clearer mobile categories and present them as stacked rows, pill buttons, or short accordions with consistent ordering. Add stronger section dividers or a compact in-page nav so the privacy policy reads as structured guidance rather than one long scroll wall.
- Acceptance criteria: Public legal links no longer wrap into dense inline clusters on mobile, each destination has a comfortable tap area, and the privacy page exposes visibly distinct section groupings or navigation anchors that improve scanability.

### P1 - Sync Bum dashboard payout cards with the live earnings totals model
- Evidence: `mobile-chrome-bum-bum-dashboard.png` still shows `Pending Earnings` and `Lifetime Payouts` as `$0` on the dashboard, while `mobile-chrome-bum-bum-earnings.png` shows the dedicated earnings page using live payout-summary cards. Source confirms the dashboard still hardcodes `$0` values in [src/pages/bum/BumDashboard.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/bum/BumDashboard.tsx), while [src/pages/bum/BumEarnings.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/bum/BumEarnings.tsx) computes pending and paid totals from payout records.
- Why it matters: A Bum-facing finance summary that looks placeholder-driven weakens payout trust even when the detailed earnings page is already source-of-truth-backed.
- Recommendation: Use the same payout aggregation on the dashboard as the earnings page, or intentionally relabel the dashboard cards as unavailable/pending until live totals are loaded there.
- Acceptance criteria: Dashboard payout cards either match the `/bum/earnings` totals logic or explicitly communicate an intentional unavailable state. Loading, empty, and true-zero states are visually distinct.
