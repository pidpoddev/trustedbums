# Trusted Bums UI Optimization Backlog

_Last updated: 2026-05-28 by Codex daily UI consultant automation._

## Executive Read

Current evidence still supports three high-value authenticated UI fixes: restore route context in the collapsed mobile header, stop showing fake `$0` earnings on the Bum dashboard, and rebalance the six-card KPI rows so they scan intentionally on large screens. This run also adds one trust-facing public UI issue: once the consent banner is dismissed, the remaining `Privacy` reopen control is reduced to a tiny bottom tab that is easy to miss and visually inconsistent with the stronger privacy language on the policy page. Live screenshots are still blocked because the QA host does not resolve from this environment, and even local `pnpm preview` currently renders the Clerk configuration-needed state without a preview-safe publishable key.

## Active Recommendations

### P1 - Restore current-route context in mobile authenticated headers
- Evidence: The current-route label is still hidden below `sm` in [AdminLayout.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/layouts/AdminLayout.tsx:87), [ClientLayout.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/layouts/ClientLayout.tsx:125), and [BumLayout.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/layouts/BumLayout.tsx:95). The mobile search sheet remains available, but Apple’s search-field guidance still treats search scope and context as complementary, not interchangeable, cues.
- Why it matters: On phones, admins, client finance users, and Bums lose the visible “where am I?” signal at the same breakpoint where the sidebar collapses. That weakens navigation confidence during quick in-portal checks.
- Recommendation: Add one compact current-route treatment below `sm` across all authenticated shells, such as a short page label beside the sidebar trigger or a second compact header row that coexists with search and account actions.
- Acceptance criteria: Below `sm`, each authenticated layout shows the active route name without opening the sidebar; the label stays readable beside existing header actions; the header does not clip or wrap unpredictably on narrow mobile widths.

### P1 - Replace hardcoded Bum earnings dashboard stats with live totals or honest placeholders
- Evidence: [BumDashboard.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/bum/BumDashboard.tsx:227) still renders `Pending Earnings` and `Lifetime Payouts` as fixed `$0` cards, while [BumEarnings.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/bum/BumEarnings.tsx:13) already computes live pending and paid totals from `listBumPayouts()`.
- Why it matters: Placeholder finance numbers read as stale or untrustworthy faster than an explicit empty state does, especially when the dedicated earnings screen can already show real values.
- Recommendation: Reuse the payout totals from `BumEarnings` on the dashboard, or replace those cards with clearly labeled “View earnings” placeholders until live payout data is ready.
- Acceptance criteria: The Bum dashboard values match `/bum/earnings`, or the cards explicitly communicate that live earnings are unavailable instead of showing `$0`; loading and empty states remain visually distinct from actual zero earnings.

### P2 - Rebalance the six-card KPI rows on client and Bum dashboards
- Evidence: [ClientDashboard.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientDashboard.tsx:252) and [BumDashboard.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/bum/BumDashboard.tsx:227) still place six equal-weight `StatCard`s inside `lg:grid-cols-4`. The shared card primitive in [StatCard.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/StatCard.tsx:13) gives every metric the same emphasis, so large screens still fall into a visual `4 + 2` leftovers pattern. Microsoft’s dashboard-card guidance continues to recommend keeping the most important header content prominent and interactions intentionally structured.
- Why it matters: These are high-frequency first-scan surfaces for core roles. The stranded final row makes the dashboards feel less composed and slows metric comparison.
- Recommendation: Move the six-card sections to an intentional desktop composition such as `xl:grid-cols-3` for a balanced `3 x 2` layout, or split them into primary and secondary metric groups if some cards deserve stronger hierarchy.
- Acceptance criteria: On large screens, both dashboard KPI sections render as balanced rows with no stranded two-card row; card heights remain aligned within each row; labels and values stay readable at tablet and desktop breakpoints; keyboard focus order still matches visual reading order.

### P2 - Make the privacy-choices re-entry control discoverable after banner dismissal
- Evidence: Once the banner closes, [ConsentManager.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/ConsentManager.tsx:141) leaves only a fixed `Privacy` tab with `h-5` and `text-[10px]`, while [PrivacyPolicy.tsx](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/PrivacyPolicy.tsx:97) presents consent management as a prominent trust control. Current CNIL guidance still warns against reject or privacy controls becoming hard to notice through size, styling, or placement.
- Why it matters: Trusted Bums is now explicitly presenting consent choice as part of its trust posture. Shrinking the persistent reopen affordance into a tiny bottom tab makes that promise harder to act on and easier to miss on mobile browsers.
- Recommendation: Replace the tiny closed-state tab with a larger, consistently labeled `Privacy choices` chip or button that remains visible above mobile browser chrome and visually matches the legal/privacy treatment elsewhere in the public experience.
- Acceptance criteria: After dismissing the banner, users can reopen consent settings from a control that remains visible on mobile and desktop, uses the same `Privacy choices` wording as the banner/policy flow, and meets a clearly tappable target size without overlapping core page actions.
