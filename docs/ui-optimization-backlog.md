## Active Recommendations

### P1 - [TB-0061] Turn mobile reports into progressive disclosure instead of a full control stack
- Evidence: Exact-head mobile reports still spend most of the opening scroll on configuration before the first useful preview. [mobile-chrome-client_finance-client-finance-reports.png](/private/tmp/trustedbums-visual-27395701277/visual-ui-audit/test-results/visual-ui-audit-authentica-c1452-portal-pages-render-cleanly-mobile-chrome/mobile-chrome-client_finance-client-finance-reports.png) shows the report chooser, date window, and field checklist consuming the top of the viewport before the preview card starts. Current shared source in [ReportsWorkspace.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/reports/ReportsWorkspace.tsx) still renders the full report list, date selector, and visible-field checklist ahead of the preview on narrow screens. W3C’s current accordion guidance still positions accordions as the pattern for reducing scroll across stacked sections: [WAI APG accordion pattern](https://www.w3.org/WAI/ARIA/apg/patterns/accordion/).
- Why it matters: Reports should answer a question quickly. On mobile, the current sequence still feels like setup work before results.
- Recommendation: Collapse report selection and field toggles into progressive disclosure on narrow screens, such as an accordion, sheet, or drawer, while keeping export, selected-report context, top metrics, and the first preview block within the first viewport or short scroll.
- Acceptance criteria: Mobile screenshots for `/admin/reports`, `/client/reports`, and `/bum/reports` show the active report summary and first chart or table preview before a long settings stack. Field toggles are collapsed by default, and export remains reachable without scrolling through the whole checklist.

### P1 - [TB-0062] Rebuild privacy and legal mobile controls as trust navigation, not micro-utilities
- Evidence: Exact-head [mobile-chrome-public-privacy-policy.png](/private/tmp/trustedbums-visual-27395701277/visual-ui-audit/test-results/visual-ui-audit-public-vis-0a461-ivacy-states-render-cleanly-mobile-chrome/mobile-chrome-public-privacy-policy.png) still shows a tiny bottom-left `Privacy` reopen chip plus a dense wrapped `Related Legal Links` cluster. Current source still renders the closed-state entry point as `h-5` with `text-[10px]` in [ConsentManager.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/ConsentManager.tsx), and [PrivacyPolicy.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/PrivacyPolicy.tsx) still relies on a wrapped inline legal-link row. WCAG 2.2 still sets a `24 by 24 CSS pixels` minimum target size for pointer inputs: [W3C target size minimum](https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum.html).
- Why it matters: Privacy and legal routes are public trust surfaces. Tiny reopen controls and cramped legal navigation still make these pages feel improvised instead of deliberate and credible.
- Recommendation: Replace the tiny reopen chip with a normal-size `Privacy choices` button or pill that matches the shared support pattern, and restyle related legal destinations into stacked rows, grouped pills, or compact accordions with clearer section rhythm on mobile.
- Acceptance criteria: The dismissed consent entry point is visibly tappable without precision aiming, legal destinations no longer wrap into a cramped inline cluster, and the privacy page exposes separated section groups or navigation that scans like an intentional trust surface.

### P1 - [TB-0098] Default mobile client opportunities to workflow context before the full new-opportunity form
- Evidence: Exact-head [mobile-chrome-client_admin-client-register-opportunity.png](/private/tmp/trustedbums-visual-27395701277/visual-ui-audit/test-results/visual-ui-audit-authentica-71083-portal-pages-render-cleanly-mobile-chrome/mobile-chrome-client_admin-client-register-opportunity.png) shows `/client/opportunities/new` landing directly in the long `New opportunity` form after only a small action row and `Show` selector. The completed exact-head [mobile-chrome-client_admin-client-opportunities.png](/private/tmp/trustedbums-complete-27400338279/complete-visual-ui-audit/test-results/visual-ui-audit-authentica-71083-portal-pages-render-cleanly-mobile-chrome/mobile-chrome-client_admin-client-opportunities.png) confirms the pipeline view is already viable on mobile, so the remaining issue is route default and information order, not lack of a mobile pipeline surface. Current source in [ClientOpportunityNew.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/client/ClientOpportunityNew.tsx) still defaults `activeView` to `register` when the route ends with `/new`, leaving `Pipeline`, `Bum-originated`, and `Responses` one selector step away on phone-sized screens. The current client dashboard already exposes `Create opportunity` and related next actions in [ClientDashboard.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/client/ClientDashboard.tsx), so the dedicated opportunity workspace does not need to start with the longest possible form.
- Why it matters: On mobile, clients need orientation and current-work context before another long entry flow. Starting in the form hides drafts, published opportunities, and incoming Bum work behind setup chrome.
- Recommendation: Default the narrow-screen opportunity workspace to a pipeline-first summary or action chooser, then open the full `New opportunity` form only after an explicit action. Keep `New opportunity` available, but do not make it the first-screen default on `/client/opportunities/new`.
- Acceptance criteria: The first mobile viewport for `/client/opportunities` or `/client/opportunities/new` shows current pipeline context, draft/published state, or incoming workflow queues before the full form. The full registration form remains reachable in one tap, but it no longer blocks initial orientation.

### P2 - [TB-0060] Narrow the remaining admin mobile utility anchors into one intentional support zone
- Evidence: The old cross-role utility overlap is now narrower in exact-head source, but admin mobile still carries two independent support anchors. [mobile-chrome-admin-admin-dashboard.png](/private/tmp/trustedbums-visual-27395701277/visual-ui-audit/test-results/visual-ui-audit-authentica-1b0bc-portal-pages-render-cleanly-mobile-chrome/mobile-chrome-admin-admin-dashboard.png) shows the floating chat launcher and the tiny `Privacy` reopen chip on the same screen, while committed source still mounts the launcher in [AdminLayout.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/layouts/AdminLayout.tsx) and still renders the closed-state consent chip in [ConsentManager.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/ConsentManager.tsx). MDN’s `env()` guidance still supports safe-area-aware fixed placement near viewport edges: [MDN env()](https://developer.mozilla.org/en-US/docs/Web/CSS/Reference/Values/env).
- Why it matters: Admin mobile still has two separate support utilities competing for the lower corners, which makes the shell feel incidental instead of composed.
- Recommendation: Collapse admin-only support utilities into one safe-area-aware help/support zone or coordinated launcher pattern instead of leaving chat and privacy as two unrelated fixed anchors.
- Acceptance criteria: The next mobile screenshots for `/admin`, `/admin/opportunities`, and `/admin/live-conversations` show one intentional support anchor pattern rather than two independent corner controls. The surviving anchor respects `env(safe-area-inset-bottom)` and does not cover action buttons or queue controls.
