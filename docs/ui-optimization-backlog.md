## Active Recommendations

### P1 - [TB-0060] Keep the mobile `Privacy choices` control clear of live authenticated content
- Evidence: Exact-head hosted `Visual UI Audit` run `27857691601` on `e231cc0` shows the collapsed `Privacy choices` chip still covering live mobile UI even after the chat-launcher cleanup. In `mobile-chrome-admin-admin-dashboard.png` it lands on the `Priority queues` tab row while admin still stacks the orange chat launcher in the same corner; in `mobile-chrome-admin-admin-scrum-tracker.png` it sits on the summary-to-table transition; in `mobile-chrome-admin-admin-handoffs.png` it overlaps the mobile search/filter zone; in `mobile-chrome-client_admin-client-register-opportunity.png` it sits on the `Estimated value` card; and in `mobile-chrome-bum-bum-dashboard.png` plus `mobile-chrome-bum-bum-contacts.png` it still intrudes into live form or card space. Current source confirms the narrowed defect: [ClientLayout.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/layouts/ClientLayout.tsx) and [BumLayout.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/layouts/BumLayout.tsx) now render `<ConversationDock showLauncher={false} />`, but [ConsentManager.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/ConsentManager.tsx) still mounts the privacy control as a fixed bottom-right overlay. Current platform guidance still supports safe-area-aware placement through [MDN `env()`](https://developer.mozilla.org/en-US/docs/Web/CSS/Reference/Values/env), and current accessibility guidance keeps the issue framed as placement rather than target size because [WCAG 2.5.8](https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum.html) centers on minimum target size and spacing, not permission to cover live content.
- Why it matters: The product now looks materially cleaner than yesterday’s artifact, but the remaining fixed privacy chip still makes authenticated mobile routes feel improvised because it covers metrics, queue transitions, filters, and in-progress form areas exactly where users need visual confidence.
- Recommendation: Treat the privacy launcher as a route-aware mobile utility instead of a generic floating overlay. On authenticated mobile shells, either move it into an intentional surface such as the account or support menu, or reserve a bottom-right exclusion lane so the chip never covers cards, tabs, filters, queue copy, or form controls. Admin still needs coordinated placement with the chat launcher; client and Bum shells now only need privacy-control clearance.
- Acceptance criteria: Fresh exact-head mobile screenshots for `/admin`, `/admin/scrum`, `/admin/handoffs`, `/client/opportunities/new`, and one Bum workflow route show no `Privacy choices` overlap with tabs, filters, cards, queues, or CTAs. On admin, the privacy chip and chat launcher no longer stack into live content. The control remains reachable and safe-area-aware on devices that expose a bottom inset.
