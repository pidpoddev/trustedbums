## Active Recommendations

### P1 - [TB-0060] Replace competing fixed utilities with one coordinated mobile utility rail
- Evidence: Fresh mobile screenshots from GitHub `Visual UI Audit` run `27167324836` still show the fixed `Chat` launcher and the `Privacy` reopen chip competing with live product controls on [mobile-chrome-admin-admin-handoffs.png](/Users/macdaddy/tmp/trustedbums-visual-27167324836/test-results/visual-ui-audit-authentica-1b0bc-portal-pages-render-cleanly-mobile-chrome/mobile-chrome-admin-admin-handoffs.png) and [mobile-chrome-client_finance-client-finance-reports.png](/Users/macdaddy/tmp/trustedbums-visual-27167324836/test-results/visual-ui-audit-authentica-c1452-portal-pages-render-cleanly-mobile-chrome/mobile-chrome-client_finance-client-finance-reports.png). Source still mounts those controls separately in [ConversationDock.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/ConversationDock.tsx) and [ConsentManager.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/ConsentManager.tsx). MDN’s current `env()` guidance still recommends safe-area-aware placement for fixed controls near viewport edges: [MDN env()](https://developer.mozilla.org/en-US/docs/Web/CSS/env).
- Why it matters: Admins, clients, and Bums are seeing trust/support utilities cover the filters, selectors, and search controls they need to finish real work.
- Recommendation: Merge chat and privacy entry points into one safe-area-aware mobile utility rail or stacked launcher zone with route-aware collapse rules when the lower viewport already contains active controls.
- Acceptance criteria: The next completed mobile screenshots for `/admin/handoffs`, `/client/reports`, `/client/payments`, and `/bum/dashboard` show no overlap between utility controls and tappable product UI. The shared utility zone respects `env(safe-area-inset-bottom)` and still exposes both chat and privacy choices.

### P1 - [TB-0061] Turn mobile reports into progressive disclosure instead of a full control stack
- Evidence: [mobile-chrome-client_finance-client-finance-reports.png](/Users/macdaddy/tmp/trustedbums-visual-27167324836/test-results/visual-ui-audit-authentica-c1452-portal-pages-render-cleanly-mobile-chrome/mobile-chrome-client_finance-client-finance-reports.png) still spends most of the first 2,239px of page height on report selection, date range, and field toggles before the first real preview block. The shared layout in [ReportsWorkspace.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/reports/ReportsWorkspace.tsx) still renders the full report list, date selector, and visible-field checklist ahead of the preview on narrow screens for Admin, Client, and Bum reports.
- Why it matters: Reporting should answer a question quickly. On mobile, the current stack makes the reporting workspace feel like setup work before the user reaches any result.
- Recommendation: Collapse report selection and field toggles into progressive disclosure on narrow screens, such as an accordion, drawer, or sticky “Customize report” sheet, while keeping the selected report summary, export action, metrics, and first preview block within the first viewport or short scroll.
- Acceptance criteria: Mobile screenshots for `/admin/reports`, `/client/reports`, and `/bum/reports` show the active report summary and first chart/table preview before a long settings scroll. Field toggles are collapsed by default, and export plus key metrics stay reachable without passing through the entire checkbox list.

### P1 - [TB-0062] Rebuild privacy and legal mobile controls as trust navigation, not micro-utilities
- Evidence: [mobile-chrome-public-privacy-policy.png](/Users/macdaddy/tmp/trustedbums-visual-27167324836/test-results/visual-ui-audit-public-vis-0a461-ivacy-states-render-cleanly-mobile-chrome/mobile-chrome-public-privacy-policy.png) still shows a tiny left-edge `Privacy` reopen chip, a dense wrapped `Related Legal Links` cluster, and a long uninterrupted section stack. Source still renders the closed-state control as `h-5` with `text-[10px]` in [ConsentManager.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/ConsentManager.tsx), and [PrivacyPolicy.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/PrivacyPolicy.tsx) still uses a simple wrapped link row plus long section blocks. W3C WCAG 2.2 added `Target Size (Minimum)` and Android’s current accessibility guidance still recommends at least `48dp x 48dp` touch targets: [W3C WCAG 2.2 overview](https://www.w3.org/WAI/standards-guidelines/wcag/new-in-22/) and [Android accessibility views guidance](https://developer.android.com/guide/topics/ui/accessibility/views/apps-views?hl=en).
- Why it matters: Privacy and legal screens are part of the public trust journey. Precision-sized reopen controls and dense link clusters make compliance surfaces feel hidden and improvised instead of credible and reassuring.
- Recommendation: Replace the tiny reopen chip with a normal-size `Privacy choices` pill/button that matches the coordinated utility pattern, and restyle related legal links plus section navigation into stacked rows, pills, or compact accordions with clearer scanning rhythm on mobile.
- Acceptance criteria: The dismissed consent entry point is visibly tappable without precision aiming, related legal destinations no longer wrap into a cramped inline cluster, and the privacy page exposes visibly separated section groups or in-page navigation that scans like an intentional trust surface.

### P1 - [TB-0063] Replace narrow-screen admin tables with responsive queue cards, and add hosted visual coverage for the new route pair
- Evidence: [mobile-chrome-admin-admin-handoffs.png](/Users/macdaddy/tmp/trustedbums-visual-27167324836/test-results/visual-ui-audit-authentica-1b0bc-portal-pages-render-cleanly-mobile-chrome/mobile-chrome-admin-admin-handoffs.png) still shows table-first queue sections with clipped column headers and phone-width scanning pressure. Source confirms that [AdminHandoffs.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/admin/AdminHandoffs.tsx) still renders three seven-column tables, while the new [AdminScrumTracker.tsx](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/admin/AdminScrumTracker.tsx) uses a five-column interactive table with wide `min-w-*` cells behind horizontal overflow. The current visual workflow spec in [visual-ui-audit.spec.ts](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/tests/e2e/visual-ui-audit.spec.ts) covers `/admin/handoffs` but still omits the new `/admin/scrum` route and the new public `/bums` landing page. USWDS’s current table guidance still warns that long-form or non-tabular content should move out of dense tables and that mobile stacked variants need deliberate handling: [USWDS Table guidance](https://designsystem.digital.gov/components/table/).
- Why it matters: Admin triage is operational work, not spreadsheet review. On phones, horizontally-scrolled queue tables and uncaptured new routes make the highest-risk admin workflows harder to trust, harder to scan, and harder to review after changes.
- Recommendation: Keep full tables for desktop, but add card/list summaries or stacked row treatments for narrow widths on `/admin/handoffs` and `/admin/scrum`. In the same pass, extend the hosted visual audit to capture `/admin/scrum` and `/bums` on desktop and mobile so future UI reviews stop relying on source-only inference for those routes.
- Acceptance criteria: Narrow-screen admin queue screenshots show readable row summaries without cropped headers or horizontal table scanning. The next completed `visual-ui-audit` artifact includes `/admin/scrum` and `/bums` screenshots on both desktop and mobile, plus at least one meaningful active state for the scrum tracker or public signup flow.
