## Active Recommendations

### P1 - Reserve a mobile-safe zone for floating privacy and chat controls
- Evidence: GitHub Visual QA run `26931897223` Mobile Chrome screenshots still show fixed utilities covering live UI: `mobile-chrome-admin-admin-dashboard.png` and `mobile-chrome-admin-admin-global-search-results.png` place `Chat (4)` over the `Priority queues` tabs while the `Privacy` tab sits on the same card edge; `mobile-chrome-admin-admin-opportunities.png` places both utilities over opportunity cards; `mobile-chrome-bum-bum-dashboard.png` places `Chat` over the profile-completeness form and `Privacy` over helper copy. Source confirms `ConversationDock` remains fixed at `bottom-4 right-4` in [src/components/ConversationDock.tsx](src/components/ConversationDock.tsx:150) and the consent reopen tab remains fixed at `bottom-0 left-3` in [src/components/ConsentManager.tsx](src/components/ConsentManager.tsx:146). Apple's current layout guidance still says content and controls should respect safe areas and avoid obscuring essential information.
- Why it matters: Persistent utilities should not obscure primary workflow controls. Overlapping chat/privacy affordances make the mobile portal feel crowded and can hide action tabs, form text, and card content for Admin, Client, and Bum users.
- Recommendation: Create a coordinated mobile floating-control layout, such as a single bottom utility rail, safe-area-aware stacked controls, or route-level bottom padding plus collision rules. Keep chat and privacy reachable without covering cards, tab rows, form fields, or primary actions.
- Acceptance criteria: In GitHub Visual QA mobile screenshots for Admin dashboard, Client dashboard, Bum dashboard, and public privacy/home pages, `Chat` and `Privacy choices` controls remain visible but do not overlap interactive tabs, form content, primary buttons, or card text; pages reserve adequate bottom/side inset for fixed utilities; desktop placement remains unchanged or intentionally refined.

### P1 - Replace the tiny consent reopen tab and keep privacy controls unobscured on mobile
- Evidence: `mobile-chrome-public-privacy-policy.png` still shows the dismissed consent entry point as a very small left-edge `Privacy` tab beside the `Related Legal Links` module. Source still renders the closed-state trigger at `h-5` with `text-[10px]` in [src/components/ConsentManager.tsx](src/components/ConsentManager.tsx:146). The current Visual QA suite captures `/?consent=reset` on the homepage but not `/privacy-policy?consent=reset`, so the privacy-page open state remains unverified in the authoritative artifact. [W3C WCAG 2.2](https://www.w3.org/TR/WCAG22/) still sets a 24x24 CSS pixel minimum target in SC 2.5.8.
- Why it matters: Consent is a visible trust signal. A tiny reopen affordance and unverified privacy-page open state make privacy controls feel improvised on the most trust-sensitive public page.
- Recommendation: Replace the closed-state tab with a larger `Privacy choices` chip or button on mobile, and add GitHub Visual QA coverage for the consent manager opened from the privacy page. Present the open state so it preserves visibility of the page's `Manage Consent` and legal modules.
- Acceptance criteria: On mobile widths, the dismissed consent entry point clearly meets minimum touch-target expectations; GitHub Visual QA includes a privacy-page open-consent screenshot; opening consent does not cover the privacy page's `Manage Consent` card or `Related Legal Links`; the same pattern works on public and authenticated pages.

### P1 - Turn public legal links into grouped mobile trust navigation
- Evidence: `mobile-chrome-public-privacy-policy.png` still renders `Related Legal Links` as a dense multi-line word cluster. Source confirms the homepage footer still renders every legal destination in one wrapping row in [src/pages/Index.tsx](src/pages/Index.tsx:666), and the privacy page mirrors that pattern in [src/pages/PrivacyPolicy.tsx](src/pages/PrivacyPolicy.tsx:123). Apple's current layout guidance still recommends grouping related items and giving essential information enough space to stay legible.
- Why it matters: These links are part of the site's legitimacy proof. On phones they currently read like compressed fine print instead of a confident legal-navigation system.
- Recommendation: Group legal destinations into clearer mobile clusters such as `Privacy`, `Terms`, and `Operations`, then render them as stacked rows, pill links, or compact accordions with more spacing and clearer labels.
- Acceptance criteria: On mobile, legal destinations read as grouped navigation rather than a wrapped word cloud; each item has a comfortable tap area; the homepage footer and privacy page use the same grouping pattern.

### P1 - Replace hardcoded Bum earnings KPIs with live totals or an explicit pending state
- Evidence: `mobile-chrome-bum-bum-dashboard.png` still shows `Pending Earnings` and `Lifetime Payouts` as `$0` while the same dashboard shows live opportunity counts and request activity. Source still hardcodes both dashboard values to `$0` in [src/pages/bum/BumDashboard.tsx](src/pages/bum/BumDashboard.tsx:232), while [src/pages/bum/BumEarnings.tsx](src/pages/bum/BumEarnings.tsx:15) already computes pending and paid payout totals from live data.
- Why it matters: Static finance metrics look like placeholder UI, which undermines payout trust on the most credibility-sensitive Bum surface.
- Recommendation: Feed the dashboard cards from the same totals used on `/bum/earnings`, or replace them with an explicit earnings-status module until live payout data is intentionally unavailable.
- Acceptance criteria: Dashboard earnings metrics match `/bum/earnings`, or the dashboard explicitly communicates that payout totals are unavailable; loading and empty states are visibly distinct from a true zero balance.

### P2 - Restore current-route context in mobile authenticated headers
- Evidence: Mobile GitHub screenshots for Admin and Bum routes still show header controls, avatar, and search affordances without a persistent current-page label in the shell chrome. Source still hides the route label below `sm` in [src/layouts/AdminLayout.tsx](src/layouts/AdminLayout.tsx:133), [src/layouts/ClientLayout.tsx](src/layouts/ClientLayout.tsx:128), and [src/layouts/BumLayout.tsx](src/layouts/BumLayout.tsx:98).
- Why it matters: The current-page cue disappears at the same breakpoint where navigation collapses, which weakens orientation for every authenticated role on small screens, especially after scrolling past the page H1.
- Recommendation: Add a compact below-`sm` route label beside the sidebar trigger or in a second thin header row that coexists with search and account controls across all three portal shells.
- Acceptance criteria: On mobile authenticated routes, users can identify the current page without opening the sidebar; the label remains readable beside existing controls; no header clipping or wrap regressions appear on narrow widths in GitHub Visual QA.

### P2 - Add a responsive presentation and GitHub Visual QA coverage for Admin handoff queues
- Evidence: [src/pages/admin/AdminHandoffs.tsx](src/pages/admin/AdminHandoffs.tsx:355) still renders three separate seven-column tables with row-level claim/status actions, but [tests/e2e/visual-ui-audit.spec.ts](tests/e2e/visual-ui-audit.spec.ts:20) still does not include `/admin/handoffs` in `routesByRole`. GitHub Visual QA run `26931897223` therefore produced no handoff screenshot. The current [U.S. Web Design System table guidance](https://designsystem.digital.gov/components/table/) still recommends choosing a scrollable or stacked variant for tables with more than two columns on small screens.
- Why it matters: This is an operational queue surface. Dense multi-table layouts without responsive treatment are hard to scan on laptop widths and are likely to collapse poorly on mobile or smaller desktop breakpoints.
- Recommendation: Add a narrower-screen presentation for handoff rows, such as priority fields stacked into cards or expandable rows, and then add `/admin/handoffs` to the GitHub Visual QA suite.
- Acceptance criteria: `/admin/handoffs` is covered by GitHub Visual QA; at narrower widths the queue remains readable without hidden critical fields or uncontrolled horizontal scroll; claim and status actions stay visible without requiring precision scrolling.
