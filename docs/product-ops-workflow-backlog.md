## Workflow Map

- Public intake and handoffs: public visitor submits contact intent -> `submit-contact` creates `contact_submissions` -> Admin qualifies, claims, prioritizes, schedules follow-up, escalates, or archives the queue item.
- Access-review and Bum signup exceptions: `profile-bootstrap` creates access requests when auto-approval is unsafe -> `admin-access-requests` lets Admin approve or deny -> `BUM_SIGNUP` can now approve a Bum and trigger the approval-email workflow through the audited admin path.
- Client team and role governance: Client Admin invites same-domain client users and assigns company-scoped roles -> `client-team` can approve only `SAME_DOMAIN_ACCESS` requests for that company -> related-domain, public-email, Bum-signup, and future beta-integration exceptions stay in an Admin review lane.
- Source-only beta integration workflow: `Client IT` or `Client Admin` configures provider, auth method, required fields, approval handling, and fallback workflow in `/client/profile` -> Admin and Product Ops must still define when that setup becomes operationally enabled because production does not yet have the live schema or queue states for this branch.
- Target response workflow: Bum proposes a warm path on a client target -> Admin or client reviews the response -> the item is claimed, accepted, declined, or formalized into later opportunity work.
- Client-to-Bum intro requests: client requests a specific Bum intro -> Admin claims and advances the request through `SUBMITTED`, `IN_REVIEW`, `INTRO_REQUESTED`, and `CLOSED`.
- Bum-sourced customer lead workflow: Bum submits a customer lead in `/bum/reverse-opportunities` -> `reverse_opportunities` enters the admin opportunity queue -> Admin routes it toward outreach, client interest, conversion, or close-lost follow-up.
- Finance workflow: customer payment report is created -> invoice is generated, sent, and marked paid -> payout is allocated, approved, and marked paid, or the item should enter an explicit finance exception lane before real volume arrives.
- Admin scrum workflow: QA, release, security, Product Ops, or lead-dev findings create `admin_scrum_items` -> admins triage owner, priority, and proof -> issue closes only after a documented resolution or waiver.
