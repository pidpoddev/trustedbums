## Business Access Rule Recommendations

- Treat `customer_target_responses` as a distinct access object. `PROPOSED`, `ACCEPTED`, and `DECLINED` responses should be visible to Admins, the submitting Bum, and target-managing client roles only; Client Finance should default to deny.
- Define `client_bum_intro_requests` by lifecycle. Client Admin and Client Member can create and read company requests, the addressed Bum can read only assigned requests, Admin can manage statuses, and Client Finance stays out by default.
- Define accepted or locked visibility triggers for opportunity handoffs. Once an opportunity has an accepted or approved claim or a formalized target response, unrelated Bums should lose claimable visibility.
- Keep finance-linked conversations and transcripts out of broad Client Finance access. If finance needs dispute context, use explicit case participation or a dedicated finance-note surface rather than company-wide transcript visibility.

## Workflow Map

- Public support intake: Visitor submits contact form -> admin reviews the contact queue -> operator replies, archives, escalates to client target, or converts the submitter into a prospective Bum.
- Bum-to-client lead flow: Bum submits `customer_target_response` against a client target -> client reviews and either advances or declines it -> accepted responses can become an opportunity plus approved claim -> admin still lacks a dedicated stale or rescue queue for this handoff.
- Client finance flow: Client or admin reports customer payment -> invoice is generated -> admin marks invoice sent or paid -> paid invoice can create Bum payout allocation -> exception and reconciliation handling is still weak compared with the happy path.
- Client-to-Bum direct intro flow: Client submits `client_bum_intro_request` from the Bum directory -> request is visible in client history and queryable by the requested Bum or Admin -> no current admin queue, Bum queue, or in-product status-change tooling exists to run the rest of the lifecycle.
- Live conversation flow: Admins and Bums schedule Teams meetings from target queues -> attendance can be refreshed -> transcripts are attached or synced later -> transcript rescue currently depends on manual tools rather than an explicit exception queue.

## Current Standards And Time-Sensitive Notes

- Payments and marketplace risk operations: Stripe’s current Connect guidance says dispute responsibility, negative balances, and reserves are normal platform operations that must be handled explicitly depending on charge type and liability model. That supports first-class exception queues rather than off-platform notes. Sources: [Stripe disputes on Connect](https://docs.stripe.com/connect/disputes?locale=en-GB), [Stripe connected account reserves](https://docs.stripe.com/connect/connected-account-reserves?locale=en-GB), [Stripe account balances](https://docs.stripe.com/connect/account-balances?locale=en-GB).
- Support workflow: Zendesk’s current SLA guidance still centers support operations on explicit priorities and measurable first-reply and resolution targets, which aligns with adding owner, age, and breach visibility to Trusted Bums support queues. Sources: [Zendesk SLA policies overview](https://support.zendesk.com/hc/en-us/articles/5600997516058-About-SLA-policies-and-how-they-work), [Zendesk SLA policy API model](https://developer.zendesk.com/api-reference/ticketing/business-rules/sla_policies/).
- Transcript operations: Microsoft’s current Graph guidance still treats transcript availability as a subscription and lifecycle workflow, not a guaranteed immediate artifact. That keeps pending or failed transcript visibility operationally important after meetings complete. Source: [Microsoft Graph transcript change notifications](https://learn.microsoft.com/en-us/graph/teams-changenotifications-callrecording-and-calltranscript).
- Access-control baseline: OWASP’s current access-control guidance still recommends deny-by-default, least privilege, and validating permissions on every request. That supports making status-triggered visibility rules explicit in `docs/business-access-rules.md` before further RLS tightening. Source: [OWASP Access Control](https://owasp.org/www-community/Access_Control).
