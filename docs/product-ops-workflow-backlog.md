## Workflow Map

- Public intake and handoffs: public visitor submits contact intent -> `submit-contact` creates `contact_submissions` -> Admin claims, prioritizes, follows up, escalates, or archives the queue item.
- Target response workflow: Bum proposes a warm path on a client target -> Admin or client reviews the response -> the item is claimed, accepted, declined, or formalized into later opportunity work.
- Client-to-Bum intro requests: client requests a specific Bum intro -> Admin claims and advances the request through `SUBMITTED`, `IN_REVIEW`, `INTRO_REQUESTED`, and `CLOSED`.
- Finance workflow: customer payment report is created -> invoice is generated, sent, and marked paid -> payout is allocated, approved, and marked paid, or the item should enter an explicit finance exception lane.
- Admin scrum workflow: QA, release, security, Product Ops, or lead-dev findings create `admin_scrum_items` -> admins triage owner, priority, and proof -> issue closes only after a documented resolution or waiver.

- Zendesk’s current SLA guidance says SLA targets depend on the system Priority field and that policies should measure reply, update, and resolution targets against that priority. That supports making Trusted Bums handoff priority editable and operationally meaningful rather than display-only. Sources: [Defining SLA policies](https://support.zendesk.com/hc/en-us/articles/4408829459866-Defining-SLA-policies) and [About SLA policies and how they work](https://support.zendesk.com/hc/en-us/articles/5600997516058-About-SLA-policies-and-how-they-work).
- Stripe’s current payout reconciliation guidance still centers finance operations on matching each payout to the batch of transactions it settles and explicitly separates failed payouts from normal reconciliation views. That supports dedicated paid-not-allocated, failed, and exception buckets instead of forcing operators to infer finance rescue work from raw statuses. Sources: [Payout reconciliation report](https://docs.stripe.com/reports/payout-reconciliation?locale=en-GB) and [Payout reconciliation](https://docs.stripe.com/payouts/reconciliation?locale=en-GB).
- Microsoft Entra’s current entitlement-management approval guidance still emphasizes approver choice, fallback approvers, decision time limits, and requestor or approver justification. That is a useful benchmark for Trusted Bums admin approval workflows where Product Ops needs durable reason capture rather than silent state flips. Source: [Change approval and requestor information settings for an access package in entitlement management](https://learn.microsoft.com/en-us/entra/id-governance/entitlement-management-access-package-approval-policy).
