### P1 - [TB-0051] Make handoff triage editable and age work by last operator action, not only by create time
- Evidence: [`src/pages/admin/AdminHandoffs.tsx`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/admin/AdminHandoffs.tsx:297) now filters and counts by `admin_priority`, `admin_next_action`, owner, stale age, and notification failure, but queue mutations still only change status or claim ownership. [`src/lib/contactApi.ts`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/lib/contactApi.ts:170) updates contact submissions with status and notes, while [`claimContactSubmission()`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/lib/contactApi.ts:222) hard-codes `admin_next_action` to `Admin follow-up in progress`. [`updateClientBumIntroRequestStatus()`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/lib/portalApi.ts:5930) and [`updateCustomerTargetResponseStatus()`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/lib/portalApi.ts:6007) only persist status, while the claim helpers again hard-code `admin_next_action`.
- Why it matters: Admins can now see urgent or stale work, but they still cannot actually re-prioritize the queue, change the next step, set a due date, or record why a handoff is blocked from the handoff screen. The current stale metric also uses `created_at`, so recently touched work continues to age as if nobody acted on it.
- Recommendation: Add editable `admin_priority`, `admin_next_action`, optional `due_at` or SLA bucket, and operator notes to contact submissions, customer target responses, and client-to-Bum intro requests. Recompute stale state from the latest operator touch or due date rather than only the original create timestamp.
- Acceptance criteria: `/admin/handoffs` lets admins edit priority, next action, and notes inline; stale counts are based on a queue-management timestamp or due date; claim, reprioritize, block, and close actions produce audit history with the acting admin and the new triage state.

### P1 - [TB-0052] Add explicit finance exception lanes and note capture across payment reports, invoices, and payouts
- Evidence: [`src/pages/admin/AdminPayments.tsx`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/admin/AdminPayments.tsx:47) still filters payment reports only by source or `INVOICE_GENERATED`, and invoices only by `GENERATED`, `SENT`, or `PAID`. The invoice actions in [`src/pages/admin/AdminPayments.tsx`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/admin/AdminPayments.tsx:341) only mark invoices sent or paid. [`updateClaimInvoiceStatus()`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/lib/portalApi.ts:3463) auto-creates a `PENDING_ALLOCATION` payout with a generic note. - Why it matters: Finance rescue work still depends on operators inferring exception state from raw status columns. There is no dedicated lane for disputed payment reports, sent-not-paid invoices, paid-not-allocated invoices, payout holds, or void decisions, and there is no durable note workflow from the admin UI when those cases happen.
- Recommendation: Add exception cards, filters, and aging views for uninvoiced payment reports, disputed or voided reports, sent-not-paid invoices, paid-not-allocated invoices, payout holds, and voided payouts. Surface editable notes and owner history in the admin UI for each exception transition.
- Acceptance criteria: Admin finance users can isolate each exception bucket directly in product; invoice and payout transitions can capture a note and owner; at least one admin summary view groups open finance exceptions by age and owner; audit events preserve both status and operator-provided reason text for exception transitions.

The admin-only route is now live in [`src/pages/admin/AdminScrumTracker.tsx`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/admin/AdminScrumTracker.tsx:224), and any admin can close an item from the row UI even when `closureNote` or `evidenceLinks` are empty in [`src/pages/admin/AdminScrumTracker.tsx`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/admin/AdminScrumTracker.tsx:193). The persistence helpers in [`src/lib/portalApi.ts`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/lib/portalApi.ts:6624) create and update rows without writing audit events.
- Why it matters: The scrum tracker is now an operational command surface for QA, release, security, and Product Ops work. Without required proof and audit history, admins can close or retag issues in a way that breaks the paper trail between GitHub evidence, route fixes, and release decisions.
- Recommendation: Require at least one evidence link plus a closeout note before `CLOSED` or `WONT_FIX`, emit audit events for create/update/close actions, and surface `updated_by` and `updated_at` in the tracker UI. Keep `source_key` as the dedupe key for workflow-imported items and avoid mutating it after creation.
- Acceptance criteria: Closing or waiving a scrum item is blocked until proof and decision notes exist; every scrum item change writes an audit event; the tracker shows the last editor and timestamp; release or QA sourced items keep a stable dedupe key tied to their source.

## Workflow Map

- Public intake and handoffs: public visitor submits contact intent -> `submit-contact` creates `contact_submissions` -> Admin claims, prioritizes, follows up, escalates, or archives the queue item.
- Target response workflow: Bum proposes a warm path on a client target -> Admin or client reviews the response -> the item is claimed, accepted, declined, or formalized into later opportunity work.
- Client-to-Bum intro requests: client requests a specific Bum intro -> Admin claims and advances the request through `SUBMITTED`, `IN_REVIEW`, `INTRO_REQUESTED`, and `CLOSED`.
- Finance workflow: customer payment report is created -> invoice is generated, sent, and marked paid -> payout is allocated, approved, and marked paid, or the item should enter an explicit finance exception lane.
- Admin scrum workflow: QA, release, security, Product Ops, or lead-dev findings create `admin_scrum_items` -> admins triage owner, priority, and proof -> issue closes only after a documented resolution or waiver.

That supports dedicated paid-not-allocated, failed, and exception buckets instead of forcing operators to infer finance rescue work from raw statuses. Sources: [Payout reconciliation report](https://docs.stripe.com/reports/payout-reconciliation?locale=en-GB) and [Payout reconciliation](https://docs.stripe.com/payouts/reconciliation?locale=en-GB).
