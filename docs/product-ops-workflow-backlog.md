# Trusted Bums Product Ops Workflow Backlog

_Last updated: 2026-06-04 by Codex daily product ops workflow analyst automation._

## Executive Read

The current Product Ops backlog should stay anchored in shipped workflow risk: access visibility, queue ownership, finance reconciliation, and support rescue. This run revalidated the main risks from source and tests, but did not get fresh Supabase schema or status evidence because the discovered Supabase MCP app failed to initialize when called.

The top current operating risk is a new, stronger represented-contact scoping defect: the Bum contact detail workflow currently offers every accepted opportunity as a link target and accepts any existing `opportunity_registration_id`, even though business rules say represented contacts and captures must stay tied only to destinations the Bum is explicitly entitled to access. The second risk remains a shipped Client Finance export mismatch: `/client/exports` is still available to `CLIENT_FINANCE` and still exports target contacts, meeting attendees, Teams join URLs, and transcript sync status even though the business access rules deny operational exports to finance by default. The third risk is queue quality rather than route absence: Admin access-request review now exists on `/admin/clients`, but the workspace still lacks proof categories, side-effect preview, aging, and reviewer-note discipline.

## Active Recommendations

### P0 - Restrict Bum represented contacts to entitled opportunity destinations
- Evidence: [`src/pages/bum/BumContactDetail.tsx`](src/pages/bum/BumContactDetail.tsx:193) lets a Bum link a represented contact to any returned opportunity. [`supabase/functions/portal-contacts/index.ts`](supabase/functions/portal-contacts/index.ts:474) loads all accepted `opportunity_registrations` without Bum or company scoping, and [`supabase/functions/portal-contacts/index.ts`](supabase/functions/portal-contacts/index.ts:527) accepts any existing `opportunity_registration_id` without checking whether the Bum is entitled to that destination. [`docs/business-access-rules.md`](docs/business-access-rules.md:296) says captures and represented contacts should be tied only to accepted or explicitly assigned opportunity or target relationships for that Bum.
- Why it matters: This creates an avoidable cross-workflow access defect. A Bum can discover unrelated accepted opportunities in the contact-linking UI and attach relationship evidence to destinations outside their approved work, which weakens both access control and operational trust in represented-contact data.
- Recommendation: Scope represented-contact destination options and update validation to Bum-entitled opportunities and targets only. Keep raw `extension_page_captures` Bum/Admin scoped by default, and require any client-facing represented-contact or converted-capture view to inherit an already-approved workflow relationship rather than broad accepted-opportunity visibility.
- Acceptance criteria: Bum contact detail only lists eligible destinations for that Bum; portal-contact update and create paths reject unrelated `opportunity_registration_id` or `customer_target_id` values; QA proves one allowed Bum link case and one denied unrelated-destination case; raw capture rows are not used as a fallback client-visible discovery surface.

### P1 - Harden admin access-request review
- Evidence: [`src/pages/admin/AdminClients.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminClients.tsx:220) now lists admin company access requests, and [`src/pages/admin/AdminClients.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminClients.tsx:457) renders a `Company Access Review` card with approve/deny actions. The UI still shows only email, request type, company/domain, and created date; it does not surface evidence details, proof quality, stale age buckets, request-type filters, approval side effects, or required review notes.
- Why it matters: Access review is now live operational work. Admins need enough context to avoid approving public-email companies, related domains, or Bum signups without proof, and support needs a clean audit trail when a decision is questioned.
- Recommendation: Keep the admin queue, but upgrade it into a review workspace: filter by request type and age, show evidence summary and requester profile state, preview the exact approval mutation, require or strongly prompt notes for public-email and related-domain approvals, and separate same-domain requests that Client Admins can handle from requests requiring Admin proof review.
- Acceptance criteria: Admins can filter pending access requests by type, company/domain, and age; each row shows evidence state and the exact role/company/domain changes approval will make; public-email company and related-domain approvals require a review note or evidence category; approval/denial audit events include request type, actor, target profile, company/domain, and reason/category.

## Workflow Map

- Profile bootstrap and access: signed-in user declares Client or Bum intent -> domain and email proof determines auto-claim, same-domain request, public-email company request, related-domain review, or Bum signup review -> Client Admin or Admin approves the allowed request type -> approval changes `profiles.access_status`, `role`, `company_id`, `client_access_role`, or approved company domain -> pending, denied, and disabled users stay out of company data.
- Public intake and handoffs: visitor submits contact form -> abuse checks and notification attempt run -> `contact_submissions` enters admin handoff queue -> Admin owns review, escalation, conversion, reply, or closure.
- Opportunity and relationship work: client, Bum, customer need, or admin action creates opportunity-like work -> claim, intro request, target response, conversation, meeting, and transcript activity attaches -> accepted work eventually drives payment, invoice, and payout events.
- Finance flow: client or admin records customer payment -> invoice is generated, sent, and paid -> payout allocation should follow or enter an exception bucket.
- Capture and represented-contact flow: Bum capture creates raw relationship evidence -> raw capture and represented-contact linking stay limited to Bum-entitled destinations -> approved conversion creates sanitized contact or workflow context -> client-visible fields depend on the approved converted-field rule and inherited workflow relationship.
- Support and transcript rescue: meeting or feedback event occurs -> sync or manual record is expected -> missing, failed, or unresolved item enters admin rescue ownership with age and next action.

## Current Standards And Time-Sensitive Notes

- Zendesk’s SLA guidance centers support operations on visible response and resolution targets, which supports owner, aging, and breach-risk views in admin rescue queues. Source: [Zendesk SLA policies overview](https://support.zendesk.com/hc/en-us/articles/5600997516058-About-SLA-policies-and-how-they-work).
- Stripe Connect documentation ties dispute responsibility to charge type and negative-balance ownership, and Stripe payout reconciliation requires mapping payouts back to included transactions. That supports dedicated finance exception and reconciliation queues rather than flat invoice or payout status lists. Sources: [Stripe Connect disputes](https://docs.stripe.com/connect/disputes?locale=en-GB), [Stripe payout reconciliation](https://docs.stripe.com/payouts/reconciliation?locale=en-GB).
- Microsoft Entra entitlement-management guidance still treats access-request approval windows, fallback approvers, and periodic access reviews as first-class controls. That supports explicit reviewer deadlines, escalation paths, and recurring access-review follow-up for bootstrap exceptions instead of one-off manual approvals. Sources: [Microsoft Entra approval settings](https://learn.microsoft.com/en-us/entra/id-governance/entitlement-management-access-package-approval-policy), [Microsoft Entra access reviews](https://learn.microsoft.com/en-us/entra/id-governance/entitlement-management-access-reviews-create).
- Microsoft Graph Teams transcript notifications require subscription lifecycle handling for longer-lived transcript subscriptions, and the current Teams change-notification docs still define transcript notification resource paths and lifecycle requirements. That supports transcript sync monitoring and rescue queues. Sources: [Microsoft Graph transcript notifications](https://learn.microsoft.com/en-us/graph/teams-changenotifications-callrecording-and-calltranscript), [Microsoft Teams change notifications](https://learn.microsoft.com/en-us/graph/teams-change-notification-in-microsoft-teams-overview), [Microsoft Graph lifecycle notifications](https://learn.microsoft.com/en-us/graph/change-notifications-lifecycle-events).
