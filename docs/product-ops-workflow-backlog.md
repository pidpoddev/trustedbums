### P1 - [TB-0051] Make handoff aging reflect operator action and finish triage parity

[`ContactSubmissionsPanel.tsx`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/components/admin/ContactSubmissionsPanel.tsx) already supports editable owner, priority, next action, deadline, and notes for contact work. [`portalApi.ts`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/lib/portalApi.ts) still leaves reverse opportunities without the same editable triage metadata.
- Why it matters: Trusted Bums already has live proof that a queue item can sit without owner discipline or follow-up planning. If target responses, intro requests, or Bum-originated reverse opportunities wake up before last-touch and due-date semantics are standardized, operators inherit inconsistent queue behavior and stale counts that do not distinguish ignored work from actively managed work.
- Recommendation: Reuse the contact-submission triage model across `customer_target_responses`, `client_bum_intro_requests`, and `reverse_opportunities`, then switch stale logic from `created_at` alone to due date or explicit last-operator-touch state. A claimed queue item should not remain without a visible next action.
- Acceptance criteria: Admins can set owner, priority, next action, due date, and notes across `contact_submissions`, `customer_target_responses`, `client_bum_intro_requests`, and `reverse_opportunities`; stale filters are driven by due date or explicit last-touch state rather than only `created_at`; and a claimed queue item cannot remain without an actionable next step.

## Workflow Map

- Public contact intake: public visitor submits a contact or signup-intent form -> `submit-contact` creates `contact_submissions` with default admin priority and next action -> Admin claims, qualifies, escalates, or archives the queue item.
- Admin handoffs: Bum target responses and client-to-Bum intro requests surface in `/admin/handoffs` -> Admin claims and advances status -> Product Ops still needs due-state and editable next-step parity with contact submissions.
- Access-review and bootstrap exceptions: `profile-bootstrap` and `client-team` create `client_company_access_requests` when same-domain auto-approval is unsafe or when manual review is required -> Admin or same-company Client Admin resolves the request by request type -> production now proves at least the `BUM_SIGNUP` path is live.
- Shared mailbox intake: `admin-shared-mailbox` syncs `bums@trustedbums.com` into `admin_shared_mailbox_messages` -> Admin Inbox filters by category and status -> admins reply or mark handled -> Product Ops still needs category-specific visibility, assignment, and retention rules before live traffic appears.
- Managing Bum workflow: Admin enables `is_managing_bum` on a Bum profile and assigns the user-level Managing Bum terms record -> the Managing Bum can invite other Bums -> `profile-bootstrap` attaches pending invited signups to `bum_team_memberships` -> team claims, payouts, and future commission allocations stay scoped to that Managing Bum or Admin.
- Client role and beta integration workflow: Client Admin and Client IT can configure beta deal-registration setup in source -> Admin still needs readiness governance before that configuration becomes an operationally enabled workflow.
- Finance workflow: customer payment report -> invoice -> payout -> future Managing Bum allocation, with exception lanes still pending before first live volume.
