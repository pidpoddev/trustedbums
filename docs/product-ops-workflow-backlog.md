## Active Recommendations

### P1 - Surface priority, next action, and notification health inside `/admin/handoffs`
- Evidence: [`src/pages/admin/AdminHandoffs.tsx`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/admin/AdminHandoffs.tsx:258) still filters only by `OPEN`, `UNOWNED`, or search, and queue rows still show status, owner, created time, status picker, and actions only. The underlying records already carry `admin_priority`, `admin_next_action`, and `notification_error` in [`src/lib/contactApi.ts`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/lib/contactApi.ts:29) and [`src/lib/portalApi.ts`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/lib/portalApi.ts:755), and claim actions already stamp `admin_next_action` to `Admin follow-up in progress` in [`src/lib/contactApi.ts`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/lib/contactApi.ts:229) and [`src/lib/portalApi.ts`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/lib/portalApi.ts:5878).
- Why it matters: The queue already stores urgency and handoff intent, but operators cannot scan it in one place. That slows rescue work, hides failed public-contact notifications, and makes the queue less usable as volume grows.
- Recommendation: Promote `admin_priority`, `admin_next_action`, age, and notification health into the `/admin/handoffs` table and summary cards. Add one-click filters for urgent, stale, assigned-to-me, and notification-failed work.
- Acceptance criteria: Handoff rows show priority and next action; contact submissions with notification failure are visibly flagged in the queue; admins can isolate urgent, stale, unowned, and assigned-to-me work without leaving the page; queue summaries distinguish operationally meaningful buckets instead of only open versus unowned.

### P1 - Add explicit finance exception queues and note capture
- Evidence: [`src/pages/admin/AdminPayments.tsx`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/admin/AdminPayments.tsx:79) filters payment reports only by source and invoice-generated state, and [`src/pages/admin/AdminPayouts.tsx`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/pages/admin/AdminPayouts.tsx:21) filters payouts only by status. Current code now correctly centers business dates for finance reporting through [`paymentReportBusinessDate`](/Users/macdaddy/CodexWork/TrustedBums/trustedbums/src/lib/portalApi.ts:3206), but there is still no product-visible queue for uninvoiced payments, sent-not-paid invoices, paid-not-allocated invoices, pending payout allocation, or dispute and variance review.
- Why it matters: The business-date fix removed one stale reporting concern, but the actual finance operations gap remains in exception handling. Finance and support teams need to separate happy-path totals from work that is blocked, aging, or awaiting notes.
- Recommendation: Add exception cards and filters for uninvoiced payment reports, sent-not-paid invoices, paid-not-allocated invoices, pending payout allocation, and dispute or variance review. Require timestamped notes on exception transitions so finance rescue work survives handoff.
- Acceptance criteria: Admin finance users can isolate each exception bucket directly in-product; queue transitions capture notes plus timestamps; paid invoices without payout progression are visible; payout and invoice actions no longer require operators to infer exception state from raw status alone.

## Workflow Map

- Profile bootstrap and access review: user signs up -> domain and signup intent determine auto-claim, same-domain request, public-email request, related-domain request, or Bum signup review -> Client Admin or Admin reviews the allowed request type -> approved path updates profile, company, and domain authority together.
- Public intake and handoffs: public visitor submits contact intent -> `submit-contact` writes `contact_submissions` and attempts notification -> Admin claims, updates, escalates, converts, or archives the record.
- Relationship work: clients create targets and opportunities -> Bums respond, claim, or receive intro requests -> accepted work can produce meetings, transcripts, and later payment workflows.
- Finance flow: customer payment report is created -> invoice is generated, sent, and marked paid -> Bum payout is allocated, approved, and marked paid, or the item should enter an explicit finance exception queue.
- Extension capture flow: Bum captures prospect context against an entitled opportunity or client target -> raw capture remains private workflow evidence -> approved converted workflow outputs can later surface to the client company.
