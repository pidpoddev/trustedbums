## Active Recommendations

### P0 - Split portal routes and heavy report/pdf code out of the startup bundle
- Evidence: `pnpm run build` on 2026-05-28 emitted one app chunk at `1,943.89 kB` minified and `510.23 kB` gzip with Vite's chunk-size warning. [`src/App.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/App.tsx:17) still statically imports every portal layout and page. [`src/components/reports/ReportsWorkspace.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/components/reports/ReportsWorkspace.tsx:1) imports `recharts` into the shared app graph, and [`src/pages/client/ClientTerms.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientTerms.tsx:17) plus [`src/pages/client/ClientAgreements.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientAgreements.tsx:11) eagerly import the inline PDF generator.
- Why it matters: Public and first-portal visits still pay parse and execution cost for admin, client, Bum, charting, and PDF code that most sessions never need, which directly harms first-load responsiveness and LCP.
- Recommendation: Move route groups and heavy leaf pages to module-scope `React.lazy` plus `Suspense`, then lazy-load `ReportsWorkspace` and the PDF generator behind route or click boundaries before considering `manualChunks`.
- Acceptance criteria: Production build no longer emits a single monolithic app chunk, no emitted JS chunk exceeds `500 kB` minified, and initial gzip JS falls below the current `510.23 kB` baseline.

### P1 - Stop report and dashboard routes from loading broad datasets before the user chooses a workflow
- Evidence: [`src/pages/admin/AdminReports.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminReports.tsx:30) mounts `14` queries on entry; [`src/pages/client/ClientReports.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientReports.tsx:42) mounts up to `6`; [`src/pages/bum/BumReports.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/bum/BumReports.tsx:24) mounts `5`; [`src/pages/client/ClientDashboard.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientDashboard.tsx:59) still mounts `4` to `6` queries depending on role. The underlying helpers fetch full joined lists with `select("*", ...)` and no pagination in [`listOwnOpportunityRegistrations`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/portalApi.ts:2568), [`listCustomerPaymentReports`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/portalApi.ts:2754), [`listClaimInvoices`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/portalApi.ts:2861), [`listBumPayouts`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/portalApi.ts:3016), [`listClientReverseOpportunities`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/portalApi.ts:5090), [`listCustomerTargets`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/portalApi.ts:5193), and [`listVisibleBumProfiles`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/lib/portalApi.ts:6529).
- Why it matters: These routes spend network, memory, and main-thread time assembling data the user may never inspect, which raises route load time and interaction cost as table sizes grow.
- Recommendation: Keep landing states on summary counts or the first visible widget only, then fetch detailed datasets after explicit report, tab, or filter selection. For reports, defer chart rendering and chart-module loading until a report with visible chart data is selected.
- Acceptance criteria: Landing on `/admin/reports`, `/client/reports`, and `/bum/reports` only requests the default visible dataset; dashboard routes limit first render to immediately visible cards; switching reports or tabs triggers scoped follow-up fetches instead of full-route fan-out.

### P1 - Clear advisor-backed database friction on report and admin tables
- Evidence: Live Supabase performance advisors on 2026-05-28 still flag unindexed foreign keys on active admin/reporting tables including `audit_events`, `bum_contacts`, `bum_payouts`, `claim_invoices`, and multiple `admin_email_*` tables. The same advisor run also flags duplicate permissive policies on `opportunity_claims`, `opportunity_questions`, `opportunity_registrations`, `profiles`, `reverse_opportunities`, `teams_meetings`, `terms_assignments`, and `training_materials`. Current session evidence is advisor-backed only; live SQL and planner access were not available.
- Why it matters: These tables sit behind reports, admin queues, finance workflows, and audit views. Missing FK indexes and duplicated permissive RLS checks add avoidable latency and make growth regressions predictable.
- Recommendation: Add covering indexes for advisor-flagged foreign keys on the currently queried admin/reporting tables first, then merge overlapping permissive policies per role and action without changing authorization behavior.
- Acceptance criteria: Supabase performance advisors clear the currently flagged hot-path report/admin tables and duplicated policy warnings are reduced for the named tables above without changing role-visible behavior.

### P2 - Resolve unstable array fallbacks that still force avoidable recomputation in table-heavy pages
- Evidence: `pnpm run lint` on 2026-05-28 still passes with `7` `react-hooks/exhaustive-deps` warnings in [`src/pages/admin/AdminCommissionPlans.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminCommissionPlans.tsx:158), [`src/pages/admin/AdminPayments.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminPayments.tsx:89), [`src/pages/admin/AdminPayouts.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/admin/AdminPayouts.tsx:92), [`src/pages/client/ClientPayments.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientPayments.tsx:173), and [`src/pages/client/ClientTargets.tsx`](/Users/ryan.peterson/Documents/Trusted%20Bums%20LOCAL/trustedbums/src/pages/client/ClientTargets.tsx:203).
- Why it matters: These pages already do client-side filtering, totals, and grouping across payment, invoice, payout, plan, and target datasets. Unstable fallback references create unnecessary recomputation on re-render.
- Recommendation: Hoist stable empty-array fallbacks into memoized values or query transforms, then use those stable references as memo dependencies. Only add `useDeferredValue` if inputs still stall after dependency cleanup.
- Acceptance criteria: The current `7` hook warnings are resolved and the affected table pages no longer rerun derived calculations on unrelated renders.

- [React's `lazy` reference](https://react.dev/reference/react/lazy) still requires declaring lazy imports at module scope and pairing them with `Suspense`.

- [Vite's supported release policy](https://vite.dev/releases) currently lists regular fixes for `vite@8.0`, important fixes and security patches for `vite@7.3`, and security fixes for `vite@6.4`; this repo is still on `vite@5.4.21`, which is outside supported versions.
- [Vite 8.0 release notes](https://vite.dev/blog/announcing-vite8), published March 12, 2026, describe the current Rolldown-based build pipeline and materially faster builds. Route-level code splitting still has higher payoff than a toolchain bump, but the support gap keeps the upgrade time-sensitive.
